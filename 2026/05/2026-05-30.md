  - `vllm/model_executor/kernels/linear/scaled_mm/zentorch.py`
  - `vllm/model_executor/kernels/linear/zentorch_utils.py`

### Summary

**What changed and why**  
Added two new zentorch-backed linear kernels for AMD Zen CPUs: ZentorchInt8ScaledMMLinearKernel for dynamic-symmetric W8A8 int8 and ZentorchWNA16LinearKernel for W4A16 GPTQ weight-only quantization. These are registered ahead of existing generic CPU kernels to provide optimized inference on Zen CPUs. Also updated the zentorch dependency version in setup.py and added utility functions for op availability checks.

**Technical impact**  
On AMD Zen CPUs with zentorch installed, W8A8 and W4A16 linear layers will now use zentorch's optimized kernels instead of the generic oneDNN-backed implementations, potentially improving inference performance. The changes are gated on platform and op availability, so no impact on other platforms or when zentorch is absent. The kernel selection logic in __init__.py is updated to prioritize these new kernels on CPU.

**Potential risks**  
The new kernels may have bugs or performance regressions on Zen CPUs, especially for edge cases like AWQ-packed weights or layers with activation reordering (g_idx). The fallback to existing kernels via can_implement and super() should mitigate this, but untested configurations could lead to silent fallback or errors. The dependency version change from a weekly release to a fixed version may cause compatibility issues if zentorch API changes.

**Key insights**  
The PR introduces platform-specific optimizations for AMD Zen CPUs, leveraging zentorch's custom ops. The design uses a fallback mechanism to ensure robustness, and the code includes detailed eligibility checks for the W4A16 kernel to handle different quantization formats (GPTQ vs AWQ). The scoping note indicates a previous draft included a TorchAO DA8W8 fast path that was removed, suggesting a focused scope for this PR.

## 4. [[Bug] Fix gemma4 MTP IMA issue when TP>1, `CUDA error: an illegal memory access was encountered`](https://github.com/vllm-project/vllm/pull/43909)


### Base Information

- **PR Number:** #43909
- **Author:** [yewentao256](https://github.com/yewentao256)
- **Merged By:** [yewentao256](https://github.com/yewentao256)
- **Merged time:** 2026-05-30 07:34:33
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/43909/files) (1):**
  - `vllm/model_executor/models/gemma4_mtp.py`

### Summary

**What changed and why**  
Fixed a CUDA illegal memory access error in Gemma4 MTP speculative decoding when tensor parallelism > 1. The issue was that the temporary variable `lm_head_weight` (result of all-gather) was being released after the function returned, causing a dangling pointer when the CUDA graph replayed. The fix caches the gathered weight in `self._stable_full_lm_head_weight` and ensures it is contiguous, preventing memory reuse.

**Technical impact**  
Resolves a crash (CUDA illegal memory access) during speculative decoding with Gemma4 MTP and TP>1. The fix adds a small memory overhead (caching the gathered weight) but ensures correctness and stability. The benchmark shows successful execution with expected latency.

**Potential risks**  
Low risk. The cached weight is reset to None in `load_weights` to avoid stale data after weight updates. The change only affects the TP>1 path and does not alter behavior for TP=1. The contiguous() call may add a minor overhead but is necessary for CUDA graph replay.

**Key insights**  
The root cause was a common pitfall in CUDA graph replay: temporary tensors that are freed after the capturing function returns can cause illegal memory access during replay. Caching the tensor as an instance attribute ensures its lifetime extends beyond the function scope. This fix is specific to the Gemma4 MTP model but the pattern is applicable to other models using CUDA graphs with temporary tensors.
