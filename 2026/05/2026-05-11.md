  - `vllm/v1/engine/utils.py`
  - `vllm/v1/worker/xpu_worker.py`

### Summary

**What changed and why**  
The PR modifies XPU worker device assignment to no longer rely on environment variable isolation (ZE_AFFINITY_MASK) for data parallelism. Instead, it computes the device index as DP_LOCAL_RANK * TP_PP_WORLD_SIZE + TP_LOCAL_RANK, aligning with the GPU worker approach. This fixes accuracy issues observed when combining data parallelism with expert parallelism on XPU. Additionally, the PR refactors the XPU worker's init_device and profile methods, and updates the engine utils to exclude XPU from needing device env isolation.

**Technical impact**  
Changes device assignment logic for XPU workers in data-parallel configurations, affecting how workers map to physical devices. The init_device method now adjusts local_rank based on DP rank and TP/PP world size, and includes bounds checking. The profile method is overridden to support torch profiling with proper trace naming. The engine utils change ensures that XPU, like CUDA, does not use set_device_control_env_var for DP, relying instead on the new device index calculation.

**Potential risks**  
The new device index calculation assumes that visible devices are contiguous and that the DP rank and TP/PP sizes are correctly configured. If the visible device count is insufficient or the rank calculation is off, it could cause out-of-bounds errors or incorrect device assignments. The change removes environment variable isolation, which might affect other platform-specific behaviors that depended on ZE_AFFINITY_MASK. The refactored profile method introduces a dependency on TorchProfilerWrapper and may break if profiling is not enabled.

**Key insights**  
The PR addresses a real accuracy issue in XPU DP+EP by adopting the same device assignment strategy as GPU workers, which is more robust. The refactoring of init_device and profile methods improves code organization and adds profiling support. The change is well-tested with a specific model and configuration, but broader testing across different DP/TP/PP setups would be beneficial.

  - `vllm/v1/worker/gpu/sample/sampler.py`

### Summary

**What changed and why**  
Replaced tl.static_range(MAX_SPEC_LEN) with tl.range(pos) in _penalties_kernel to avoid a Triton 3.7.0 compiler crash on sm_89 GPUs when num_speculative_tokens >= 10. Also removed the separate draft_counts accumulator and instead accumulate directly into output_bin_counts to eliminate a layout conversion. Removed num_speculative_tokens parameter from kernel and callers.

**Technical impact**  
Fixes a crash on L4 GPUs with speculative decoding when num_speculative_tokens >= 10. The change also improves performance by up to 12% in some cases (e.g., spec=7, tokens=64) by avoiding unnecessary layout conversions and using a more compact loop. No regression observed in benchmarks.

**Potential risks**  
The change modifies the accumulation logic: previously draft_counts was computed separately and added to base_output_counts; now accumulation is done directly into output_bin_counts. This should be functionally equivalent but could introduce subtle differences if base_output_counts is reused elsewhere. The removal of MAX_SPEC_LEN parameter may affect other callers if any exist (none in this PR). The change relies on tl.range which may have different behavior in older Triton versions.

**Key insights**  
The crash was caused by Triton's TritonGPURemoveLayoutConversions pass failing on unrolled scf.if blocks from tl.static_range. Using tl.range emits a compact scf.for loop that avoids the issue. The optimization of accumulating directly into output_bin_counts leverages the fact that base_output_counts is in #blocked2 layout, so the loop stays in that layout without conversion. This is a good example of how compiler-specific workarounds can also yield performance improvements.

## 11. [[BugFix] Prevent orphaned process on NCCL destroy](https://github.com/vllm-project/vllm/pull/39846)


### Base Information

- **PR Number:** #39846
- **Author:** [jeffreywang-anyscale](https://github.com/jeffreywang-anyscale)

- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/39846/files) (2):**
  - `vllm/distributed/device_communicators/pynccl.py`
  - `vllm/distributed/device_communicators/pynccl_wrapper.py`

### Summary

**What changed and why**  
Changed PyNcclCommunicator.destroy() from calling ncclCommDestroy (which deadlocks during uncoordinated shutdown) to calling ncclCommAbort in a daemon thread with a 5-second timeout. This prevents orphaned GPU processes when one rank exits before others, because ncclCommAbort does not require collective synchronization and the daemon thread avoids self-deadlock from CUDA graph references held by the main thread.

**Technical impact**  
Fixes a hang/crash bug in multi-process (mp) backend with TP > 1 inside Ray actors. The change ensures that NCCL communicators are torn down safely even when peer ranks are gone, allowing subsequent tests or runs to reclaim GPU resources. The daemon thread pattern mirrors PyTorch's c10d implementation.

**Potential risks**  
The 5-second timeout may be insufficient in some environments, leading to incomplete cleanup. The daemon thread is not joined after timeout, so resources might leak if abort takes longer. However, this is a pragmatic trade-off to avoid deadlock. The change also introduces a new NCCL function binding (ncclCommAbort) which must be available in the linked NCCL library.

**Key insights**  
ncclCommDestroy is a collective operation that deadlocks during uncoordinated shutdown. ncclCommAbort is non-collective but can still block if CUDA graphs reference the communicator; running it in a daemon thread allows the main thread to release those graphs. The fix is inspired by PyTorch's ProcessGroupNCCL::abort().
