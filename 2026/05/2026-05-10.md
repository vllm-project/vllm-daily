## 5. [[ROCm][Bugfix]: dynamically align BLOCK_DMODEL with Lv in MLA decode kernel](https://github.com/vllm-project/vllm/pull/41119)


### Base Information

- **PR Number:** #41119
- **Author:** [vllmellm](https://github.com/vllmellm)
- **Merged By:** [tjtanaa](https://github.com/tjtanaa)
- **Merged time:** 2026-05-10 20:14:20
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/41119/files) (1):**
  - `vllm/v1/attention/ops/triton_decode_attention.py`

### Summary

**What changed and why**  
The PR fixes a Triton compilation error on ROCm for MLA-based models (e.g., Mistral-Small) by dynamically aligning BLOCK_DMODEL with the local latent rank (Lv) instead of the total head dimension (Lk) when is_mla=True. It replaces hardcoded dimension checks with generalized next_power_of_2 logic and adjusts BLOCK size to 16 for HIP/ROCm to balance shared memory pressure. Additionally, it adds a num_stages=1 fallback for NVIDIA when BLOCK_DMODEL >= 1024 to avoid shared memory overflow.

**Technical impact**  
The change ensures shape compatibility for tl.dot operations in MLA decode kernels, resolving a ValueError on ROCm. It generalizes the kernel to support a wider range of MLA configurations while maintaining regression-free accuracy for DeepSeek-V3/V3.1 and other models. Performance benchmarks show improved Time to First Token (TTFT) on ROCm with no throughput regression.

**Potential risks**  
The dynamic alignment may alter memory access patterns, potentially affecting performance on other hardware or configurations not tested. The BLOCK=16 for HIP/ROCm could reduce occupancy on some GPUs. The NVIDIA num_stages=1 fallback might increase latency for large BLOCK_DMODEL cases. However, the PR includes regression testing on multiple MLA models to mitigate these risks.

**Key insights**  
The root cause was a mismatch between BLOCK_DMODEL (aligned to Lk) and the actual latent dimension Lv used in MLA, causing shape incompatibility in Triton's dot product. The fix leverages triton.next_power_of_2 for dynamic alignment, which is more robust than hardcoded values. The BLOCK size reduction for ROCm is a pragmatic workaround for shared memory limits on MI300X.

  - `vllm/model_executor/layers/mhc.py`
  - `vllm/model_executor/models/deepseek_v4.py`

### Summary

**What changed and why**  
Added a fused mHC post-pre kernel that combines the hc_post operation with the prenorm_gemm portion of hc_pre, improving performance at low concurrency. The approach uses FMA-based GEMM instead of tensor cores, adapted from TRTLLM. Also updated the DeepSeek-V4 model to use the fused kernel and added unit tests.

**Technical impact**  
Introduces a new tilelang kernel `mhc_fused_tilelang` that fuses the post-mapping and pre-norm GEMM steps, reducing kernel launch overhead and memory traffic. The kernel uses FMA instructions and manual warp-level reductions, which is beneficial for low token counts. The Python wrapper `mhc_fused_post_pre` selects between the fused kernel (for <=16 tokens) and the existing separate kernels (for larger batches), with heuristic tile sizes and split-k parameters. The model forward pass is refactored to pass residual and mix tensors between layers, enabling the fusion across consecutive layers.

**Potential risks**  
The fused kernel uses hard-coded metaparameters (tile_n, split_k) that may not be optimal for all shapes; the PR notes autotuning is still pending. The FMA-based approach may not be as efficient as tensor cores for larger batch sizes, but the fallback to separate kernels mitigates this. The model changes require careful handling of the first and last layers (standalone hc_pre and hc_post). Accuracy is validated via benchmarks and eval results, but edge cases (e.g., very small hidden sizes, non-standard hc_mult) may not be covered.

**Key insights**  
The fusion of post and pre operations is a common optimization in transformer inference, and this PR adapts it for the mHC architecture. The use of FMA instead of tensor cores is a deliberate trade-off to improve low-concurrency throughput, which is critical for serving scenarios. The heuristic selection between fused and separate kernels based on token count is a pragmatic approach to balance performance across batch sizes.

### Summary

**What changed and why**

The changes include conditional creation of embed_tokens, norm, lm_head, and mtp_hidden_buffer based on PP rank, implementation of make_empty_intermediate_tensors for PP intermediate tensors, and updates to forward and load_weights methods to handle PP partitioning. This is part of issue #40902 to enable PP for DeepSeek-V4.

**Technical impact**  
Enables PP for DeepSeek-V4, allowing model sharding across pipeline stages. The model now implements SupportsPP interface. PP=4 test results show comparable accuracy to TP=4 on gsm8k and gpqa_diamond tasks. The change modifies model initialization, forward pass, and weight loading to be PP-aware.

**Potential risks**  
PP introduces additional complexity and potential for bugs in weight loading and intermediate tensor handling. The mtp_hidden_buffer is only allocated on last PP rank, which could cause issues if MTP draft is used on other ranks. The make_empty_intermediate_tensors assumes hc_mult and hidden_size are available, which may not hold in all configurations. Weight loading skips parameters not present on current PP rank, which could lead to missing weights if PP rank mapping is incorrect.

**Key insights**  
The PR uses PPMissingLayer and is_pp_missing_parameter utilities to handle PP partitioning cleanly. The intermediate tensors carry multi-stream hidden_states of shape (batch_size, hc_mult, hidden_size) to support V4's head-collapse architecture. The forward method conditionally handles first/last rank for embedding and norm, and returns IntermediateTensors for non-last ranks.
