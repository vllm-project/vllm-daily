## 1. [Keep scheduler alive for delayed KV connector frees](https://github.com/vllm-project/vllm/pull/43433)

- **PR Number:** #43433

  - `tests/v1/kv_connector/unit/test_multi_connector.py`
  - `vllm/v1/core/sched/scheduler.py`
  - `vllm/v1/engine/core.py`

### Summary

**What changed and why**  
Fixed a bug where the scheduler could stop polling no-forward engine steps while a finished request was waiting on delayed KV connector cleanup, causing KV blocks to remain allocated. The fix keeps scheduler liveness true for such requests and yields briefly when scheduler work remains. Added tests for the delayed-free path and improved test robustness by filtering out 'take_events' entries.

**Technical impact**  
The scheduler's has_finished_requests() method now also considers requests that are not in scheduling queues but still in self.requests (i.e., waiting for delayed connector cleanup). The engine's _process_engine_step now yields the GIL if scheduler.has_requests() is true instead of only when there are unfinished requests, ensuring background connector progress can complete. This prevents resource leaks and ensures timely cleanup of KV blocks.

**Potential risks**  
The change to has_finished_requests() may cause the scheduler to report finished requests when there are no actual finished requests but some requests are in an intermediate state (e.g., being added or removed). However, the condition is conservative (len(self.requests) > num_in_queues) and should only be true when there are requests not in queues, which is the intended case. The yield in the engine adds a 1ms sleep when no model execution occurs but scheduler work remains, which could slightly increase latency in some edge cases but is necessary for correctness.

**Key insights**  
The root cause was that the scheduler's liveness check only considered finished_req_ids, but delayed connector cleanup keeps the request in self.requests after removal from queues. The fix ensures the scheduler remains active until all requests are fully cleaned up. The test additions validate the lifecycle and improve test reliability by ignoring 'take_events' entries that can appear nondeterministically.

### Summary

**What changed and why**  
Fixed a correctness bug in the native Triton top-k/top-p kernel where it assumed contiguous row-major layout for logits. The kernel now uses logits.stride(0) for row addressing, and a fallback to contiguous copy is added when stride(1) != 1. A regression test was added to verify non-contiguous logits produce the same results as contiguous inputs.

**Technical impact**  
The fix ensures correct behavior for non-contiguous logits views (e.g., padded vocab), preventing incorrect top-k/top-p masking and downstream NaNs. The kernel signature changed to include LOGITS_STRIDE_0, and the Python wrapper now handles unsupported layouts by making a contiguous copy. Performance impact is minimal for common contiguous cases, and the fallback only triggers for rare non-contiguous inner strides.

**Potential risks**  
The fallback to contiguous copy may increase memory usage and latency for unsupported layouts, but such layouts are rare. The change is backward-compatible for contiguous inputs. The new test covers the main scenarios, but edge cases like zero-size tensors or extreme strides are not explicitly tested.

**Key insights**  
The bug was caused by assuming row stride equals vocab size, which fails for sliced views. Using the actual stride from the tensor is the correct fix. The conservative fallback for non-unit inner stride ensures correctness without modifying the kernel further. The test uses a padded tensor to create a non-contiguous view, which is a realistic scenario.

**What changed and why**  
This PR migrates pos_encoding and norm kernels (rms_norm, fused_add_rms_norm, rotary_embedding, fused_qk_norm_rope, rms_norm_static_fp8_quant, fused_add_rms_norm_static_fp8_quant, rms_norm_dynamic_per_token_quant, rms_norm_per_block_quant) from the main _C extension to the libtorch stable ABI extension _C_stable_libtorch. This is part of an ongoing effort to reduce the number of unstable PyTorch ABI symbols in the main extension, improving binary compatibility across PyTorch versions. The source files are moved to csrc/libtorch_stable/, and the corresponding op registrations and implementations are moved from csrc/torch_bindings.cpp to csrc/libtorch_stable/torch_bindings.cpp. Declarations in csrc/ops.h are retained for CPU builds that still use torch::Tensor.

New dispatch macros for quant types, group size, vec size, and tensor rank are added to csrc/libtorch_stable/dispatch_utils.h. The CMakeLists.txt is updated to move source files from VLLM_EXT_SRC to VLLM_STABLE_EXT_SRC. The CPU build continues to use the old torch::Tensor declarations in csrc/ops.h.

**Potential risks**  
The migration changes the ABI interface for these kernels; any Python code that directly calls these ops via the old _C extension will break if it was relying on the old ABI. The CPU build still uses the old declarations, so there is a risk of divergence if the CPU and CUDA implementations are not kept in sync. **Key insights**  
This is a continuation of PR #38783, focusing on migrating norm and positional encoding kernels to the stable ABI. The PR reduces the unstable symbol count by 2, which is a modest but incremental improvement. The migration is non-trivial because it involves changing tensor types, dispatch macros, and device management (e.g., using torch::stable::accelerator::DeviceGuard instead of at::cuda::OptionalCUDAGuard). The PR also adds new dispatch utilities (quant types, group size, vec size, rank) that are reusable for future migrations. The retention of old declarations in csrc/ops.h for CPU builds is a pragmatic decision to avoid breaking the CPU path.
