  - `tests/v1/cudagraph/test_breakable_cudagraph.py`
  - `vllm/compilation/breakable_cudagraph.py`
  - `vllm/config/vllm.py`
  - `vllm/envs.py`
  - `vllm/model_executor/layers/attention/mla_attention.py`
  - `vllm/model_executor/layers/deepseek_v4_attention.py`
  - `vllm/model_executor/layers/sparse_attn_indexer.py`
  - `vllm/v1/attention/ops/rocm_aiter_mla_sparse.py`
  - `vllm/v1/cudagraph_dispatcher.py`
  - `vllm/v1/worker/gpu_model_runner.py`

### Summary

**What changed and why**  
Introduces an experimental breakable CUDA graph feature (VLLM_USE_BREAKABLE_CUDAGRAPH=1) that replaces torch.compile-based piecewise CUDA graph compilation with runtime stream-capture breaks. The core idea is to capture the entire forward pass in a single CUDA graph context, but allow 'eager breaks' at attention/kv-cache custom ops via a decorator, avoiding the need for pre-splitting the model. This reduces dependency on torch.compile and can improve throughput for certain models (e.g., DeepSeek).

**Technical impact**  
Adds a new BreakableCUDAGraphCapture context manager and BreakableCUDAGraphWrapper class that mirrors the existing CUDAGraphWrapper interface. The decorator @eager_break_during_capture marks functions that should run eagerly during capture, splitting the graph into segments. The implementation uses thread-local state to manage capture contexts and weak references to avoid pinning memory. Configuration changes disable torch.compile when the flag is set. The feature is currently limited to DeepSeek models and is incompatible with sequence_parallel.

**Potential risks**  
Experimental feature with limited testing; may have startup time regression (noted but later removed). Only tested on specific models (DeepSeek, Qwen) and hardware (B300, GB200). Not compatible with sequence_parallel pass. The decorator order is critical and misuse could cause hangs. Thread-local state could leak across tests if not properly reset. The feature is gated behind an environment variable, so default behavior is unchanged.

**Key insights**  
The approach is inspired by sglang's similar technique and aims to simplify the CUDA graph pipeline by avoiding torch.compile's FX graph splitting. Performance improvements are modest (2-7% throughput gain) but the main benefit is reduced complexity and faster compilation. The implementation carefully handles weak references and thread safety. The PR includes comprehensive unit tests for the capture/replay primitives.

  - `docker/Dockerfile`

### Summary

**What changed and why**  
Added build args and Dockerfile logic to optionally install a custom mooncake-transfer-engine wheel from a URL, replacing the PyPI version. This enables using a wheel built with specific CMake flags (e.g., WITH_NVIDIA_PEERMEM=OFF, USE_MNNVL=ON) to fix dmabuf MR registration for GPU memory sub-views and enable MNNVL on NVL72. The change is gated by INSTALL_KV_CONNECTORS=true and uses new environment variables for wheel URLs.

**Technical impact**  
Introduces a new build-time mechanism to override the mooncake-transfer-engine package, allowing non-default configurations. The Dockerfile now conditionally installs a wheel from a URL and creates a symlink for libcudart.so if needed. The release pipeline passes the new build args to all relevant build steps. This increases build flexibility but adds complexity and potential failure points if wheel URLs are invalid or incompatible.

**Potential risks**  
If the wheel URL is unreachable or the wheel is incompatible with the base image's glibc, the build will fail. The symlink creation assumes specific CUDA library naming conventions, which may not hold for all CUDA versions. The change only applies when INSTALL_KV_CONNECTORS=true, so default builds are unaffected. However, the new build args are passed unconditionally in the pipeline, which could cause issues if the Dockerfile doesn't handle empty values gracefully (though the RUN script checks for non-empty WHEEL).

**Key insights**  
The change addresses a specific hardware configuration (GB200, NVL72) where the published PyPI wheel does not work due to missing dmabuf support and MNNVL flags. By building from source with custom flags, the team gains control over the compiled features. The use of pre-built wheels from S3 avoids the need to compile Mooncake during every Docker build, saving time. The commit hash and PR reference provide traceability to the upstream fix.
