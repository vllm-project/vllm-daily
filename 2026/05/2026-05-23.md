  - `tests/v1/kv_connector/unit/test_mooncake_store_scheduler.py`
  - `vllm/distributed/kv_transfer/kv_connector/v1/mooncake/store/scheduler.py`
  - `vllm/distributed/kv_transfer/kv_connector/v1/mooncake/store/worker.py`

### Summary

**What changed and why**  
The PR fixes a bug where MooncakeStore full hits caused kvpool_cached_tokens to be sub-block aligned, leading to silent tail drops or empty key_list in the recv thread. It rounds full hits down to a block boundary, removes the downstream +1 compensation paths, and eliminates the discard_partial_chunks config option. Tests are added for the full-hit case.

**Technical impact**  
Ensures block-aligned load boundaries, preventing silent data loss and ZeroDivisionError in the recv thread. Simplifies code by removing conditional compensation logic and the discard_partial_chunks option. The change affects scheduler, worker, and test files, with 74 additions and 39 deletions.

**Potential risks**  
If the block-aligned rounding is incorrect for edge cases (e.g., num_tokens < block_size), it could skip loading entirely. The removal of discard_partial_chunks may affect users who relied on that config, though it was default true and now always applied. The fix assumes block_size is consistent; mismatched block sizes could cause issues.

**Key insights**  
The root cause was a mismatch between block-aligned load masks and sub-block-aligned kvpool_cached_tokens. The fix aligns the full-hit decrement to a block boundary, ensuring the recv thread's load mask covers all yielded chunks. The removal of compensation paths and config option simplifies the code and reduces potential for future bugs.

  - `tests/parser/test_streaming.py`
  - `vllm/parser/abstract_parser.py`

### Summary

**What changed and why**  
Fixed two bugs in DelegatingParser.parse_delta that caused reasoning content to be silently dropped when a streaming delta chunk spans the reasoning/tool-call boundary. First, replaced is_reasoning_end(delta_token_ids) with is_reasoning_end_streaming(current_token_ids, delta_token_ids) to correctly detect the end-of-reasoning marker in multi-token chunks. Second, saved the reasoning field from delta_message before calling the tool parser, and restored it afterward to prevent overwriting. Added three regression tests covering boundary scenarios with and without tool parser, and a token-by-token streaming test.

**Technical impact**  
Fixes a correctness bug where reasoning content could be lost during streaming, particularly when a single chunk contains the last reasoning tokens, the </think> token, and the first tool-call tokens. The change ensures that reasoning is preserved across boundary deltas and that the end-of-reasoning detection works correctly with accumulated token context. The new helper method is_reasoning_end_streaming is added to the abstract parser interface.

**Potential risks**  
Low risk. The changes are localized to the streaming delta parsing logic and are guarded by existing reasoning/tool parser conditions. The new method is_reasoning_end_streaming defaults to False when no reasoning parser is present. The regression tests cover the main boundary scenarios. However, any other code paths that rely on the old is_reasoning_end behavior with delta-only tokens might be affected if they are not updated, but no such usages exist in the modified files.

**Key insights**  
The root cause was twofold: boundary detection using only delta tokens fails when the </think> token is split across chunks, and the tool parser overwrites delta_message without preserving reasoning. The fix uses the full accumulated token list for boundary detection and explicitly saves/restores reasoning. The test helper _boundary_chunks cleverly splits the model output into chunks that straddle the boundary, enabling targeted regression testing.

  - `vllm/model_executor/kernels/linear/__init__.py`

### Summary

**What changed and why**  
Added an explicit batch-invariant mode for NVFP4 CUTLASS linear kernels (SM100 and SM120) that uses a persistent tile scheduler to ensure deterministic numerical results across different batch sizes. Previously, batch invariance was accidentally preserved by the default scheduler; this change makes it explicit with a static assert to prevent future breakage. Also updated the Python-side kernel selection logic to prefer the batch-invariant CUTLASS path when VLLM_BATCH_INVARIANT is set, falling back to emulation only if unsupported. Added a new test file for batch invariance and updated CI configuration.

**Technical impact**  
Introduces a dedicated cutlass config with PersistentScheduler for batch-invariant mode, adds static_assert to enforce it, and refactors SM120 dispatch to be templated and file-local to avoid symbol clashes. The Python init logic now checks CutlassNvFp4LinearKernel.is_supported() before falling back to emulation. Performance benchmarks show the batch-invariant CUTLASS path is slightly slower than the default CUTLASS for small M but still faster than Triton and much faster than emulation.

**Potential risks**  
The batch-invariant mode uses a fixed large-M tiling (sm100_fp4_config_default) for all batch sizes, which may be suboptimal for very small M. The static_assert ensures the scheduler is persistent, but any future changes to the config must be carefully coordinated. The SM120 dispatch was refactored into a template function inside an anonymous namespace; if other translation units define a similar function, there could be ODR violations, but the anonymous namespace prevents that. The new test requires a fresh process and VLLM_BATCH_INVARIANT=1, which may be missed in CI if not configured correctly.

**Key insights**  
The batch-invariant mode is achieved by using a persistent tile scheduler (PersistentScheduler) instead of the default stream-K or split-K schedulers, which would introduce non-determinism. The change is minimal and leverages existing cutlass infrastructure. The performance impact is negligible for large M and acceptable for small M, making it a practical solution for deterministic inference.
