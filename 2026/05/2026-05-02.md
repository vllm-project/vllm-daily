- **[Changed Files](https://github.com/vllm-project/vllm/pull/41297/files) (2):**
  - `vllm/v1/worker/gpu/model_runner.py`
  - `vllm/v1/worker/gpu/shutdown.py`

### Summary

**What changed and why**  
Added a shutdown() method to the model runner and a helper function free_before_shutdown() to release GPU resources (model weights, KV caches, workspace) when running in the same process. This is needed to properly clean up memory before shutdown, preventing resource leaks.

**Technical impact**  
Introduces a new public method shutdown() on the model runner and a new module shutdown.py. The method synchronizes the accelerator, clears KV caches, attention groups, deletes model and cache config, calls free_before_shutdown() which resets workspace manager, clears rotary embedding cache, and sets num_gpu_blocks to None. Finally runs garbage collection and empties GPU cache. This ensures memory is reclaimable.

**Potential risks**  
If shutdown() is called while other operations are in progress, it may cause errors. The use of hasattr checks suggests some attributes may not exist, but if they do and are referenced elsewhere after shutdown, it could lead to crashes. The deletion of self.model and self.kv_cache_config could cause issues if other parts of the code hold references. The function relies on internal imports (e.g., _ROPE_DICT) which may change.

**Key insights**  
The change is a targeted cleanup addition for MRV2, following patterns seen in other parts of the codebase. It uses gc.collect() and torch.accelerator.empty_cache() to force memory release. The helper function free_before_shutdown() centralizes clearing of global state like rotary embeddings and workspace manager, which is important for multi-process or repeated model loading scenarios.

### Summary

**What changed and why**  
This PR adds a new 'humming' MoE backend for MXFP4 quantization, enabling support for DeepSeek-V4 models with improved performance. It introduces the humming backend option in kernel config, modifies the MoE layer to use humming experts, adds weight conversion utilities, and refactors the existing humming integration to work with the new MoE modular kernel system. The changes also include removing the old humming_moe_utils.py and adding humming_utils.py for shared functionality.

**Technical impact**  
The PR adds a new MoE backend that can be selected via `--moe-backend humming`, providing alternative kernel implementations for MXFP4 quantized models. It modifies the MoE layer initialization and weight loading to support humming's weight format and kernel dispatch. The changes also refactor the humming quantization method to use the new modular kernel interface, improving code organization and maintainability. Performance benchmarks show up to 84% improvement in prefill throughput and 5% in decoding throughput compared to the Marlin baseline.

**Potential risks**  
The humming backend is optional and requires explicit flag to use, so default behavior is unchanged. However, the refactoring of the humming quantization method (e.g., moving weight_schemas/input_schemas to layer objects) could affect existing humming users if they rely on the old attribute locations. The new humming_utils.py introduces dependencies on the humming package, which is not yet a mandatory dependency, so users must install it separately. There is also a risk of regressions in the existing humming linear layer support due to the refactoring of `process_weights_after_loading`.

**Key insights**  
The humming backend leverages optimized MoE kernels (grouped and indexed) and a custom MoE sum kernel to achieve significant performance gains, especially for prefill. The PR demonstrates a clean integration pattern by separating humming-specific utilities into a dedicated module and using the modular kernel system. The accuracy results show negligible difference from the baseline, confirming that the optimization does not compromise model quality.

**What changed and why**  
Changed the default value of VLLM_MULTI_STREAM_GEMM_TOKEN_THRESHOLD from 4096 to 1024 based on empirical throughput data on GB300 and GB200 hardware showing 1024 yields better or comparable performance across configurations.

**Technical impact**  
Alters the threshold at which multi-stream GEMM is enabled, affecting GPU kernel scheduling and overlap behavior. This can improve throughput for typical workloads (e.g., 8k/1k) by up to ~10% on GB300 and maintains similar performance on GB200.

**Potential risks**  
The new default may degrade performance for workloads with very high token counts per request (e.g., >1024 tokens) where the previous threshold was more optimal. However, the PR notes that in disaggregated serving, prefill nodes already disable multi-stream, mitigating this risk.

**Key insights**  
The change is data-driven, with benchmarks on two GPU architectures showing 1024 as a robust default. The comment update references the PR for empirical results, improving documentation traceability.

- **[Changed Files](https://github.com/vllm-project/vllm/pull/41522/files) (1):**
  - `vllm/model_executor/models/deepseek_v4.py`

### Summary

**What changed and why**  
The PR modifies the DeepSeek V4 model to guard the 'use_mega_moe' flag with Pure TP (tensor parallelism). Previously, the flag was only set when expert parallelism was enabled; now it is always set based on the moe_backend config, and an error is raised if mega moe is used without expert parallelism. This ensures that the mega moe backend is only used when expert parallelism is enabled, preventing silent misconfiguration.

**Technical impact**  
The change alters the initialization logic of two classes in deepseek_v4.py. It removes the conditional dependency on enable_expert_parallel for setting use_mega_moe, making the flag always reflect the moe_backend config. It adds an explicit check that raises NotImplementedError if mega moe is selected without expert parallelism, which could break existing configurations that inadvertently used mega moe without expert parallel.

**Potential risks**  
Existing code that relied on the old behavior (where use_mega_moe was False when expert parallel was disabled) may now raise an error if moe_backend is set to 'deep_gemm_mega_moe'. Users must ensure they enable expert parallelism when using this backend. The change is backward-incompatible for those configurations.

**Key insights**  
The PR enforces a hard requirement that the DeepSeek V4 MegaMoE backend must be used with expert parallelism. This is a safety guard to prevent unsupported configurations. The change is small but semantically significant, as it moves from a silent fallback to an explicit error.

### Summary

**What changed and why**  
Modified Dockerfile to support a manylinux build OS branch (using dnf and pre-installed Python interpreters) alongside the existing Ubuntu path, and updated the release pipeline to use the manylinux base image for CUDA 13.0 wheel builds. This lowers the glibc floor from 2.34 to 2.28, restoring compatibility with older Linux distributions like RHEL 8 and Ubuntu 20.04.

**Technical impact**  
The change introduces a new BUILD_OS build argument that controls package manager (apt vs dnf) and Python bootstrap method. The manylinux path uses /opt/python/cpXY-cpXY interpreters and dnf, while the Ubuntu path remains unchanged. Both converge on /opt/venv, so downstream stages are unaffected. The release pipeline now passes BUILD_OS=manylinux and a PyTorch manylinux base image for CUDA 13.0 wheels, while CUDA 12.9 and the final runtime image stay on Ubuntu.

**Potential risks**  
The manylinux path may have subtle differences in system libraries (e.g., libibverbs vs rdma-core-devel) that could cause build failures or runtime issues on older glibc. The Python interpreter from /opt/python may have different behavior than the uv-fetched one. The change only affects CUDA 13.0 wheel builds; other configurations are unchanged. Validation is pending, so there is a risk of undetected issues.

**Key insights**  
This PR aligns the glibc floor of vLLM wheels with PyTorch's published wheels, enabling compatibility with older OSes. The Dockerfile branching pattern (BUILD_OS arg) is clean and maintainable, with both paths converging on /opt/venv. The change is limited to build-time images; the runtime image remains Ubuntu, so no impact on deployment environments.
