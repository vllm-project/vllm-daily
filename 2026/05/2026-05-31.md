## 3. [[BugFix] Fix `_has_module` to verify native deps via trial import](https://github.com/vllm-project/vllm/pull/44035)


### Base Information

- **PR Number:** #44035
- **Author:** [jeffreywang-anyscale](https://github.com/jeffreywang-anyscale)
- **Merged By:** [vllm-bot](https://github.com/vllm-bot)
- **Merged time:** 2026-05-31 22:06:34
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/44035/files) (2):**
  - `tests/utils_/test_import_utils.py`
  - `vllm/utils/import_utils.py`

### Summary

**What changed and why**  
Fixed `_has_module` to perform a trial import after `find_spec` to verify native dependencies are loadable. Previously, `find_spec` could return a valid spec for modules with missing shared libraries, causing false positives and import-time crashes. The fix adds `importlib.import_module()` and returns `False` on failure, with broad exception handling for `ImportError`, `OSError`, and unexpected errors.

**Technical impact**  
Improves robustness of optional dependency detection by ensuring that modules reported as available are actually importable. This prevents crashes when native extensions have missing shared libraries. The change is backward-compatible for modules that import successfully, and the `@cache` decorator maintains performance by caching results per module.

**Potential risks**  
The trial import may have side effects for modules with initialization code, but since `_has_module` is used for optional dependencies, this is acceptable. The broad exception handling (catching `ImportError` and `OSError` and logging at `WARNING`) may mask genuine import issues, but the previous behavior was to crash, so this is an improvement. The `find_spec` call is now wrapped in a try-except to handle cases where it raises for dotted names.

**Key insights**  
The root cause was that `find_spec` only checks for module spec existence, not importability. The fix adds a trial import, which is the only reliable way to verify native dependencies. The test suite covers various failure modes including `ImportError`, `OSError`, unexpected errors, and `find_spec` failures, ensuring the fix is comprehensive.

## 4. [[MRV2] Remove Eagle's dedicated CUDA graph pool](https://github.com/vllm-project/vllm/pull/44078)


### Base Information

- **PR Number:** #44078
- **Author:** [LucasWilkinson](https://github.com/LucasWilkinson)
- **Merged By:** [WoosukKwon](https://github.com/WoosukKwon)
- **Merged time:** 2026-05-31 22:00:33
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/44078/files) (2):**
  - `vllm/v1/worker/gpu/spec_decode/eagle/cudagraph.py`
  - `vllm/v1/worker/gpu/spec_decode/eagle/speculator.py`

### Summary

**What changed and why**  
Removed the dedicated CUDA graph pool for Eagle speculative decoding, making the main model and Eagle share a single graph memory pool. The separate pool was initially introduced to avoid memory conflicts, but testing showed it is unnecessary and wastes memory.

**Technical impact**  
Reduces GPU memory usage for graph pools by ~0.27 GiB (from 2 pools to 1) without affecting accuracy, acceptance rates, or performance. Simplifies code by removing the EagleCudaGraphManagerBase class and the pool sharing logic in speculator.py.

**Potential risks**  
Low risk. The change has been validated with GSM8K accuracy and acceptance rate tests showing no regression. However, edge cases with different model architectures or larger batch sizes might expose memory conflicts that were previously avoided by separate pools.

**Key insights**  
The dedicated pool was a conservative design choice that turned out to be unnecessary. Sharing the pool saves memory and simplifies the codebase. The PR demonstrates the value of empirical validation over assumptions about memory conflicts.
