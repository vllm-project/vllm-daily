## 2. [[ROCm][CI] Align spec decode logprob test prefill settings](https://github.com/vllm-project/vllm/pull/41335)


### Base Information

- **PR Number:** #41335

- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/41335/files) (1):**
  - `tests/v1/sample/test_logprobs.py`

### Summary

**What changed and why**  
Refactored test_spec_decode_logprobs to share LLM configuration between reference and speculative decoding runs. Previously, the speculative LLM used chunked prefill while the reference did not, causing numerical divergence on ROCm due to non-associative floating-point reductions. Now both use identical settings via a shared llm_kwargs dict, isolating speculative decoding behavior.

**Technical impact**  
Fixes a ROCm-only test failure by ensuring consistent execution geometry between the two LLM instances. The test now correctly compares logprobs under identical scheduling, so any difference is attributable to speculative decoding rather than numerical noise from different reduction orders.

**Potential risks**  
Low risk. The change only affects test configuration and does not modify production code. The shared settings (chunked prefill, max_num_batched_tokens=32) are already used by the speculative LLM; applying them to the reference LLM may slightly change its behavior but is necessary for a fair comparison.

**Key insights**  
Floating-point non-associativity in GPU kernels can cause test flakiness when execution configurations differ. Sharing configuration via a common dict reduces duplication and enforces consistency, which is a good pattern for tests that compare outputs from multiple model instances.

## 4. [[CI] Clean up remote servers on pytest parent exit](https://github.com/vllm-project/vllm/pull/41570)


### Base Information

- **PR Number:** #41570
- **Author:** [AndreasKaratzas](https://github.com/AndreasKaratzas)
- **Merged By:** [DarkLight1337](https://github.com/DarkLight1337)
- **Merged time:** 2026-05-03 20:11:23
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/41570/files) (1):**
  - `tests/utils.py`

### Summary

**What changed and why**  
Added tracking of live RemoteVLLMServer instances and registered atexit/signal handlers to clean them up when the pytest parent process exits. This prevents detached vLLM server process groups from surviving an interrupted test run, which could keep ROCm VRAM full and cause subsequent server startup failures.

**Technical impact**  
Introduces class-level sets and locks for active server tracking, atexit registration, and signal handler chaining for SIGTERM/SIGINT. The _shutdown method now uses a flag to prevent double cleanup, and shutdown_many is updated to mark servers as shutdown before parallel termination and unregister them after memory release. This ensures robust cleanup even on abrupt exits.

**Potential risks**  
Signal handler chaining may interfere with other signal handlers if not properly restored. The atexit and signal hooks are registered only once, but if multiple RemoteVLLMServer subclasses exist, the classmethod approach uses the root class, which is correct. There is a small risk of deadlock if _shutdown is called from a signal handler while holding the lock, but the lock is only used for registration/unregistration, not during shutdown itself.

**Key insights**  
The change addresses a real CI reliability issue where orphaned server processes consumed GPU memory. The design uses class-level state and thread-safe registration to ensure cleanup happens regardless of how the parent exits. The signal handler chains to previous handlers to preserve existing behavior. The shutdown_many optimization marks servers as shutdown before parallel termination to avoid redundant work.

## 6. [[Bugfix] Fix degenerate KV cache stride causing TMA cudaErrorIllegalInstruction](https://github.com/vllm-project/vllm/pull/40737)


### Base Information

- **PR Number:** #40737
- **Author:** [the-david-oy](https://github.com/the-david-oy)
- **Merged By:** [robertgshaw2-redhat](https://github.com/robertgshaw2-redhat)
- **Merged time:** 2026-05-03 16:52:19
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/40737/files) (6):**
  - `tests/v1/attention/test_kv_head_stride_canonicalization.py`
  - `vllm/utils/cpu_resource_utils.py`
  - `vllm/utils/torch_utils.py`
  - `vllm/v1/attention/backends/flash_attn.py`
  - `vllm/v1/attention/backends/flash_attn_diffkv.py`
  - `vllm/v1/attention/backends/flashinfer.py`

### Summary

**What changed and why**  
Fixed a bug where degenerate strides on size-1 dimensions in the KV cache caused CUDA TMA illegal instruction errors on H100+ GPUs. The root cause was that PyTorch allows any stride on size-1 dims, but TMA requires non-outermost strides to be multiples of 16 bytes. Added canonicalize_singleton_dim_strides() to patch stride metadata for size-1 dims to canonical C-contiguous values, and applied it in FlashInfer, FlashAttention, and FlashAttentionDiffKV backends after KV cache slicing. Also tightened TRTLLM stride assertions and added unit tests.

**Technical impact**  
Fixes a crash (cudaErrorIllegalInstruction) that occurred with prefix caching enabled and num_kv_heads_per_rank == 1 (e.g., Qwen3.5-397B with TP=8). The fix is zero-copy (only stride metadata updated) and safe because size-1 dimensions are never stepped across in pointer arithmetic. All three attention backends are affected and now protected. The change also improves debuggability by adding logging and stricter assertions.

**Potential risks**  
Low risk: the canonicalization function only modifies stride metadata for size-1 dims and returns the same tensor if no change is needed. It preserves data pointers and storage offsets. The function is applied only to KV cache tensors after slicing, not to other tensors. The main risk is if a size-1 dim is used in a way that relies on its original stride, but that is unlikely since stride on a size-1 dim is irrelevant for indexing.

**Key insights**  
PyTorch's is_contiguous() returns True for any stride on a size-1 dimension, so .contiguous() does not fix degenerate strides. This is a subtle interaction between PyTorch's stride semantics and CUDA TMA hardware requirements. The fix is a targeted, zero-copy stride correction that can be applied generically to any tensor with size-1 dims. The bug affects multiple attention backends and was triggered by prefix caching's block reallocation.

  - `vllm/model_executor/model_loader/reload/meta.py`

### Summary

**What changed and why**  
Fixed a bug in materialize_layer where non-meta tensors (like bias) were being overwritten during weight materialization. The issue caused bias values to be corrupted when loading FP8 models with bias=True, leading to NaN outputs. The fix adds a check `tensor.is_meta` before materializing, ensuring only meta tensors are replaced.

**Technical impact**  
Resolves incorrect model outputs for FP8 quantized models with bias layers. The change is minimal (one line) but critical for correctness. Also adds a test to prevent regression.

**Potential risks**  
Low risk. The change only affects the materialization path for meta tensors, which is already guarded by SKIP_TENSORS. The added test covers the specific scenario. No other code paths are affected.

**Key insights**  
The root cause was that materialize_layer iterated over all tensors and replaced them without checking if they were already on-device. The fix ensures that only meta tensors are materialized, preserving pre-existing non-meta tensors like bias.

### Summary

**What changed and why**  
In vllm/v1/engine/utils.py, when data_parallel_size > 1, the instance_id in each DP engine's VllmConfig is appended with the DP rank to ensure unique Ray actor names. Previously, all DP replicas shared the same instance_id, causing ActorAlreadyExistsError in RayExecutorV2.

**Technical impact**  
Fixes a crash when using RayExecutorV2 with data_parallel_size > 1. Enables multi-DP deployments with the new Ray executor backend, which was previously broken. Single-DP deployments are unaffected due to the guard.

**Key insights**  
The bug was caused by copy.deepcopy preserving instance_id across DP replicas, while RayExecutorV2 uses instance_id to name actors uniquely. The fix appends the DP rank to instance_id, ensuring uniqueness. The legacy executor doesn't use named actors, so it was unaffected.
