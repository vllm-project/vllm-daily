This PR reverts merge commit 8189a15914ca48461acf106f126c58ef7e41c9ee, which introduced a device-cache and async D2H pipeline for routing replay. The revert restores the previous SharedMemory-based design because the original author from nemo RL cannot support prefix caching and other features in time for urgent releases. The net diff is -1495/+733 across 15 files, including deletion of documentation and near-full rewrite of routed_experts_capturer.py.

**Technical impact**  
The revert removes the device-cache-based routing capture and async D2H pipeline, replacing them with the older SharedMemory and file-locking approach. This changes the data flow for routed expert information: instead of per-layer GPU buffers and async copies, the capturer now writes to shared memory and the scheduler reads via file locks. The scheduler now uses a RoutedExpertsReader to fetch routing data from shared memory based on slot mappings.

**Potential risks**  
Reverting to SharedMemory may reintroduce multi-node scalability issues and performance bottlenecks from synchronous D2H transfers and file locking. The removal of the async pipeline could increase GPU stall time. The revert also drops support for prefix caching and other features that were validated with the new design, potentially breaking workflows that depend on those features. The codebase now has two different routing capture mechanisms, which may cause confusion or maintenance overhead.

**Key insights**  
This revert prioritizes stability and urgent release needs over the architectural improvements of the device-cache approach. The decision was made collaboratively to use an alternative PR (#39568) instead. The revert is substantial, affecting core routing capture logic, API outputs, and scheduler integration. The SharedMemory approach is simpler but less scalable and performant, especially for multi-node deployments.

Previously, GPUModelRunner._sample() passed None for draft_probs, forcing the rejection sampler onto a fallback. The change captures draft probabilities in the proposer, preserves them across the runner boundary, realigns them by request, and passes them into RejectionSampler so probabilistic rejection sampling uses the intended p(x)/q(x) logic. The change is limited to the V1 speculative decoding path and only affects the probabilistic rejection method. Tests cover the new functionality, but edge cases like empty draft probs or missing request IDs are handled with fallback warnings.

**Key insights**  
The root cause was that draft_probs were never wired through the V1 path, so the rejection sampler always fell back to a heuristic. The fix required changes across the proposer, runner, and config layers. The manual GPU validation shows clear improvement in acceptance metrics.
