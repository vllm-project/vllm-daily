## 11. [[Bugfix] Fix persistent_topk inter-CTA init race on RadixRowState](https://github.com/vllm-project/vllm/pull/41444)

- **PR Number:** #41444
- **Author:** [zyongye](https://github.com/zyongye)

- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/41444/files) (2):**
  - `csrc/persistent_topk.cuh`
  - `csrc/topk.cu`

### Summary

**What changed and why**  
Fixed a race condition in persistent_topk where RadixRowState initialization was done by CTA-0 only with intra-CTA synchronization, lacking inter-CTA ordering. This caused two races: within a launch, CTA-1+'s atomicAdd could be overwritten by CTA-0's plain store; across launches, stale arrival_counter values broke the cooperative barrier. The fix replaces in-kernel init with host-side cudaMemsetAsync before launch, providing stream-ordered visibility for all CTAs and resetting state between launches.

**Technical impact**  
Eliminates undefined behavior in the persistent kernel path for large rows (max_seq_len > RADIX_THRESHOLD) when multiple CTAs cooperate. The memset adds ~92KB of overhead, negligible compared to kernel runtime. The kernel prologue is simplified by removing the CTA-0-only init branch. FilteredTopK paths are unaffected.

**Potential risks**  
Low risk. The memset is conditional on needs_cooperative, so it only runs when the radix path is active. The change is localized to two files and the removed code was buggy. Potential regression if cudaMemsetAsync fails, but error is checked. No impact on non-cooperative launches.

**Key insights**  
The bug was subtle because the chunk-load loop often masked the race in single launches, but cross-launch state leak was deterministic. Using host-side memset is a clean solution that leverages CUDA's stream ordering guarantees, avoiding complex device-side synchronization. The fix also improves code clarity by removing conditional init logic from the kernel.

The change introduces a token-count threshold: when the batch size (number of tokens) is at or below the threshold, multi-stream overlap is enabled; above it, the overlap is disabled to avoid performance degradation from saturated SMs. The default threshold is 4096 tokens, based on empirical data for B300 GPUs.

The change is limited to DeepSeek-V4 attention and does not affect other models.

**Key insights**  
The PR addresses a performance regression observed in large-batch forward passes where multi-stream GEMM overlap becomes detrimental. The solution uses a simple token-count heuristic to dynamically enable/disable overlap, with a configurable threshold via environment variable. The test results show a slight accuracy improvement (0.9507 to 0.9545) on gsm8k with the env var enabled, though the difference is within noise. The change is well-documented with comments explaining the rationale and expected crossover points for different GPU architectures.
