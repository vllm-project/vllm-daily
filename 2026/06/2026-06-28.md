### Summary

**What changed and why**  
Added CUDA graph support for the Gemma3 encoder (vision tower + multimodal projector) to reduce TTFT and ITL latency. This is a partial implementation of issue #38175, enabling encoder CUDA graphs for image inputs only.

**Technical impact**  
Implements the SupportsEncoderCudaGraph interface in Gemma3ForConditionalGeneration, adding methods for graph configuration, budget range, item specs, selection, capture inputs, replay buffers, and forward passes. The token budget range is bounded by max_model_len and max_num_batched_tokens, which could limit graph reuse for very long sequences. The test configuration uses a single token budget of 512, which may not cover all production scenarios.

**Key insights**  
The encoder CUDA graph provides meaningful latency improvements for Gemma3, especially for TTFT and ITL tail latencies. The implementation follows the established pattern for other models (e.g., DeepSeekOCR, InternVL). The benchmark shows that the graph does not significantly affect TPOT or throughput, making it a safe optimization. The code includes proper handling of multi-patch images via num_patches and cum_patches logic.

Added a `num_unpadded_tokens` field to EPLB state to exclude padding tokens (from CUDA graphs or DP padding) from expert load recording. The router kernel now only counts the first N tokens as real, while the logical-to-physical mapping still applies to all tokens. This prevents padded tokens from polluting the expert load statistics used for EPLB rebalancing.

**Technical impact**  
Introduces a new per-ubatch tensor list `num_unpadded_tokens_tensors` in `EplbModelState` and `EplbLayerState`, propagated to all layers. The Triton kernel `_eplb_map_and_record_i32_kernel` gains a `HAS_NUM_UNPADDED` flag and conditional skip for padded tokens. Callers (model runner, speculators) must call `prepare_forward` before each forward pass. Also adds a caching utility `compute_hash_cached` for config hashes.

**Potential risks**  
If `num_unpadded_tokens_tensors` is not properly set (e.g., missing `prepare_forward` call), the kernel defaults to counting all tokens, which may silently degrade EPLB balancing. The change touches many files across the stack (routers, speculators, model runner), so integration bugs are possible. The caching of config hashes by object identity assumes config objects are never mutated after construction, which could cause stale hashes if violated.

**Key insights**  
The PR correctly separates the mapping (applied to all tokens) from recording (only real tokens), preserving correctness for downstream kernels. The use of `dbo_current_ubatch_id()` to index into the tensor list supports dynamic batching. The addition of `compute_hash_cached` is a minor optimization that reduces repeated SHA-256 computations. Tests cover both padded and unpadded scenarios, including the case where no padding info is provided (backward compatibility).

### Summary

**What changed and why**  
Replaced the all-reduce + chunk pattern in MoE layers with a reduce-scatter approach to reduce communication overhead. Previously, the full hidden states were all-reduced across ranks, then chunked for local MoE computation, followed by all-gather. Now, after attention, a reduce-scatter distributes shards to each rank, local MoE runs on the shard, and all-gather is performed only if needed. This reduces the communication volume from full all-reduce to reduce-scatter, improving throughput.

**Technical impact**  
Improves total token throughput by ~3-5% (from ~491k to ~508k tok/s) and reduces TTFT by ~5-7% (from ~1171ms to ~1101ms). The change introduces a new `use_sequence_parallel_moe` flag and modifies the forward pass of DeepseekV2DecoderLayer to conditionally perform reduce-scatter before MoE and all-gather after attention. It also adds padding logic for non-divisible sequence lengths and handles residual alignment. The `reduce_results` parameter is added to attention layers to skip unnecessary all-reduce when SP MoE is enabled.

**Potential risks**  
The change is complex and touches core model execution logic. Potential risks include: (1) Incorrect handling of sequence parallel boundaries, especially with pipeline parallelism (currently disabled for SP MoE). (2) Padding and truncation logic may introduce off-by-one errors or shape mismatches. (3) The `already_sequence_parallel` flag and conditional all-gather in the decoder layer forward could lead to subtle bugs if not all paths are covered. (4) Performance regression for non-SP configurations due to added condition checks. (5) The change assumes that `residual` is not None when input is sequence-parallel; if that invariant is violated, it may cause errors.

**Key insights**  
The optimization leverages the fact that MoE computation is naturally sharded across experts, so a reduce-scatter (which combines reduction and scatter) is more efficient than an all-reduce followed by chunking. The implementation carefully manages the sequence-parallel state across attention and MoE layers, using all-gather to reconstruct full states when needed (e.g., for attention). The performance gains are modest but consistent, and the change is well-tested with accuracy benchmarks (GSM8K 94.4% exact match). The TODO comment about PP boundary indicates future work to extend this optimization to pipeline-parallel scenarios.
