## 3. [fix: prevent MM cache hang from stale LRU order keys](https://github.com/vllm-project/vllm/pull/43595)

  - `vllm/utils/cache.py`

### Summary

**What changed and why**  
Fixed LRUCache.touch() to not insert ghost keys into the internal order when the key is missing, and hardened popitem() to clean stale order-only keys. This prevents an infinite eviction loop that could cause the multimodal cache to hang, leaving requests stuck in processing.

**Technical impact**  
Eliminates a hang in EngineCore input processing by ensuring eviction always makes progress. touch() now only updates recency for existing keys, and popitem() skips stale order entries. The fix is minimal (1 file, +1/-3 lines) but critical for correctness of the LRU eviction logic.

**Potential risks**  
Low risk. The change is defensive and only affects edge cases where touch() was called on missing keys or stale order entries existed. Existing behavior for normal cache operations is unchanged. The popitem() hardening ensures backward compatibility with any existing stale state.

**Key insights**  
The root cause was a mismatch between touch() (which created order-only entries) and popitem() (which assumed every order key had a corresponding data entry). The fix aligns touch() with standard LRU semantics and makes popitem() robust against inconsistent state. The multimodal cache's pre-touch pattern is now safe.
