## 2. [[GLM5.2 Perf] `fused_indexer_q_rope_quant` triton kernel, 1.9% ~ 3.3% E2E Throughput improvement.](https://github.com/vllm-project/vllm/pull/46862)

  - `vllm/model_executor/layers/sparse_attn_indexer.py`
  - `vllm/model_executor/models/deepseek_v2.py`

### Summary

**What changed and why**  
Added a fused Triton kernel `_fused_indexer_q_rope_quant_kernel` that combines Q RoPE, FP8 quantization, and q_scale folding into weights in a single kernel, replacing the previous separate operations. This improves performance by reducing kernel launch overhead and memory traffic. The kernel is enabled for GLM models when conditions (CUDA, head_dim=128, rope_dim=64, etc.) are met.

**Technical impact**  
Introduces a new Triton kernel that fuses three operations: applying rotary position embedding to Q, quantizing to FP8, and folding the quantization scale into attention weights. The kernel uses a block size of 128 (head_dim) and handles both Neox and interleaved RoPE layouts. The change modifies the forward pass of the attention layer in DeepseekV2 model to use this fused kernel when applicable, resulting in 1.9% to 3.3% end-to-end throughput improvement as measured by benchmarks.

**Potential risks**  
The fused kernel is only enabled under specific conditions (CUDA, head_dim=128, rope_dim=64, quant_block_size == head_dim, scale_fmt not None). If these conditions are not met, the original code path is used, so there is no regression. However, the new kernel has not been tested on non-CUDA platforms or with different head dimensions. Additionally, the kernel assumes FP8 quantization with e8m0 format; any changes to quantization parameters could break the kernel. The accuracy was validated on GSM8K, but other tasks may be affected.

**Key insights**  
The fusion of RoPE, quantization, and scale folding into a single kernel reduces kernel launch overhead and memory bandwidth usage, leading to measurable throughput gains. The kernel is implemented in Triton and leverages the existing `get_fp8_min_max` utility. The change is specific to the GLM model's attention mechanism and is conditionally enabled to avoid breaking other configurations.

The bug stemmed from scanning the entire token sequence for `<\\|END_THINKING\\|>` without considering turn boundaries. ### Summary

**What changed and why**  
Introduces a GPU video decoding backend using PyNvVideoCodec and a VRAM semaphore (MultiModalGPUMemoryPool) to prevent OOM during heavy load. The semaphore allows the frontend to acquire a budget of GPU memory for decoding, blocking if insufficient, and releases after copying to pinned host memory. Also adds configuration for the budget (mm_ipc_gpu_memory_gb) and reserves fixed decoder overhead from KV cache.

**Technical impact**  
Adds a new video decoding backend (pynvvideocodec) that leverages NVIDIA hardware decoding, improving performance for GPU-accelerated preprocessing. The VRAM semaphore ensures safe memory usage under concurrency, preventing OOM. The KV cache budget is reduced by the configured semaphore budget plus fixed decoder overhead, which may affect throughput for models with large KV cache. The change introduces new dependencies (PyNvVideoCodec) and new configuration parameters.

**Potential risks**  
The new backend is only available on CUDA systems; non-CUDA platforms will fall back to existing backends. The semaphore budget is a tunable parameter that may require careful tuning to balance KV cache and decoding throughput. The fixed decoder overhead (e.g., 1.8 GiB per API server for CUDA context) may be significant for small GPUs. The implementation assumes GPU index 0 for decoding, which may conflict with other GPU usage. The decoder slot management uses threading primitives that could introduce contention under extreme load.

**Key insights**  
The PR implements a practical admission control mechanism for GPU memory in multimodal preprocessing, enabling safe hardware acceleration. The design separates raw frame budget from decoder overhead, and scales the overhead by the number of API server processes. The decoder slot reuse via reconfigure_decoder reduces per-request overhead. The approach is a stepping stone toward full zero-copy decode-to-inference as described in RFC #30839.
