This is part of a larger effort to make vLLM's custom CUDA extensions compatible with the stable libtorch ABI, enabling them to work across different PyTorch versions without recompilation.

**Technical impact**  
Replaces the `_moe_C` extension with `_moe_C_stable_libtorch`, which uses `torch::stable::Tensor` and `STABLE_TORCH_LIBRARY_IMPL` macros instead of the unstable ABI. All MoE ops are now registered under the `_moe_C` namespace in the stable library. The CMake build system is updated to compile the new extension with stable ABI flags and link against the appropriate CUDA/HIP libraries. The Python import path changes from `vllm._moe_C` to `vllm._moe_C_stable_libtorch`. The number of unstable symbols in the MoE extension drops from 82 to 0, while the stable shim count increases from 0 to 70.

**Potential risks**  
The migration changes the extension name and internal API, which could break any code that directly imports `vllm._moe_C` or relies on the old ABI. The new extension requires PyTorch >= 2.10 (ROCm) or >= 2.11 (CUDA) due to the `TORCH_TARGET_VERSION` compile definitions. There is a risk of subtle behavioral differences if the stable ABI wrappers do not perfectly replicate the original tensor operations. The ROCm-specific workaround for linking the bundled `libamdhip64` may introduce issues if the library path is incorrect.

**Key insights**  
This PR is a pure refactoring/migration with no intended functional changes; all existing tests pass. The use of `STABLE_TORCH_LIBRARY_IMPL` and `TORCH_BOX` macros ensures ABI stability. The migration also includes adding new dispatch macros (`VLLM_STABLE_DISPATCH_INTEGRAL_TYPES`) and a new `moe_ops.h` header for the stable library. The PR is part of a series (labeled 10c/n) and can be merged independently, though CMakeLists.txt and torch_binding.cpp may have merge conflicts with other PRs in the series.

Added force-reinstall of nvidia-cutlass-dsl-libs-cu13 in three Dockerfile stages to fix CUDA 13 builds. **Technical impact**  
Resolves a race condition in uv's wheel extraction order for nvidia-cutlass-dsl[cu13] on CUDA 13. **Key insights**  
The root cause is a packaging issue in nvidia-cutlass-dsl where two wheels (libs-base and libs-cu13) share paths with different content. uv's parallel extraction can interleave files, causing corruption. The fix is a targeted reinstall of the cu13-specific wheel after all other installs, ensuring correct file state.

The async batched lookup pattern effectively hides I/O latency by batching requests per scheduling step and processing them in a background thread.   - `vllm/model_executor/models/qwen3_5_mtp.py`
  - `vllm/v1/spec_decode/llm_base_proposer.py`

### Summary

**What changed and why**  
Introduced a LocalArgmaxMixin and get_top_tokens method to replace all-gathering full-vocabulary logits with local argmax + gather reduction in speculative decoding (DFlash, PARD). This reduces TP communication from O(batch_size × vocab_size) to O(batch_size × 2 × tp_size). The mixin is added to LlamaForCausalLM, Qwen3ForCausalLM, Eagle3DeepseekV2ForCausalLM, and Qwen3_5MTP. The previous fallback logic in Llama4Eagle and the warning in llm_base_proposer are removed because the mixin now handles D2T remapping correctly.

**Technical impact**  
Significant reduction in communication overhead for large-vocab draft models, improving total token throughput by 9-31% in tested scenarios. The change is backward-compatible and does not affect accuracy (acceptance rates unchanged). The mixin design allows easy adoption by other models.

**Potential risks**  
The mixin assumes the subclass exposes logits_processor, lm_head, and optionally draft_id_to_target_id. If a model does not have these attributes, it will raise an AttributeError. The D2T remapping logic assumes draft_id_to_target_id is a 1D tensor; incorrect shapes could cause silent errors. The change removes a warning about fallback, so users with D2T remapping will now use the optimized path, which is correct but could expose bugs if the remapping logic is not equivalent.

**Key insights**  
The optimization leverages the fact that argmax over a sharded vocabulary can be computed locally and then reduced with minimal communication. The D2T remapping is handled by adding the offset to the local argmax index, preserving correctness. The mixin pattern allows clean reuse across multiple model classes. The performance gains increase with larger TP sizes and larger vocabularies.
