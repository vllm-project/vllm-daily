## 1. [[ROCm][gpt-oss] Hybrid CDNA4 swizzle gate for A8W4 MoE](https://github.com/vllm-project/vllm/pull/44804)

  - `vllm/model_executor/layers/quantization/utils/mxfp4_utils.py`

### Summary

**What changed and why**  
Gated the CDNA4 swizzle for MXFP4 scale tensors on tensor model parallel size <= 2, falling back to StridedLayout for TP >= 4. This is because at higher TP, the smaller per-rank shapes require BLOCK_K < 256, which is incompatible with the swizzle. A helper function `should_use_cdna4_mx_scale_swizzle()` was introduced to ensure consistency between weight-load layout and kernel argument.

**Technical impact**  
Changes the scale tensor layout and kernel behavior for TP >= 4 on gfx950, switching from CDNA4 swizzled layout to StridedLayout. This prevents silent corruption of scale tensors that could occur if the weight-load layout and kernel swizzle argument were mismatched. Performance impact: strided layout wins most cases (23/28 cells) with up to +19% improvement on decode-heavy workloads, while swizzle only benefits high-concurrency scenarios.

**Potential risks**  
Low risk: the change is gated on TP size and platform, and the helper ensures consistency. However, any future changes to the swizzle logic must update the helper to avoid drift. Performance regression for TP=1/2 is not expected as swizzle remains enabled. The fallback to StridedLayout for TP>=4 is well-tested and shows performance gains.

**Key insights**  
The PR addresses a correctness issue where mismatched swizzle settings could corrupt scale tensors. The helper function centralizes the decision logic, preventing future drift. Performance data shows that strided layout is generally better for TP>=4, so the gate is both a correctness fix and a performance improvement.
