  - `vllm/distributed/kv_transfer/kv_connector/v1/nixl/push_scheduler.py`

### Summary

**What changed and why**  
This PR adds support for hybrid KV-cache models (e.g., Qwen3.5 with GDN/MambaSpec) to the Mooncake connector for P/D disaggregation. Previously, Mooncake treated all cache regions as homogeneous full-attention KV pages, which broke hybrid models. The changes introduce cache-spec-aware transfer planning: separate handling for full-attention and GDN state regions, proper group identity preservation, N-1 token truncation for MambaSpec decode, and deduplication of shared backing memory registration. The Nixl scheduler method is also renamed for consistency.

**Technical impact**  
The Mooncake connector now correctly handles hybrid models by distinguishing between full-attention and GDN/MambaSpec cache groups. Key changes include: (1) resolving all attention backends instead of assuming the first one, (2) separating memory registration from transfer-region metadata to handle shared allocations, (3) preserving KV-cache group identity in transfer planning, (4) applying logical-to-physical block expansion only for attention groups, and (5) implementing N-1 token truncation for MambaSpec decode. The Nixl scheduler method is renamed from `_mamba_prefill_token_count` to `_get_remote_prefill_token_count` for consistency. Tests are added for hybrid GDN scenarios.

**Potential risks**  
The changes are substantial and touch core transfer logic. Risks include: (1) regression in pure full-attention models (mitigated by regression test showing identical accuracy), (2) incorrect handling of other hybrid models not tested (e.g., Mamba2), (3) potential issues with hetero-TP or PP configurations (explicitly out of scope), (4) the N-1 truncation logic may interact poorly with preemption or rescheduling, and (5) the new group index matching could cause alignment failures if metadata is inconsistent.

**Key insights**  
The PR demonstrates that Mooncake's homogeneous KV-page assumption was a fundamental limitation for hybrid models. The solution carefully separates concerns: memory registration is allocation-based, while transfer regions are spec-aware. The N-1 truncation mirrors the Nixl approach, ensuring the decoder recomputes the last prompt token locally to obtain correct logits. The extensive test results show no accuracy regression for pure attention models and comparable accuracy to Nixl for hybrid models, with Mooncake often achieving higher throughput.

### Summary

**What changed and why**  
Added an `active_lora_requests` counter to `RequestRegistry` to track the number of in-flight LoRA requests. The `lora_adapter_states()` method now returns empty sets immediately when the counter is zero, avoiding an O(N) scan of all requests. The counter is incremented on registration with a LoRA adapter, decremented on removal/abort/finish, and reset on close. Two new tests verify the counter behavior and that the early return works correctly.

**Technical impact**  
Eliminates unnecessary iteration over all in-flight requests in `lora_adapter_states()` when no LoRA requests are active. **Potential risks**  
The counter must be kept in sync with the actual number of LoRA requests. If any code path adds or removes a LoRA request without updating the counter, the count could drift, causing either missed early returns (still correct but slower) or incorrect early returns (returning empty sets when LoRA requests exist). The current implementation updates the counter in all relevant methods (register, remove, finish_many, abort_many, close), so risk is low. The new tests cover the main paths.

**Key insights**  
The optimization leverages a simple counter to avoid a linear scan in the common case of no LoRA requests. This is a classic space-for-time tradeoff that is well-suited to the Rust frontend's performance-sensitive path. The counter is maintained with minimal overhead and is reset on close to ensure consistency.
