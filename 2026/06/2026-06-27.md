The changes include a new model file, config, processor, attention layer (RSWAAttention), KV cache spec and manager (RSWASpec/RSWAManager), and modifications to the FlexAttention and FlashAttention backends to support R-SWA mask_mod. Also updated the DeepSeek-OCR processor to accept a max_crops parameter and fixed a bug in crop mode logic for multi-image requests.

**Technical impact**  
Introduces a new model architecture with a novel attention mechanism (R-SWA) that bounds per-request KV cache memory to O(prefix + window) by evicting gap blocks during decode. This requires changes across the attention backend (FlexAttention and FA4), KV cache manager, scheduler, and model runner. The R-SWA mask_mod is implemented for both FlexAttention (Triton block mask) and FlashAttention 4 (CuTE-DSL mask_mod). The model also disables prefix caching and forces FA4 or FlexAttention backends. **Potential risks**  
The R-SWA implementation is complex and touches multiple subsystems (attention backends, KV cache manager, scheduler). Incorrect mask_mod or block eviction logic could cause attention errors or memory corruption. The forced FA4 requirement may break on devices without FA4 support. Disabling prefix caching may increase memory usage for some workloads. The multi-image fallback (no crop) changes token count calculation, which could cause mismatches if not handled consistently. The new model files and configs may have compatibility issues with future transformers versions.

**Key insights**  
Unlimited-OCR reuses the DeepSeek-OCR vision stack but uses a DeepSeek-V2 MoE language backbone with R-SWA instead of dense MLA. R-SWA is implemented as a custom mask_mod that keeps prefix tokens globally visible while applying a sliding window to generated tokens. The KV cache manager evicts gap blocks between the prefix and the decode window, reducing memory usage. The model supports up to 32 image crops (vs 6 for DeepSeek-OCR) and disables cropping for multi-image requests. The PR includes comprehensive test updates and documentation.

Low risk. The change is minimal and only affects the detection of the return type. The runtime check is safe and handles both cases. The removed property was only used in one place, so no external code should break. However, if any custom speculator subclass overrode `model_returns_tuple` and relied on it being called, that override will no longer have any effect.

**Key insights**  
The fix addresses a real bug where MTP draft models that return a tuple (for final-norm correctness) caused a crash. The solution is elegant: instead of a static property, use a runtime type check. This is more robust and handles edge cases automatically. The PR also adds unit tests to verify both tuple and single-tensor returns work correctly.

**Key insights**  
The bug was introduced by PR #44577 which added cross-layer layouts for DSV4 but incorrectly assumed all KV cache tensors are layer-packed.
