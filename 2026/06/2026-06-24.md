### Summary

**What changed and why**  
Refactored `ToolParserOutput` from a struct with separate `normal_text` and `calls` fields into a `Vec<ToolParserEvent>` enum sequence to preserve the order of text and tool-call events. Previously, the flattened representation lost ordering information when text appeared both before and after tool calls. The change introduces `ToolParserEvent` (Text or ToolCall) and updates all parser implementations to use `push_text`/`push_call` methods. Backward-compatible accessors `normal_text()`, `calls()`, and `coalesce()` are provided. The PyO3 bridge and Python API are updated accordingly.

**Technical impact**  
Changes the core data structure of tool parser output from a flat struct to an ordered event list. All 32 files are updated, including all tool parser implementations (DeepSeek, GLM, Hermes, InternLM2, Llama, Mistral, etc.), the unified parser, Python bindings, and tests. The `coalesce_calls` method is renamed to `coalesce` and now also flattens text events. The `append` method now merges events while preserving order and coalescing adjacent text events. The `push_text` method now coalesces consecutive text events into one. The `calls()` method returns a vector of references instead of owned values. The `normal_text()` method concatenates all text events. The `ToolParserOutput` struct fields are no longer public; access is through methods.

**Potential risks**  
The change is large and touches many parser implementations. There is a risk of subtle bugs if any parser incorrectly uses the new API (e.g., forgetting to call `push_text`/`push_call`). The `coalesce` method now also flattens text, which may change behavior for callers that relied on the old `coalesce_calls` only merging tool calls. The Python API method `coalesce_calls` is renamed to `coalesce`, which could break external Python code. The `calls()` method now returns references, which may require changes in callers that mutate the returned vector. The `normal_text()` method now returns a `String` instead of `&str`, which could affect performance or borrowing patterns.

**Key insights**  
The refactoring preserves the original order of text and tool-call events, which is essential for correctly handling outputs where text appears both before and after tool calls. The backward-compatible accessors and `coalesce` method allow existing callers to continue working with minimal changes. The test coverage is extensive, with all existing tests passing and new tests added for coalescing behavior. The change is well-structured and follows a clear pattern across all parser implementations.

However, the risk is low given the targeted nature of the optimization and the presence of fallback paths.

**Key insights**  
The PR demonstrates a targeted performance optimization for a specific GPU architecture (gfx1151) by tuning kernel parameters based on empirical benchmarks. Non-CP deployments are unaffected as cp_world_size=1 preserves existing behavior.

**Potential risks**  
Low risk. Changes are localized to three lines where cp_world_size is multiplied, and all existing tests pass. The fix is consistent with the pattern already merged in #39831. A targeted regression test is added to verify the corrected paths under CP.

**Key insights**  
The bug was a systematic oversight in the original CP implementation: every global-to-block conversion must use effective block size (physical * cp_world_size). The fix follows the same pattern as the two conversions already corrected in #39831. The added test directly asserts the correct block counts for store and load under CP, providing regression coverage.

### Summary

**What changed and why**  
In the OAI Triton backend for MXFP4 MoE, the original block scales are swizzled into precision configs and no longer needed by the kernel. This change deletes the now-unused weight_scale parameters to free GPU memory, and adjusts the quant config retrieval to use the swizzled scales from precision configs for Triton backends, falling back to the original scales for other backends.

**Technical impact**  
Reduces GPU memory consumption by approximately 6 GB for the Triton MoE backend by freeing the original MXFP4 block scales after they are swizzled. The change is backward-compatible for non-Triton backends as they continue to use the original scales.

**Potential risks**  
If any code path still accesses the deleted weight_scale parameters after deletion, it will cause a runtime error. However, the deletion occurs after the scales are swizzled and the quant config is built, so the risk is low. The fallback logic ensures non-Triton backends are unaffected.

**Key insights**  
This is a memory optimization specific to the Triton backend where the original scales become dead after swizzling. The change is minimal (deleting two parameters per layer) but yields significant memory savings. The code comments clearly explain the rationale.
