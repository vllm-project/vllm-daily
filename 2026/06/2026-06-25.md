## 7. [[Rust Frontend] Make Granite4 string argument scanning incremental](https://github.com/vllm-project/vllm/pull/46507)


### Base Information

- **PR Number:** #46507
- **Author:** [reidliu41](https://github.com/reidliu41)
- **Merged By:** [BugenZhao](https://github.com/BugenZhao)
- **Merged time:** 2026-06-25 20:54:03
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/46507/files) (4):**
  - `rust/src/parser/Cargo.toml`
  - `rust/src/parser/benches/granite4.rs`
  - `rust/src/parser/src/tool/json/granite4.rs`
  - `rust/src/parser/src/utils.rs`

### Summary

**What changed and why**  
Made Granite4 string argument scanning incremental by introducing a scan state (JsonStringScanState) that tracks the byte offset and escape status across streaming chunks. Previously, the one-shot json_str helper restarted scanning from the beginning on each retry, causing O(n²) time for long string arguments. The new take_json_string function resumes from the last scanned position, and decode_json_str decodes the complete string once the closing quote is found. Added a Criterion benchmark and unit tests for the new functionality.

**Technical impact**  
Reduces time complexity for long string-valued arguments from O(n²) to O(n) when streamed in small chunks. Benchmark shows ~2800x speedup for 64 KiB payload (107.961 ms → 0.038 ms). The change is backward-compatible: emitted arguments remain unchanged. The new scan state is stored in the parser's mode enum, increasing memory usage slightly but negligibly.

**Potential risks**  
The new scan state must be correctly reset between tool calls; the existing code resets mode to Text after each call, which drops the state. Edge cases with deeply nested escapes or invalid UTF-8 in the string could cause incorrect scanning, but the existing tests cover escapes across chunks. The decode_json_str function assumes the raw slice is a complete JSON string; if take_json_string returns a length that doesn't include the closing quote (should not happen), decode would fail.

**Key insights**  
The performance issue was caused by re-parsing the entire buffered string from scratch on each chunk. By making the scanner stateful, we avoid redundant work. The same pattern could be applied to other streaming parsers that handle long strings. The benchmark shows that the new approach scales linearly with input size, which is critical for large tool call arguments.

### Summary

**What changed and why**  
Added support for `use_local_argmax_reduction` in Model Runner V2's `DraftModelSpeculator` to reduce TP communication during greedy draft token generation. Previously, greedy sampling always called `compute_logits()` and all-gathered full vocabulary logits before argmax. Now, when the flag is enabled, it uses `get_top_tokens()` which only communicates O(2*tp_size) tokens instead of O(vocab_size). This change was motivated by performance benchmarks showing ~7.8% throughput improvement and ~8.3% TPOT reduction on a 2-GPU setup.

**Technical impact**  
Reduces inter-GPU communication volume for greedy draft sampling from O(vocab_size) to O(2*tp_size) per token, improving throughput and latency. The change is opt-in via the existing `use_local_argmax_reduction` config flag, defaulting to off to preserve current behavior. Validation ensures compatibility: the flag is rejected for probabilistic sampling and requires the draft model to implement `get_top_tokens()`. All MR V2 speculators inheriting `DraftModelSpeculator` benefit without subclass modifications.

**Potential risks**  
If a draft model claims to implement `get_top_tokens()` but returns incorrect tokens, silent correctness issues could arise. The validation only checks for the method's existence, not its correctness. Additionally, the change only affects greedy sampling; probabilistic sampling remains unchanged. There is a minor risk of regression if the `get_top_tokens()` implementation has different numerical behavior than `compute_logits().argmax()`, though this is unlikely.

**Key insights**  
The PR demonstrates a targeted optimization that leverages an existing config flag to reduce communication overhead in speculative decoding. The single-file change with minimal code addition (+34/-3) yields significant performance gains. The design ensures backward compatibility and easy adoption across multiple speculator types. The benchmark results validate the approach on a realistic setup.
