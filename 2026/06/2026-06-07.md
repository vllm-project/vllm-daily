### Summary

**What changed and why**  
This PR stabilizes ROCm test teardown by adding explicit VRAM release waits and increasing shutdown timeout. In conftest.py, the HF model context manager now calls wait_for_rocm_memory_to_settle() after cleanup, and the VLLMRunner context manager uses a 60s shutdown timeout on ROCm to allow graceful teardown. **Technical impact**  
Reduces flaky OOMs on ROCm (gfx942) by ensuring VRAM is actually freed before subsequent test startup. The unit conversion fix in memory measurement and the minimum threshold prevent premature success of memory wait. **Potential risks**  
Increased test execution time due to waits (up to 240s) and longer shutdown (60s). The minimum threshold of 4 GiB may mask actual memory leaks if they are below that threshold. The changes are specific to ROCm and should not affect CUDA behavior, but any regression in ROCm test stability could be harder to diagnose.

**Key insights**  
ROCm's lazy VRAM reclamation requires explicit waits after model teardown to prevent OOMs in subsequent tests. The unit mismatch (MiB vs bytes) in memory reporting was causing the wait to succeed prematurely. The PR consolidates memory wait logic into a reusable function and applies it consistently across test fixtures.
