- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/38919/files) (1):**
  - `csrc/cache_kernels.cu`

### Summary

**What changed and why**  
The change replaces compile-time branching for `cuMemcpyBatchAsync` with runtime resolution via `cuGetProcAddress`. This addresses two issues: 1) pre-built wheels hard-linking the symbol causing crashes on CUDA drivers <12.8, and 2) compilation errors on CUDA 13.0 due to header macro remapping. The runtime lookup ensures compatibility across driver versions and avoids symbol linkage problems.

**Technical impact**  
The code now dynamically loads `cuMemcpyBatchAsync` at runtime if the CUDA version is ≥12.8, caching the function pointer for efficiency. This eliminates binary dependencies on newer CUDA driver symbols and circumvents CUDA 13.0's header-defined signature changes. The fallback to individual `cudaMemcpyAsync` calls remains for unsupported configurations (older drivers, ROCm).

**Potential risks**  
If `cuGetProcAddress` fails silently (returns null without error), the fallback path is used, which may impact performance for large batch operations but maintains correctness. There's a minor overhead for the first call due to runtime resolution. The static function pointer is not thread-safe during initialization, though this is likely negligible given typical usage patterns.

**Key insights**  
Runtime symbol resolution is a robust solution for CUDA API versioning and driver compatibility issues. Developers should note that the fallback path may be slower for batches, but correctness is preserved. This pattern could be useful for other CUDA APIs with similar versioning challenges. Ensure future changes maintain the thread safety of the static initializer if concurrency becomes a concern.

### Summary

**What changed and why**  
The PR removes the explicit zero initialization of the scale buffer (`torch::stable::zero_(output_s_packed)`) before launching the FP8 quantization kernel. Instead, the kernel now handles zeroing padded indices internally. This optimization eliminates a separate CUDA kernel launch and memory operation, resulting in a measured ~1% end-to-end speedup in decoding for specific workloads.

**Technical impact**  
The kernel now accepts padded dimensions (`num_groups_padded`, `padded_groups_per_row`) and writes zeros to padding positions within the scale buffer. This requires careful index mapping between flat group IDs and 2D (mn, k) coordinates, with validation for valid groups. The atomic OR operation is replaced with direct byte writes, simplifying the scale packing logic while maintaining the required column-major TMA-aligned layout.

**Potential risks**  
The kernel now assumes the output buffer has a specific stride layout ([1, tma_aligned_mn]), which is verified with new checks. If callers provide buffers with different strides, the kernel may write to incorrect memory locations. The padding logic increases kernel complexity, and incorrect `num_scale_elems` calculation could lead to out-of-bounds writes. The removal of the atomic OR operation assumes no concurrent writes to the same scale element, which should hold given the kernel's design.

**Key insights**  
This is a performance optimization that moves initialization logic into the compute kernel, reducing kernel launch overhead. Developers must ensure the scale buffer is allocated with the correct TMA-aligned strides. The comprehensive test suite validates various padding scenarios and includes a "poisoned scales" test to verify padding zeroing, which is crucial for correctness. The change demonstrates how fusing memory operations with compute kernels can yield measurable performance gains in quantization workloads.

**Technical impact**  
This change ensures that FLA kernels properly skip padded entries in CUDA graph block tables, preventing illegal memory accesses that corrupt state for sequence 0. **Potential risks**  
If any valid state indices could be zero (e.g., for sequence 0), this fix might incorrectly skip them, though the description indicates `NULL_BLOCK_ID=0` is reserved for padding. There is a risk of similar issues in other kernels if they rely on the old `PAD_SLOT_ID` assumption. The fix's effectiveness depends on consistent use of `NULL_BLOCK_ID=0` across the codebase.

**Key insights**  
Always validate guard conditions when underlying constants change. The fix highlights the importance of aligning kernel logic with system-wide constants like `NULL_BLOCK_ID`. Developers should audit other kernels for similar assumptions about padding values, especially in performance-critical paths like CUDA graphs.
