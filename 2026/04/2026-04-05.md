### Summary

**What changed and why**  
This PR adds asymmetric INT8 quantization support to the TritonInt8ScaledMMLinearKernel for ROCm, enabling models with asymmetric zero-point quantization (e.g., W8-Channel-A8-Dynamic-Asym-Per-Token). Previously, the kernel only supported symmetric input quantization, which limited compatibility with certain INT8 models on ROCm when AITER is disabled.

**Technical impact**  
The changes extend the kernel to handle both static per-tensor and dynamic per-token asymmetric quantization schemes by precomputing weight column sums (`azp_adj`) during weight loading and applying a zero-point correction after the GEMM operation. This aligns the Triton kernel's math with the existing Cutlass implementation, ensuring consistent behavior across backends and enabling broader model support on ROCm platforms.

**Potential risks**  
The asymmetric path introduces additional branching and tensor operations (e.g., `azp_adj` computation and subtraction), which may slightly impact performance compared to the symmetric case. Edge cases include ensuring proper broadcasting of scales and zero-point tensors, especially for dynamic per-token quantization where `x_zp` is per-token and `azp_adj` is per-channel. Incorrect handling could lead to numerical errors or shape mismatches.

**Key insights**  
Developers should verify that asymmetric quantization is only enabled when `input_symmetric=False`, as the symmetric path remains optimized. The implementation correctly separates static and dynamic quantization logic, but performance profiling is recommended to assess any overhead. Additionally, the added GPU architecture checks in test configurations ensure compatibility but may skip tests on unsupported hardware.

## 9. [[Bugfix][MoE] Fix 6-8% decode regression: prefer multi-stream shared expert overlap](https://github.com/vllm-project/vllm/pull/38990)


### Base Information

- **PR Number:** #38990
- **Author:** [voipmonitor](https://github.com/voipmonitor)

- **[Changed Files](https://github.com/vllm-project/vllm/pull/38990/files) (1):**
  - `vllm/model_executor/layers/fused_moe/runner/shared_experts.py`

### Summary

**What changed and why**  
The fix addresses a 6-8% decode throughput regression in MoE models for TP-only configurations by reordering condition checks in `SharedExperts._determine_shared_experts_order()`. Previously, `_has_external_experts` (renamed to `_use_external_experts`) was evaluated first, forcing sequential execution (`EXTERNAL` order) and preventing multi-stream overlap. Now, the logic prioritizes `MULTI_STREAM_OVERLAPPED` when an auxiliary CUDA stream is available, restoring parallel execution of shared experts.

**Technical impact**  
This change restores performance for TP-only MoE deployments (e.g., DeepSeek-V3, GLM-5) by enabling shared experts to run concurrently with routed experts via an auxiliary stream, matching the behavior of vLLM 0.18.x. The renamed property `_use_external_experts` now explicitly returns `True` only when EPLB with a non-default backend or flashinfer two-sided kernels are active, clarifying its role as a disablement condition for overlap.

**Potential risks**  
If the auxiliary stream management is improperly synchronized, race conditions could arise, though existing usage patterns suggest this is low risk. The change assumes `_use_dp_chunking` correctly indicates when DP chunking is active; incorrect state could inadvertently disable overlap. Edge cases involving EPLB with custom backends or flashinfer two-sided kernels remain forced to sequential execution, which may affect performance in those configurations.

**Key insights**  
Always validate performance-critical condition ordering, especially when introducing new checks that may inadvertently block optimizations. For MoE deployments, ensure TP-only configurations leverage multi-stream overlap to maximize throughput. Developers should verify that `_use_dp_chunking` and backend flags are set correctly for their hardware and parallelization setup to avoid unintended performance penalties.

### Summary

**What changed and why**  
The fix addresses an issue where `ExtractHiddenStatesConfig` incorrectly flattens Vision-Language Model (VLM) configurations, converting nested `text_config` PretrainedConfig objects into plain dictionaries. This breaks downstream validation in `get_hf_text_config()`, which uses `hasattr()` to check for attributes like `num_attention_heads`. The solution preserves the original `PretrainedConfig` object for `text_config` after serialization, ensuring attribute access works while keeping serialization unaffected.

**Technical impact**  
This change ensures VLMs with nested `text_config` (e.g., Kimi-K2.5, Qwen3-VL, LLaVA) function correctly with speculative decoding via `extract_hidden_states`. Text-only models remain unaffected because `get_text_config()` returns `self`, skipping the validation. The fix maintains backward compatibility and does not alter serialization behavior (`to_dict()`/`to_json_string()`).

**Potential risks**  
If `source_text_config` is incorrectly set (e.g., due to a malformed `get_text_config()` implementation), downstream attribute validation could still fail. Additionally, any code relying on `text_config` being a dictionary after `to_dict()` may break, though the serialized output remains unchanged. Edge cases where `text_config` is not a `PretrainedConfig` object could lead to runtime errors.

**Key insights**  
Developers should verify that VLM configurations properly implement `get_text_config()` to return a valid `PretrainedConfig`. The fix highlights the importance of preserving object types during configuration transformations, especially for composite models. Ensure any future changes to `ExtractHiddenStatesConfig` continue to handle both flattened and nested configurations correctly.
