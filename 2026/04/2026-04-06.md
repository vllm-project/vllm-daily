## 3. [[BugFix][MRV2] Fix cuda event reuse race](https://github.com/vllm-project/vllm/pull/39115)


### Base Information

- **Author:** [njhill](https://github.com/njhill)
- **Merged By:** [WoosukKwon](https://github.com/WoosukKwon)

- **[Changed Files](https://github.com/vllm-project/vllm/pull/39115/files) (2):**
  - `vllm/v1/worker/gpu/async_utils.py`
  - `vllm/v1/worker/gpu/model_runner.py`

### Summary

**What changed and why**  
The fix removes the reuse of a single CUDA event (`output_copy_event`) across successive inference steps in Model Runner V2. Previously, the event was shared to avoid creation/destruction overhead, but this caused a race condition where event recording for step n+1 could overwrite step n's completion signal, delaying synchronization.

**Technical impact**  
Each `AsyncSamplingOutput` and `AsyncPoolingOutput` now creates its own CUDA event instance. This eliminates the race condition, ensuring proper synchronization between the copy stream and waiting threads. The `model_runner.py` no longer manages a shared event, simplifying the flow and removing a subtle dependency.

**Potential risks**  
Frequent event creation could introduce minor overhead if inference runs at very high step rates, though the description notes this is negligible. If future pooling is implemented, similar race conditions must be avoided—e.g., by ensuring events are not reused across overlapping steps.

**Key insights**  
Correctness and clarity are prioritized over micro-optimizations. Developers should note that CUDA event reuse requires careful lifecycle management to avoid synchronization hazards. Consider event pooling only with clear guarantees of non-concurrent reuse per step.

**Technical impact**

Edge cases in chunk slicing (e.g., `chunk_start >= num_tokens`) require careful handling, as noted in the TODO.

**Key insights**  
Developers should treat `ChunkingMoERunner` as a transparent wrapper—its `reduce_results` always returns `False` because reduction is handled elsewhere. The factory pattern centralizes runner creation, ensuring DP chunking is applied consistently. When extending MoE runners, ensure new implementations inherit from `MoERunnerBase` and implement the required abstract methods. The removal of `SharedExpertsOrder.EXTERNAL` simplifies the shared experts flow, but the `_replace_quant_method` workaround should be addressed in future cleanup.
