Backward compatibility: existing configurations that relied on the drafter inheriting the target's attention backend may now auto-select a different backend, potentially changing behavior. However, the PR explicitly avoids inheritance to prevent incompatibilities. The new `use_non_causal` flag could affect other parts of the system if not properly handled. The field validator for `attention_backend` accepts 'auto' to set None, which may be unexpected.

**Key insights**  
The PR decouples drafter and target attention backends, addressing a real limitation in speculative decoding. The design choice to never inherit the target's backend (unlike MoE) is justified by common incompatibilities. The test coverage includes both explicit backend selection and auto-selection with non-causal requirement, ensuring the new feature works as intended.

### Summary

**What changed and why**  
This PR modifies the CUDA graph capture flow for speculative decoding (Eagle) in vLLM's Model Runner V2. Previously, attention metadata was rebuilt before replaying the draft prefill CUDA graph, which could cause stale data and crashes. Now, the attention metadata from the target model's capture is reused for the draft prefill, avoiding unnecessary rebuilds. The change introduces a new CapturedAttentionState type to bundle metadata and slot mappings, splits EagleCudaGraphManager into PrefillEagleCudaGraphManager and DecodeEagleCudaGraphManager to make capture contracts explicit, and updates the speculator's capture method to accept pre-built attention states.

**Technical impact**  
The change reduces redundant attention metadata rebuilds during draft prefill replay, improving performance and correctness. It also clarifies the code structure by separating prefill and decode graph managers, making the capture flow more explicit. The return type of CudaGraphManager.capture is changed to return captured attention states, which are then passed to the speculator. This may affect any code that calls capture directly, but the only caller (model_runner.py) is updated accordingly.

**Potential risks**  
The change assumes that the target model's attention metadata is compatible with the draft prefill, which may not hold for all backends or configurations. If the draft model requires different metadata (e.g., different slot mappings), reusing the target's metadata could cause errors. The split of EagleCudaGraphManager into two classes may break external code that directly instantiates EagleCudaGraphManager. Additionally, the new CapturedAttentionState type is a NamedTuple, which is immutable; any code that modifies the metadata after capture will need adjustment.

**Key insights**  
The PR optimizes speculative decoding by avoiding unnecessary attention metadata rebuilds, leveraging the fact that the draft prefill uses the same input shapes as the target prefill. The introduction of CapturedAttentionState and the split of graph managers improve type safety and code clarity. The change is validated across multiple models and benchmarks, indicating correctness and performance gains.
