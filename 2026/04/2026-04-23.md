- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/37892/files) (1):**
  - `csrc/cache_kernels.cu`

### Summary

**What changed and why**  
The PR restricts the concat_mla_q kernel to only support half types (float16 and bfloat16) by replacing VLLM_DISPATCH_FLOATING_TYPES with VLLM_DISPATCH_HALF_TYPES and adding a check that rejects float32. This fixes a bug where float32 caused incorrect rope data copying because the kernel assumed 2-byte elements for the rope section.

**Technical impact**  
The kernel now explicitly rejects float32 inputs, preventing silent data corruption. Performance for bf16/half remains unchanged as shown by benchmarks. The change is minimal (+4/-1 lines) and only affects the dispatch and input validation.

**Potential risks**  
Users relying on float32 for concat_mla_q will now get a runtime error instead of incorrect results. This is a breaking change for float32 usage, but the kernel was previously broken for that dtype, so it's a necessary fix. No other dtypes are affected.

**Key insights**  
The bug stemmed from assuming the rope section size in bytes matched the element size of half (2 bytes) when using int* loads. For float32 (4 bytes), the load size was insufficient. The fix correctly limits support to half types, which is the intended use case for MLA kernels.

## 2. [[XPU][CI] Fix Docker cleanup races on Intel CI runners](https://github.com/vllm-project/vllm/pull/40761)


### Base Information

- **PR Number:** #40761
- **Author:** [zxd1997066](https://github.com/zxd1997066)
- **Merged By:** [jikunshang](https://github.com/jikunshang)
- **Merged time:** 2026-04-23 23:08:45
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/40761/files) (2):**
  - `.buildkite/scripts/hardware_ci/run-intel-test.sh`
  - `.buildkite/scripts/hardware_ci/run-xpu-test.sh`

### Summary

**What changed and why**  
Refactored Docker cleanup in Intel CI scripts to use a file lock and a new function `cleanup_old_ci_images` that selectively removes old or unused images, avoiding races with concurrent jobs. Also removed aggressive image cleanup from the EXIT trap in both Intel and XPU scripts to prevent cleanup during active jobs.

**Technical impact**  
Introduces a file-based lock (`/tmp/docker-pull.lock`) to serialize cleanup and pull operations, reducing CI failures from race conditions. The new cleanup logic is more targeted: it avoids deleting images currently in use or referenced by running containers, and respects a TTL for old images. The EXIT trap no longer removes the job's own image or prunes everything, which prevents accidental deletion of resources needed by other concurrent jobs.

**Potential risks**  
The lock file approach may cause contention if multiple jobs wait for the lock, potentially increasing CI job duration. The cleanup function relies on `docker image inspect` and `date` parsing, which could fail on different Docker versions or date formats. If the lock file is not released due to a crash, subsequent jobs may hang. The removal of `docker system prune` from the trap means disk space might not be reclaimed promptly if a job fails before cleanup.

**Key insights**  
The change addresses a real concurrency issue on shared CI runners by introducing a simple file lock and more granular image cleanup. The new `cleanup_old_ci_images` function is a good pattern for avoiding unnecessary deletions while still freeing space. However, the lock file path is hardcoded and could be improved with a unique per-node identifier to avoid cross-node conflicts if the filesystem is shared.

### Summary

**What changed and why**  
Fix a RuntimeError: Already borrowed crash in concurrent tool-call requests by replacing tokenizer.encode calls with vocab lookups. The root cause was that tool parsers called encode in __init__, which borrows HuggingFace's Rust tokenizer backend, causing a race condition with concurrent tokenizer operations in the renderer's ThreadPoolExecutor. The fix replaces encode with vocab.get, which is thread-safe and does not borrow the backend. Additionally, deadcode was removed from the Gemma function parser, and tests were updated to use real tokenizers.

**Technical impact**  
Eliminates a concurrency bug that caused HTTP 500 errors under concurrent tool-call requests. The change is minimal: only 3 files modified with +37/-38 lines. The fix is backward-compatible as tests pass unchanged. Performance impact is negligible since vocab lookups are fast and the tokenizer is no longer accessed per-request.

**Potential risks**  
Low risk. The change is well-scoped and tested. The only risk is if a tokenizer's vocab does not contain the expected token (e.g., bot_token), which is now handled by a RuntimeError with a clear message. The test updates ensure that real tokenizers are used, so this path is exercised.

**Key insights**  
The fix leverages the fact that vocab is a thread-safe dictionary in HuggingFace tokenizers, while encode is not. By moving token ID lookups to class-level attributes via vocab, the per-request code path avoids any tokenizer access, eliminating the race condition entirely. This is a clean, minimal fix that addresses the root cause without introducing locks or complex caching.

**Key insights**
