### Base Information

- **PR Number:** #39185

  - `tests/v1/kv_connector/unit/offloading_connector/utils.py`

  - `vllm/distributed/kv_transfer/kv_connector/v1/offloading/scheduler.py`
  - `vllm/v1/kv_offload/abstract.py`
  - `vllm/v1/kv_offload/cpu/manager.py`
  - `vllm/v1/kv_offload/reuse_manager.py`

### Summary

**What changed and why**  
This PR adds per-request context (`ReqContext`) to the KV offloading manager interface. The context wraps `request.kv_transfer_params` (a free-form dict from sampling parameters) and is passed through `lookup`, `prepare_load`, and `prepare_store` methods. The change is wiring-only—the context is accepted but not yet used by the CPU offloading manager, laying groundwork for future per-request offloading customization.

**Technical impact**  
The `OffloadingManager` ABC and its implementations (`CPUOffloadingManager`, `FilterReusedOffloadingManager`) now require a `ReqContext` argument, ensuring all offloading operations can be context-aware. The `ReqContext` dataclass is placed in the shared abstract module to avoid CPU-specific imports, promoting reusability across other offloading implementations. The scheduler constructs the context per request and propagates it to manager calls.

**Potential risks**  
The PR introduces a mandatory new parameter to core offloading interfaces, which could break any external or future implementations of `OffloadingManager` that haven’t been updated. Since the context is not yet consumed, there’s a risk of it being ignored inadvertently in future feature development. Test updates are extensive but mostly mechanical; however, any untested code paths might miss the new parameter.

**Key insights**  
Developers implementing custom `OffloadingManager` subclasses must now accept and handle `ReqContext`. The pattern centralizes request-specific metadata (like `kv_transfer_params`) for flexible offloading policies. Ensure all call sites—including any new offloading managers—correctly construct and pass the context. Future features can leverage this context to tailor offloading behavior per request without further interface changes.

### Summary

**What changed and why**  
Added a new method `_adjust_params_for_parsing` to the `LLM` class that automatically sets `skip_special_tokens=False` in `SamplingParams` when structured output parsing is needed (i.e., when `enable_thinking=True` or tools are provided). This ensures special tokens like Gemma 4's `<\|channel>` and `<\|tool_call>` are preserved in `output.text` for correct downstream parsing in offline `LLM.chat` calls.

**Technical impact**  
The change modifies the offline inference pipeline to conditionally preserve special tokens only for models that encode structured output syntax as special tokens (currently hardcoded for Gemma 4). This brings functional parity with the OpenAI server rendering path and ensures reasoning blocks and tool calls can be parsed correctly from raw model output.

**Potential risks**  
The implementation is currently hardcoded to Gemma 4 architectures, which may cause issues if other models use similar special tokens but aren't detected. Additionally, the check relies on the tokenizer's `all_special_ids`, which could vary across tokenizer implementations. There's also a risk of unintended side effects if `skip_special_tokens=False` is set for models where structured tokens are regular text tokens (e.g., DeepSeek).

**Key insights**  
This is a temporary fix pending a broader Renderer refactor. Developers should be aware that the logic is model-specific and may need expansion for future models. Consider abstracting the token detection logic into a configurable registry to support more models without code changes. Ensure thorough testing with all supported models to avoid regressions in token stripping behavior.

  - `csrc/libtorch_stable/torch_bindings.cpp`

### Summary

**What changed and why**  
The fix addresses a build failure on SM_120-only systems by moving the implementation registration of `mxfp4_experts_quant` and `silu_and_mul_mxfp4_experts_quant` ops from `torch_bindings.cpp` to `mxfp4_experts_quant.cu`. This ensures registration is properly guarded by the `ENABLE_NVFP4_SM100` macro, which is only defined when compiling CUDA code for SM_100 architectures. The header declarations were also removed since they're no longer needed in the C++ interface.

**Technical impact**  
This change aligns kernel implementation availability with build configuration, preventing undefined symbol errors during linking when building for SM_120-only targets. The ops remain schema-registered unconditionally, but their implementations are only available on SM_100 systems, maintaining runtime compatibility while fixing the build issue. The architecture now properly separates schema definition from implementation binding.

**Potential risks**  
If the `ENABLE_NVFP4_SM100` macro isn't consistently defined across compilation units, it could lead to registration mismatches. There's also a risk that other parts of the codebase might still reference these functions directly through the removed header declarations, though this seems unlikely given the described architecture.

**Key insights**  
The solution demonstrates good practice by moving implementation-specific registration to the implementation files themselves, where build flags are properly scoped. Developers should note that CUDA compilation flags aren't visible to C++ files, requiring careful placement of guarded registrations. This pattern could be applied to other architecture-specific kernel registrations to maintain clean separation.

## 5. [[Attention] TurboQuant: remove redundant random signs, add prior art attribution](https://github.com/vllm-project/vllm/pull/40194)


### Base Information

- **PR Number:** #40194
- **Author:** [dalistarh](https://github.com/dalistarh)
- **Merged By:** [mgoin](https://github.com/mgoin)
- **Merged time:** 2026-04-18 11:31:59
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/40194/files) (6):**
  - `tests/quantization/test_turboquant.py`
  - `vllm/model_executor/layers/attention/attention.py`
  - `vllm/model_executor/layers/quantization/turboquant/__init__.py`
  - `vllm/model_executor/layers/quantization/turboquant/config.py`
  - `vllm/model_executor/layers/quantization/turboquant/quantizer.py`
  - `vllm/v1/attention/backends/turboquant_attn.py`

### Summary

**What changed and why**  
The PR removes per-layer random sign flips from the Hadamard rotation in TurboQuant, replacing them with a pure, deterministic Hadamard matrix shared across all layers. This is because the random signs were theoretically redundant for Lloyd-Max quantization (which is symmetric around zero) and added unnecessary variance. Additionally, the PR adds attribution to prior art (HIGGS and "Cache Me If You Must") that pre-date the TurboQuant paper, clarifying the technique's origins.

**Technical impact**  
This simplifies the codebase by removing the `_tq_signs` buffer, `generate_wht_signs()` function, per-layer seed logic, and the `seed` field from `TurboQuantConfig` (though the field remains for backward compatibility). The rotation matrix is now a pure, symmetric Hadamard matrix (`H = H^T`), making the forward and inverse rotations identical. This reduces runtime overhead and memory usage slightly.

**Potential risks**  
The main risk is ensuring all references to the removed components are cleaned up, though the PR includes verification steps for this. There is a theoretical risk if any downstream code or external integrations relied on the per-layer randomization for non-quantization purposes, but the empirical validation shows no regression in quantization quality or model perplexity.

**Key insights**  
The change is a net positive, removing unnecessary complexity without impacting performance. Developers should note that the `seed` config parameter is now a no-op but retained for compatibility. The updated documentation correctly attributes the core technique to earlier work, which is important for academic integrity. All tests have been updated to reflect the pure Hadamard approach.
