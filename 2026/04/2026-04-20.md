- **[Changed Files](https://github.com/vllm-project/vllm/pull/39703/files) (1):**
  - `vllm/v1/attention/backends/rocm_aiter_fa.py`

### Summary

**What changed and why**  
Added DFlash speculative decoding support for ROCm by extending the AiterFlashAttention backend. The change propagates the `causal` flag through metadata and adds a non-causal attention path using `flash_attn_with_kvcache` for multi-token decode, while keeping the existing `unified_attention` path for causal cases. This mirrors the existing NVIDIA DFlash support.

**Technical impact**  
Enables DFlash speculative decoding on AMD MI300 GPUs, achieving 1.1x-3.9x speedups across various benchmarks. The change modifies the attention backend to conditionally select between two kernels based on the `causal` flag, and adds a `supports_non_causal` class method. It also introduces a dependency on `aiter.ops.triton.attention.mha_v3` for non-causal attention.

**Potential risks**  
The new `flash_attn_with_kvcache` kernel does not support CUDA graphs (known issue, fix pending). The non-causal path reshapes tensors and copies output, which may introduce overhead. The fallback to `unified_attention` for causal cases is preserved, but the code path selection adds complexity. The change is limited to the ROCm backend and should not affect NVIDIA or other backends.

**Key insights**  
The PR demonstrates a clean extension of the existing attention backend to support non-causal attention for speculative decoding on ROCm. The benchmark results show significant speedups, validating the approach. The known CUDA graph limitation is documented and being addressed by the AITER team. The change is focused and well-contained within a single file.

## 6. [[Startup] Parallelize torch/transformers import + weight prefetch + forkserver prewarm](https://github.com/vllm-project/vllm/pull/40331)


### Base Information

- **PR Number:** #40331
- **Author:** [simon-mo](https://github.com/simon-mo)
- **Merged By:** [chaunceyjiang](https://github.com/chaunceyjiang)
- **Merged time:** 2026-04-20 19:49:33
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/40331/files) (3):**
  - `vllm/entrypoints/cli/main.py`
  - `vllm/entrypoints/openai/api_server.py`
  - `vllm/envs.py`

### Summary

**What changed and why**  
Five overlapping startup optimizations: (1) parent-side weight shard page-cache prefetch via background thread, (2) background torch .so loading at CLI entry, (3) pre-spawn forkserver with preloaded async_llm for vllm serve, (4) background transformers import chained after torch, (5) prefetch tokenizer/config sidecar files. All aim to overlap I/O-bound work with main-thread computation to reduce cold-start latency by 11-20%.

**Technical impact**  
Cold-start time reduced by 9-19 seconds (11-20%) for tested models; warm-start also improves by ~3 seconds (5-7%). Introduces background threads for I/O-bound operations (torch/transformers imports, file reads, forkserver prewarm) that release the GIL, enabling overlap with main-thread argparse/config resolution. Adds forkserver multiprocessing method as an option for vllm serve, reducing child process spawn cost.

**Potential risks**  
Background threads may cause resource contention on memory-constrained systems (e.g., weight prefetch reading large files). Forkserver prewarm adds fixed setup cost for non-serve commands (mitigated by gating on sys.argv). Importing torch/transformers on a background thread could mask import errors (caught and suppressed). Thread safety relies on Python's import lock; edge cases with custom import hooks or monkey-patching could cause issues. The prefetch is best-effort and falls back to existing behavior on failure.

**Key insights**  
Overlapping I/O-bound startup work with main-thread computation yields significant cold-start improvements without changing runtime behavior. Background threads are sufficient for GIL-releasing operations, avoiding subprocess overhead. The forkserver prewarm is a novel application of Python's stdlib multiprocessing feature to reduce child process startup time. All changes are behavior-preserving and best-effort, ensuring no regression on failure.
