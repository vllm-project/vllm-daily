  - `vllm/v1/engine/core.py`
  - `vllm/v1/worker/gpu_model_runner.py`
  - `vllm/v1/worker/gpu_worker.py`

### Summary

**What changed and why**  
Fixed GPU memory leaks in engine shutdown for in-process mode by: (1) unwrapping bound methods in supports_kw to avoid caching model instances, (2) adding gc.unfreeze() in EngineCore.shutdown(), (3) adding explicit GPU resource teardown in GPUWorker.shutdown() and GPUModelRunner.shutdown(). Also added a test workaround for async KV prefix cache reset by sending dummy prefills to force engine steps, and bumped UCX branch to fix a memory leak. Added regression tests for GPU memory release.

**Technical impact**  
Resolves GPU memory leaks that occurred when running vLLM in-process (VLLM_ENABLE_V1_MULTIPROCESSING=0). The supports_kw LRU cache no longer pins model instances, gc.freeze() is properly undone, and GPU tensors (model weights, KV caches, rotary embeddings, workspace) are explicitly freed on shutdown. The test workaround ensures prefix cache reset can complete even when async offload transfers are in flight. UCX memory leak fix reduces driver-level memory retention.

**Potential risks**  
The supports_kw change may affect performance if the cache is now less effective (though bound methods were likely rare). The gc.unfreeze() call could cause a slight increase in GC overhead during shutdown. The GPUModelRunner.shutdown() method clears static_forward_context and _ROPE_DICT, which could affect subsequent engine reuse if not reinitialized. The test workaround sends dummy prefills which may slightly alter test timing or resource usage. UCX branch bump may introduce new issues if the new commit has regressions.

**Key insights**  
Bound method caching in lru_cache is a subtle source of memory leaks because it holds references to the instance. gc.freeze() without corresponding gc.unfreeze() makes all objects invisible to GC, causing leaks. Explicit teardown of GPU resources is necessary for in-process engine reuse. The test workaround demonstrates that engine stepping is required to drain async transfers, which is a design consideration for future async operations.

## 2. [[Feature] Warm up readonly multimodal processor during renderer startup](https://github.com/vllm-project/vllm/pull/40797)

- **PR Number:** #40797
- **Author:** [fake0fan](https://github.com/fake0fan)

  - `tests/entrypoints/serve/tokenize/test_serving_tokenization.py`
  - `tests/renderers/test_warmup.py`
  - `vllm/entrypoints/serve/render/serving.py`
  - `vllm/renderers/base.py`

### Summary

**What changed and why**  
This PR adds warmup for the readonly multimodal processor during renderer startup and fixes routing so that engine-executing requests no longer incorrectly use the readonly processor. Previously, render_chat() and render_completion() forced skip_mm_cache=True, causing normal engine-backed requests to go through the readonly processor, and the readonly processor was not warmed up, leading to cold-start latency on first renderer-only request. The change threads skip_mm_cache parameter through these methods, defaults it to False for engine-executing paths, and adds warmup for the readonly processor.

**Technical impact**  
Engine-executing paths (chat/completions, disaggregated generate) now correctly use the main multimodal cache path, improving performance and correctness. Renderer-only paths (render, tokenize) continue to use skip_mm_cache=True and benefit from readonly processor warmup, reducing first-request latency from ~8s to ~0.04s. Startup time increases slightly due to additional warmup step (from ~9.3s to ~8.7s + ~9.0s).

**Potential risks**  
The change modifies core routing logic; any missed call site that still relies on old behavior could cause incorrect cache usage. The warmup code is guarded by try/except, so failures are non-fatal but could lead to cold-start if the readonly processor warmup fails silently. The new tests cover the main paths but may not catch all edge cases (e.g., multimodal models with no readonly processor).

**Key insights**  
The root cause was two separate issues: missing warmup for readonly processor and incorrect skip_mm_cache default. The fix cleanly separates concerns by making skip_mm_cache explicit and adding dedicated warmup. The performance improvement for renderer-only requests is dramatic (8s to 0.04s). The PR also adds comprehensive regression tests for chat, completion, disaggregated, and tokenization paths.

**Key insights**  
The previous modulo-based selection caused systematic bias when top_k is a multiple of R because consecutive tokens from the same expert would land on the same replica. The Knuth hash breaks this pattern by distributing tokens more uniformly. The fix is minimal and leverages a classic multiplicative hash constant.
