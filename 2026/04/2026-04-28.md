## 2. [[KV Offload] Per-job store completion for CPU offloading connector](https://github.com/vllm-project/vllm/pull/39186)

  - `tests/v1/kv_connector/unit/utils.py`
  - `vllm/distributed/kv_transfer/kv_connector/v1/offloading/common.py`
  - `vllm/distributed/kv_transfer/kv_connector/v1/offloading/scheduler.py`
  - `vllm/distributed/kv_transfer/kv_connector/v1/offloading/worker.py`
  - `vllm/distributed/kv_transfer/kv_connector/v1/offloading_connector.py`

### Summary

**What changed and why**  
Introduces per-job store completion tracking for the CPU offloading connector. Previously, the scheduler only learned about store completions when a request finished (EOS), leaving offloaded blocks invisible for many steps. This change reports store completions per-job per-step via a new OffloadingWorkerMetadata that aggregates completed job IDs across workers. The scheduler uses a fence mechanism (_block_id_to_pending_jobs) to delay reuse of blocks whose store is still in-flight, and processes completions immediately so offloaded blocks become loadable sooner.

**Technical impact**  
Adds a new dataclass OffloadingWorkerMetadata with a completed_jobs dictionary and aggregate method. The scheduler now maintains a job counter, a _jobs dict mapping job IDs to TransferJobStatus (with pending_count, keys, is_store, gpu_block_ids), and a _block_id_to_pending_jobs dict for fence tracking. The worker no longer emits finished_sending for stores; instead, it populates completed_jobs in the metadata, which the scheduler processes via update_connector_output. The OffloadingConnectorMetadata now uses job IDs instead of request IDs for load/store jobs and flush sets. The test utilities are updated to pass worker metadata through the model runner output. 8 files changed, +430/-145 lines.

**Potential risks**  
The fence mechanism introduces complexity: if a block's pending store is not properly tracked, reuse could lead to data corruption. The new per-job completion path changes the contract between worker and scheduler; any mismatch in job ID handling could cause hangs or missed completions. The change assumes all workers report completion for the same job IDs; if a worker fails or is slow, the scheduler may stall. The test coverage is good but may not cover all edge cases (e.g., concurrent stores for the same block from different requests).

**Key insights**  
The key insight is that by tracking store completions per-job and per-step, offloaded blocks become available for reuse immediately after their DMA finishes, rather than waiting for the entire request to complete. This reduces the window where blocks are invisible, improving prefix cache hit rates and reducing recomputation. The fence mechanism ensures correctness by preventing reuse of blocks with in-flight stores, while the per-job tracking allows fine-grained completion handling.

  - `tests/compile/test_config.py`
  - `vllm/config/vllm.py`

### Summary

**What changed and why**  
Added logic to ensure that max_num_batched_tokens is included in cudagraph capture sizes even when it is not a multiple of the stride (8 or 16). Previously, only batch sizes landing on multiples of 8 or 16 were captured, causing a fallback to eager mode at peak batch size and significant performance degradation. This fix appends max_num_batched_tokens to the capture sizes list if it is within max_cudagraph_capture_size and not already present.

**Technical impact**  
Improves performance when max_num_batched_tokens is not a multiple of the cudagraph stride (e.g., 700). Without this fix, the model falls back to eager mode at max batch size, causing up to 3x throughput degradation. With the fix, cudagraph captures the exact max batch size, maintaining high performance.

**Potential risks**  
Low risk. The change only adds one extra size to the capture list if conditions are met, and the list is deduplicated and sorted. No existing behavior is altered. However, if max_num_batched_tokens is very large (close to max_cudagraph_capture_size), it may increase memory usage slightly due to an additional cudagraph capture, but this is negligible.

**Key insights**  
The bug was that cudagraph capture sizes were generated with fixed strides (8 and 16), so off-stride max batch sizes were missed. The fix ensures that the exact max_num_batched_tokens is always captured if it fits within the capture limit, preventing a performance cliff at peak load. The test case added verifies this behavior.
