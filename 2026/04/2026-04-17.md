The fix adds zero bias initialization for `k_proj` in GLM-ASR's attention layer. GLM-ASR originally only uses bias for `q_proj` and `v_proj`, but when `QKVParallelLinear` fuses these projections, the missing `k_proj` bias leaves uninitialized memory that can contain garbage values (e.g., 1e30 or NaN), causing overflows. The change ensures the bias tensor is properly zeroed during weight loading.

**Technical impact**  
This aligns GLM-ASR with the expected behavior of fused QKV projections, preventing numerical instability from uninitialized memory. The modification is minimal and localized, using a helper function (`_create_fake_bias_for_k_proj`) to inject a zero bias tensor for `k_proj` when loading weights, ensuring consistency across the fused parameter.

**Potential risks**  
If the helper function `_create_fake_bias_for_k_proj` has unhandled edge cases (e.g., incorrect tensor shapes or missing weight keys), it could lead to shape mismatches or silent failures. Additionally, any future changes to the fusion logic or weight loading process must maintain compatibility with this injected bias.

**Key insights**  
Developers should verify that similar issues don't exist in other models using fused linear projections. The fix demonstrates the importance of ensuring all components of fused parameters are properly initialized, even when the original model configuration omits them. Consider adding unit tests for weight loading with missing biases to catch such issues early.

- **PR Number:** #39617
- **Author:** [orozery](https://github.com/orozery)
- **Merged By:** [orozery](https://github.com/orozery)
- **Merged time:** 2026-04-17 05:13:55
- **Type:** `None`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/39617/files) (1):**
  - `vllm/v1/kv_offload/cpu/spec.py`

### Summary

**What changed and why**  
The PR fixes how `kv_bytes_per_block` is calculated for `UniformTypeKVCacheSpecs`. Previously, it derived this from the page size and number of tensors. Now, it calculates it directly from the total GPU KV cache size and the configured number of blocks (`kv_cache_config.num_blocks`), making the computation more accurate and directly tied to the actual allocated GPU memory structure.

**Technical impact**  
This change aligns the CPU block size calculation with the actual GPU memory layout by using the total allocated GPU KV cache bytes and the number of blocks. It ensures consistency between CPU and GPU block representations, which is crucial for correct offloading behavior. The fallback to `kv_bytes_per_block = 0` when `num_blocks` is zero handles edge cases gracefully.

**Potential risks**  
If `kv_cache_config.num_blocks` is zero but offloading is still expected to occur, the zero `kv_bytes_per_block` could lead to division-by-zero errors or incorrect `num_blocks` calculations later. The logic assumes `total_gpu_kv_bytes` is evenly divisible by `num_blocks`, which may not hold if block sizes are non-uniform.

**Key insights**  
The fix correctly bases CPU block sizing on actual GPU allocation metrics. Developers should verify that `kv_cache_config.num_blocks` is always positive when offloading is active and ensure block size uniformity aligns with this calculation. Consider adding validation for the divisibility assumption or using integer division carefully.

- **[Changed Files](https://github.com/vllm-project/vllm/pull/40114/files) (1):**
  - `.github/workflows/new_pr_bot.yml`

### Summary

**What changed and why**  
The PR modifies the GitHub Actions workflow for the new PR bot to change the trigger condition from checking if the author has exactly 1 authored PR to checking if they have 0 merged PRs. This addresses a race condition where AI agents could open multiple PRs before the workflow runs, incorrectly skipping the welcome comment.

**Technical impact**  
The bot's logic now correctly identifies first-time contributors based on merge history rather than PR creation count. This ensures the welcome comment is posted only for authors without any prior merged contributions, making the system more resilient to rapid, automated PR submissions.

**Potential risks**  
If an author has open PRs that are in review but not yet merged, they will still receive the welcome comment on new PRs, which may be redundant. There's also a minor risk if the GitHub search API has latency in reflecting recently merged PRs, though this is unlikely.

**Key insights**  
This change improves the accuracy of contributor recognition by focusing on successful contributions (merged PRs). Developers should ensure the `is:merged` filter is reliable for their repository's workflow and consider if pending PRs from trusted contributors should also skip the bot.
