  - `vllm/model_executor/layers/quantization/utils/petit_utils.py`
  - `vllm/platforms/rocm.py`

### Summary

**What changed and why**  
This PR removes the Petit NVFP4 quantization backend from vLLM. The change follows a deprecation notice in version 0.14, indicating that this experimental AMD GPU-specific FP4 quantization support is being phased out.

**Technical impact**  
The removal eliminates support for NVFP4-quantized models on AMD GPUs via the Petit kernel library. All related configuration entries, linear method classes, utility functions, and platform-specific registrations have been deleted. Users relying on this quantization method will need to switch to alternative quantization backends.

**Potential risks**  
Existing users with NVFP4-quantized models will encounter errors when attempting to load them, as the quantization method is no longer recognized. There is a risk of breaking workflows for AMD GPU users who depended on this specific FP4 implementation, especially if no direct replacement is available.

**Key insights**  
Developers should update documentation and migration guides to inform users of this removal. Consider whether alternative FP4 quantization methods (e.g., ModelOpt FP4) are suitable replacements for AMD hardware. Ensure any dependent CI configurations or model examples are updated to reflect the change.

The changes remove specialized Gemma RMS norm optimizations and revert to a more uniform implementation. ### Summary

**What changed and why**  
This PR registers a new IR kernel for Gemma RMS normalization by modifying the RMS norm implementation across multiple kernel backends (AITER, vLLM C, XPU). The changes ensure consistent dtype handling and remove custom compiled forward methods in favor of using the unified IR operation.

**Technical impact**  
The implementation now uses a single `ir.ops.rms_norm` call instead of separate static methods, simplifying the codebase and improving maintainability. **Key insights**  
The refactoring centralizes RMS norm logic but requires careful validation of dtype compatibility across all supported hardware backends. Developers should verify that models using RMS norm layers maintain correct dtype propagation, especially when using residual connections or mixed precision training.

## 4. [[Perf][GDN] Align TMA usage with upstream FLA](https://github.com/vllm-project/vllm/pull/38981)


### Base Information

- **PR Number:** #38981
- **Author:** [arpera](https://github.com/arpera)
- **Merged By:** [ZJY0516](https://github.com/ZJY0516)
- **Merged time:** 2026-04-04 09:38:02
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/38981/files) (1):**
  - `vllm/model_executor/layers/fla/ops/utils.py`

### Summary

**What changed and why**  
The change modifies the `is_tma_supported` flag in vLLM's GDN kernels to align with upstream Flash Linear Attention (FLA) behavior. Previously, TMA (Tensor Memory Accelerator) was unconditionally enabled on SM90+ (Hopper/Blackwell) GPUs. Now, TMA is disabled by default and requires the environment variable `FLA_USE_TMA=1` to enable, matching upstream FLA's decision due to Triton compiler issues on Blackwell (SM100+).

**Technical impact**  
This change ensures vLLM's GDN prefill kernels (`solve_tril`) follow the same code path as upstream FLA, preventing performance degradation on B200 GPUs. The update improves kernel performance by 5–20% in most configurations, bringing vLLM's performance in line with or better than FLA's reference implementation. The change is backward-compatible and only affects TMA-enabled hardware when the environment variable is not set.

**Potential risks**  
If users rely on TMA for optimal performance on Hopper GPUs (e.g., H100), they must now explicitly set `FLA_USE_TMA=1`. The change introduces a dependency on the `is_nvidia_hopper` variable (not shown in diff), which must be correctly defined to avoid runtime errors. There is a minor risk of regression if the environment variable check is misconfigured or if TMA becomes stable on Blackwell in future Triton versions.

**Key insights**  
Developers should be aware that TMA is now opt-in for Hopper/Blackwell GPUs. For B200 users, this change is critical to avoid performance penalties. Consider documenting the `FLA_USE_TMA` environment variable in performance tuning guides. Ensure `is_nvidia_hopper` is properly implemented to avoid breaking the flag on non-Hopper NVIDIA GPUs.
