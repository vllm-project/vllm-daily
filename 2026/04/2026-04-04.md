### Summary

**What changed and why**  
This PR removes the Petit NVFP4 quantization backend from the vLLM codebase. The change follows a deprecation notice in version 0.14, indicating that this experimental AMD GPU-specific FP4 quantization support is being phased out.

**Technical impact**  
The removal eliminates support for NVFP4 quantized models on AMD GPUs via the Petit kernel library. All related configuration entries, import mappings, linear method definitions, and utility functions have been deleted. Users relying on this quantization method will no longer be able to load or run models with `quantization="petit_nvfp4"`.

**Potential risks**  
Existing users with NVFP4 quantized checkpoints will encounter errors when attempting to load models, as the quantization method is no longer recognized. There is also a risk of breaking any downstream integrations or scripts that explicitly referenced the removed Petit classes or configuration flags.

**Key insights**  
Developers should migrate from Petit NVFP4 to alternative quantization methods supported on AMD GPUs. The removal simplifies the codebase by eliminating a platform-specific experimental feature. Ensure documentation is updated to reflect that this quantization option is no longer available.

### Summary

**What changed and why**
This PR registers a new gemma_rms_norm IR kernel by refactoring RMS normalization implementations across multiple kernel backends. The changes unify dtype handling in the base IR operation and add consistency checks for weight tensor dtype matching across CUDA, XPU, and AITER kernel implementations.

**Technical impact**
The refactoring simplifies the RMS norm implementation by removing redundant static methods and consolidating on the `ir.ops.rms_norm` primitive. **Key insights**
The changes represent a positive consolidation of normalization logic, but developers should verify that their models use consistent dtypes for weights and inputs. The performance impact of removing Torch Compile should be evaluated, particularly for models that previously benefited from graph compilation. The simplified code structure will be easier to maintain but requires thorough testing across different hardware platforms to ensure backward compatibility.

## 4. [[Perf][GDN] Align TMA usage with upstream FLA](https://github.com/vllm-project/vllm/pull/38981)


### Base Information

- **PR Number:** #38981
- **Author:** [arpera](https://github.com/arpera)
- **Merged By:** [ZJY0516](https://github.com/ZJY0516)
- **Merged time:** 2026-04-04 09:38:02
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/38981/files) (1):**
  - `vllm/model_executor/layers/fla/ops/utils.py`

### Summary

**What changed and why**  
The PR aligns vLLM's Gated Delta Rule (GDN) kernel behavior with upstream Flash Linear Attention (FLA) by modifying the `is_tma_supported` flag. Previously, Tensor Memory Access (TMA) was unconditionally enabled on SM90+ (Hopper/Blackwell) GPUs, but upstream FLA disabled it by default due to Triton compiler issues on Blackwell (SM100+). Now TMA requires the environment variable `FLA_USE_TMA=1` to be enabled, matching upstream's opt-in approach.

**Technical impact**  
This change ensures vLLM's GDN prefill kernels (`solve_tril`) follow the same code path as the reference FLA implementation on B200 (Blackwell) GPUs. Performance benchmarks show that disabling TMA by default eliminates a 5–20% slowdown, bringing vLLM's performance in line with or slightly better than FLA across most configurations. The modification affects kernel selection logic but does not alter the core algorithm.

**Potential risks**  
Users who previously relied on TMA for performance on Hopper GPUs (e.g., H100) may experience a regression unless they explicitly set `FLA_USE_TMA=1`. The change assumes that the Triton compiler issues on Blackwell are the primary concern, but it may inadvertently affect Hopper performance if TMA was beneficial there. Additionally, environment variable dependencies introduce configuration complexity and potential inconsistency across deployments.

**Key insights**  
Developers should be aware that TMA is now opt-in via `FLA_USE_TMA=1` for all SM90+ GPUs, not just Blackwell. This aligns vLLM with upstream FLA but may require adjustments for users optimizing for Hopper architectures. The performance improvements on B200 are significant, but testing on H100 is recommended to ensure no unintended regressions. Consider documenting this change in release notes due to its user-facing impact.

  - `vllm/platforms/cpu.py`
  - `vllm/utils/ompmultiprocessing.py`

### Summary

**What changed and why**  
This PR fixes macOS compatibility issues introduced by PR #36487, which added Linux-specific API calls (`os.sched_getaffinity` and `lscpu`) that broke CPU support on macOS. The changes introduce platform-aware fallbacks: using `os.cpu_count()` when `sched_getaffinity` is unavailable and synthesizing a simple CPU topology JSON on non-Linux systems instead of calling `lscpu`.

**Technical impact**  
The modifications restore cross-platform compatibility for CPU-based execution, particularly for macOS (Darwin), while maintaining existing Linux behavior. The architecture now cleanly abstracts platform-specific details behind helper functions (`_get_default_affinity`, `_get_cpu_topology_json`), improving code maintainability and reducing duplication.

**Potential risks**  
The synthesized topology on non-Linux platforms assumes a single NUMA node and one core per logical CPU, which may not accurately reflect actual hardware topology on some systems (e.g., multi-socket macOS hardware). This could lead to suboptimal thread placement or performance. Additionally, the fallback logic depends on `os.cpu_count()`, which may return `None` in edge cases.

**Key insights**  
Always use platform checks or capability detection when calling OS-specific APIs. The helper functions centralize platform logic, making future ports to other OSes (e.g., Windows) easier. Consider adding a warning log when using the synthetic topology to inform users of potential performance implications.
