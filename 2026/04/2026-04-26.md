## 1. [[Attention][TurboQuant] Share dequant buffers, eliminate float16_copy](https://github.com/vllm-project/vllm/pull/40941)


### Base Information

- **PR Number:** #40941
- **Author:** [bhoomit](https://github.com/bhoomit)
- **Merged By:** [DarkLight1337](https://github.com/DarkLight1337)
- **Merged time:** 2026-04-26 22:48:36
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/40941/files) (4):**
  - `vllm/model_executor/layers/attention/attention.py`
  - `vllm/v1/attention/backends/turboquant_attn.py`
  - `vllm/v1/attention/ops/triton_decode_attention.py`
  - `vllm/v1/attention/ops/triton_turboquant_decode.py`

### Summary

**What changed and why**  
This PR optimizes the TurboQuant attention backend by sharing dequant buffers via WorkspaceManager in the continuation prefill path, eliminating the float16_copy kernel in decode, enabling CUDA Graph capture for continuation prefill, and adding several prefill/decode optimizations (removing redundant zero_(), using fp16 Hadamard for rotation, pre-allocating K/V buffers, caching arange and cu_seqlens). These changes reduce memory usage (up to 57.6 GB at 1M context), improve prefill latency (-14% TTFT at 15K context), and fix a dtype mismatch in the shared decode buffer path.

**Technical impact**  
Memory: Shared dequant buffers reduce continuation-prefill memory from O(layers × context) to O(context), saving up to 57.6 GB at 1M context. Performance: Prefill latency reduced by 14% at 15K context due to pre-allocated buffers and fp16 rotation. Decode: Eliminated float16_copy kernel (240K fewer launches per profile window) but throughput unchanged within noise. CUDA Graphs: Enables CG capture for continuation prefill by using WorkspaceManager pre-allocation instead of per-layer lazy growth. Accuracy: Unchanged (pure computation/memory optimizations).

**Potential risks**  
WorkspaceManager must be initialized before use; fallback to per-layer buffers is removed, so if workspace is unavailable, decode buffers may be None and cause fallback to kernel-internal allocation (already handled). The fp16 Hadamard rotation assumes entries are perfectly representable in fp16; if not, precision loss could occur, but the PR states it's safe. The pre-allocated K/V buffers in continuation prefill assume seq_len is known; if seq_len exceeds allocation, it could cause out-of-bounds writes. The arange cache may grow unboundedly if max_seq_len increases across calls; currently it only grows when needed. The cu_seqlens reuse assumes single-request prefill; multi-request prefill would need different handling.

**Key insights**  
The main insight is that per-layer buffer allocation for dequantized K/V scales poorly with context length and prevents CUDA Graph capture. Using a shared workspace pool solves both issues. Additionally, many small optimizations (removing zero_(), using fp16 matmul, pre-allocating buffers, caching tensors) compound to reduce latency. The float16_copy elimination is a minor but clean improvement. The PR builds on a previous shared-decode-buffer PR and extends the pattern to the prefill path.

**Technical impact**  
Adds ~1830 lines of new code across 18 files, including new Triton kernels, config validation, attention layer modifications, and extensive test coverage. The feature is opt-in and disabled by default. Performance improvements are observed only for large image token sequences (QHD/4K resolutions) on supported hardware (GB200/GB300 with cuDNN >= 9.17.1).

**Potential risks**

The new code paths could introduce bugs in edge cases (e.g., non-contiguous tensors, head_dim not multiple of 16).

**Key insights**  
The PR uses a circular buffer of amax history for dynamic scaling, which avoids GPU-CPU synchronization during forward passes. The Triton kernel for head_dim padding is stride-aware, handling non-contiguous QKV views without extra copies. The auto-save feature allows calibration-once-reuse workflow, reducing overhead in production. The PR includes thorough performance benchmarks and accuracy validation on ChartQA.

  - `vllm/v1/core/sched/scheduler.py`
  - `vllm/v1/core/single_type_kv_cache_manager.py`
  - `vllm/v1/kv_cache_interface.py`
  - `vllm/v1/simple_kv_offload/manager.py`

### Summary

**What changed and why**  
Introduced a single-source-of-truth method `max_admission_blocks_per_request` on `SlidingWindowSpec` and `ChunkedLocalAttentionSpec` to unify startup pool sizing and runtime admission gating. Previously, the two formulas drifted, causing scheduler stalls and head-of-line blocking on hybrid full+SWA models when the pool was sized at the startup minimum. The change refactors `max_memory_usage_bytes` to call the new method and plumbs `max_num_batched_tokens` through the coordinator and manager layers so the runtime admission gate uses the same recycling-aware bound.

**Technical impact**  
Fixes a critical bug where runtime admission rejected long-but-fittable prompts on hybrid sliding-window models, leading to scheduler deadlocks. The change ensures that per-request block reservations at runtime match the startup pool sizing, preventing both false rejections and potential OOM. It also extends coverage to `ChunkedLocalAttentionSpec` which was previously unaddressed. The plumbing of `max_num_batched_tokens` through multiple classes is a moderate but necessary refactor.

**Potential risks**  
The change touches several core components (coordinator, manager, scheduler) and introduces a new parameter `max_num_batched_tokens` that defaults to `max_model_len` for backward compatibility. If any caller omits the new parameter, the cap collapses to the prior uncapped behavior, which is safe but may not fix the bug for those callers. The new tests cover the hybrid case but not all edge cases (e.g., chunked-local with varying chunk sizes). There is a risk of regression if the `max_admission_blocks_per_request` formula does not exactly match the startup sizer's logic, but the refactoring ensures they share the same code path.

**Key insights**  
The root cause was a drift between two independently maintained formulas for the same recycling-aware bound. The fix enforces a single source of truth by adding a method on the spec dataclass, which is called from both startup and runtime paths. This design prevents future drift and is extensible to other attention types. The PR also includes a clear safety argument and a reproducible test case for the head-of-line blocking scenario.

  - `vllm/v1/worker/gpu/spec_decode/rejection_sampler.py`

### Summary

**What changed and why**  
This PR modifies the rejection sampling in MRV2 to match MRV1's behavior by introducing a 'draft_sample_method' parameter. When set to 'greedy' (default), draft tokens are sampled via argmax and the draft distribution is treated as one-hot, achieving acceptance rates on par with MRV1 without storing full draft logits. The previous 'strict' and 'probabilistic' methods are replaced with 'standard' and 'synthetic'. The Triton kernels are updated to conditionally handle the presence of draft logits, and the resampling logic is adjusted for the one-hot case.

**Technical impact**  
The change improves acceptance rates for MRV2, especially at temperature > 0, bringing them in line with MRV1. It reduces memory usage when using greedy draft sampling (no draft logits stored). The refactoring simplifies the rejection sampler by removing the strict method and consolidating logic. The new draft_sample_method parameter provides flexibility for future work.

**Potential risks**  
The changes to Triton kernels (e.g., renaming _compute_block_max_and_sumexp_kernel to _compute_block_stats_kernel, adding HAS_DRAFT_LOGITS conditionals) could introduce bugs if not thoroughly tested. The removal of strict rejection sampling may break existing configurations that relied on it. The default change from 'strict' to 'standard' could alter behavior for users who upgrade without adjusting config.

**Key insights**  
The PR achieves parity with MRV1's rejection sampling by using a one-hot approximation of the draft distribution, which is both memory-efficient and yields identical acceptance rates. The conditional kernel compilation (HAS_DRAFT_LOGITS) is a clever way to avoid separate kernels for the two modes. The benchmarks show clear throughput improvements at temperature=1.0, validating the approach.

## 7. [Auto-disable expandable_segments around cumem memory pool](https://github.com/vllm-project/vllm/pull/40812)


### Base Information

- **PR Number:** #40812
- **Author:** [youkaichao](https://github.com/youkaichao)
- **Merged By:** [youkaichao](https://github.com/youkaichao)
- **Merged time:** 2026-04-26 18:37:22
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/40812/files) (1):**
  - `vllm/device_allocator/cumem.py`

### Summary

**What changed and why**  
Removed the hard assertion in CuMemAllocator.__init__ that prevented using expandable_segments:True with the cumem memory pool. Instead, in use_memory_pool, temporarily disable expandable segments during the pool context and restore them on exit. This allows expandable segments to be used outside the pool while avoiding incompatibility inside it.

**Technical impact**  
Changes the behavior of the cumem allocator when expandable_segments:True is set in PYTORCH_CUDA_ALLOC_CONF. Previously, the process would crash with an assertion error; now it gracefully disables expandable segments only during memory pool operations. The context manager now has a try/finally block to ensure proper restoration of settings and the current tag.

**Potential risks**  
The use of torch.cuda.memory._set_allocator_settings is a private API and may change in future PyTorch versions. There is a risk of race conditions if multiple threads call use_memory_pool concurrently, as the environment variable check and setting are not atomic. Also, the restore assumes the original setting was exactly 'expandable_segments:True', which may not capture all possible configurations (e.g., multiple settings).

**Key insights**  
The change addresses a user-facing limitation by allowing expandable segments and memory pool to coexist, albeit with temporary disabling of expandable segments during pool use. The approach uses a context manager pattern to safely toggle settings, which is a common pattern for managing incompatible configurations. The PR also includes a note about AI assistance, which is transparent but may require additional human review for correctness.
