## 1. [[CPU][RISC-V] Support multiple RVV VLEN targets via compile-time dispatch](https://github.com/vllm-project/vllm/pull/39478)


### Base Information

- **PR Number:** #39478
- **Author:** [velonica0](https://github.com/velonica0)

  - `cmake/cpu_extension.cmake`
  - `csrc/cpu/cpu_types_riscv.hpp`
  - `csrc/cpu/cpu_types_riscv_defs.hpp`
  - `csrc/cpu/cpu_types_riscv_impl.hpp`

### Summary

**What changed and why**  
This PR introduces compile-time VLEN dispatch for RISC‑V RVV CPU kernels, replacing hardcoded `zvl128b` in the `-march` flag. The changes allow vLLM to be built for different VLEN targets (128, 256, or scalar) via a CMake variable (`VLLM_RVV_VLEN`), with auto‑detection from `/proc/cpuinfo`. This fixes silent miscompilation and crashes when rebuilding for hardware with a different VLEN (e.g., VLEN=256 on Spacemit X100).

**Technical impact**  
The architecture is restructured to separate VLEN‑dependent definitions (`cpu_types_riscv_defs.hpp`) from VLEN‑independent implementations (`cpu_types_riscv_impl.hpp`). The CMake build system now dynamically selects the appropriate `zvl<N>b` extension and sets corresponding compiler macros. This enables correct code generation for multiple VLENs while maintaining a single source base for RVV kernels.

**Potential risks**  
Auto‑detection may fail on systems without `/proc/cpuinfo` or with unconventional ISA strings, requiring manual `-DVLLM_RVV_VLEN` specification. The mapping of LMUL suffixes assumes only VLEN=128 or 256; future VLEN values (e.g., 512) will need updates to the definitions header. There is also a risk of inconsistent VLEN settings between compile‑time detection and runtime hardware if auto‑detection picks an incorrect value.

**Key insights**  
The refactor improves maintainability by centralizing VLEN‑specific mappings and using macro‑based intrinsic dispatch. Developers adding support for new VLENs should only modify `cpu_types_riscv_defs.hpp`. Ensure that the CMake variable `VLLM_RVV_VLEN` is explicitly set in cross‑compilation or containerized environments to avoid detection failures.

Instead of tracking incremental token‑ID counts and maintaining six scattered state variables, the new implementation re‑parses the full `current_text` on each streaming delta, uses `partial_tag_overlap()` to prevent marker leakage, and diffs accumulated arguments against already‑sent content. This directly fixes multiple bugs: token leakage into content, argument truncation due to speculative‑decoding bundling, content loss before the tool‑call section, and silent drops caused by regex mismatches.

**Technical impact**  
The architecture shifts from token‑based incremental tracking to a stateless text‑parsing model. The 8 KB section limit is removed, relying instead on the engine’s `max_model_len`. The parser now holds back text that partially matches a marker tag, ensuring markers never leak into content. Streaming diffs are computed by comparing the current parsed tool‑call state with the previously sent state, eliminating the need for token‑ID counting and rolling buffers.

**Potential risks**  
If the `partial_tag_overlap()` utility does not correctly handle all edge cases (e.g., overlapping Unicode sequences or nested markers), marker leakage could resurface. The removal of the 8 KB limit assumes the engine’s `max_model_len` is always sufficient; extremely long tool arguments might still cause memory or performance issues. The new regex uses `re.DOTALL` for non‑streaming extraction, which could potentially match across unintended boundaries if the pattern is too permissive.

**Key insights**  
Developers should note that the parser now requires `skip_special_tokens=False` to expose markers as literal text—this is enforced in `adjust_request()`. The streaming state is simplified to a single `_sent_content_idx` cursor, making the logic easier to debug. Ensure that the `partial_tag_overlap()` function is thoroughly tested for all possible partial‑marker scenarios. The fix for speculative‑decoding bundling is inherent to the diff‑based approach, but verify that `_compute_args_diff()` correctly handles all delta patterns.
