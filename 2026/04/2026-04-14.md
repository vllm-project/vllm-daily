### Summary

**What changed and why**  
This PR introduces TurboQuant, a new KV cache compression backend for vLLM that performs online 2‑bit KV cache compression using PolarQuant (WHT rotation + Lloyd‑Max scalar quantization for keys, uniform quantization for values). The changes add a new attention backend, `TURBOQUANT`, with four presets offering 2.6× to 4.9× compression. Compression happens at store time via fused Triton kernels, requiring no offline calibration or model modifications.

**Technical impact**  
The implementation adds a complete new attention backend with its own metadata, cache layout, and Triton kernels. It modifies the attention backend selection logic to route `turboquant_*` cache dtypes directly to the TurboQuant backend. The KV cache shape changes from a leading‑2 (K/V separate) layout to a single combined slot per head per position, reducing memory overhead. The system also introduces boundary‑layer protection by automatically skipping quantization on the first and last two layers to preserve embedding‑adjacent representations.

**Potential risks**  
1. The cache layout change is incompatible with other attention backends; mixing TurboQuant with non‑TQ layers in the same cache group could cause shape mismatches.  
2. Hybrid attention‑Mamba models are explicitly unsupported, which may limit adoption for certain architectures.  
3. The forced override of `flash_attn_version` to 2 may conflict with future FlashAttention 3 integrations.  
4. The static `NUM_KV_SPLITS` grid dimension for CUDA graphs assumes uniform batch sizes; variable splits could break graph capture.

**Key insights**  
- TurboQuant achieves near‑baseline throughput (79‑100%) with substantial compression (2.6‑4.9×), making it suitable for memory‑constrained deployments.  
- Developers should ensure uniform attention layers when enabling TurboQuant and avoid mixing with hybrid models.  
- The automatic boundary‑layer skip is a prudent safeguard, but users can extend it via `--kv-cache-dtype-skip-layers` for finer control.  
- Performance trade‑offs are clearly documented: higher compression presets yield lower throughput, allowing users to select based on their memory/throughput requirements.
