### Summary

**What changed and why**  
The change moves the `_free_encoder_inputs` call from `_update_after_schedule` to `update_from_output`. This fixes a race condition where encoder inputs could be freed prematurely because `num_computed_tokens` was incremented immediately upon scheduling, before GPU execution completed.

**Technical impact**  
This ensures encoder inputs remain pinned in memory until the corresponding GPU kernels have actually finished processing the scheduled tokens. It aligns the freeing logic with actual computation completion, preventing use-after-free errors or incorrect model behavior in encoder-decoder architectures.

**Potential risks**  
If speculative decoding or other advanced features interact with encoder inputs, this change might introduce subtle dependencies on `update_from_output` timing. Additionally, any future modifications to token counting or scheduling logic must maintain this ordering to avoid regression.

**Key insights**  
Always synchronize resource management with actual hardware execution, not just scheduling events. Developers should audit similar patterns where resources are freed based on counters updated before operation completion. This fix highlights the importance of understanding the full pipeline from scheduling to execution in asynchronous systems.
