### Summary

**What changed and why**  
Refactored MoE LoRA from monkey-patching decorators on the modular kernel into a first-class concern. Introduced MoELoRAContext dataclass to carry all LoRA state explicitly, added LoRAExpertsMixin for expert implementations, and inlined LoRA compute into TritonExperts, UnfusedOAITritonExperts, and MarlinExperts apply() methods. The change eliminates hidden state passed via moe_state_dict and makes LoRA visible in the expert source code, improving maintainability and extensibility.

**Technical impact**  
LoRA is now integrated directly into the expert apply() functions, removing the need for decorators and state dict smuggling. The new MoELoRAContext is threaded through the modular kernel path, and PunicaWrapperBase gains add_lora_w13/add_lora_w2 methods. Backend selection in oracle modules now filters by supports_lora() flag, so unsupported backends are skipped when LoRA is enabled. The refactor reduces code in fused_moe.py by ~260 lines and adds ~400 lines across new files and expert implementations.

**Potential risks**  
The refactor touches multiple expert implementations and the LoRA layer, so there is risk of regression in LoRA accuracy or performance if the inlined LoRA calls are not correctly placed or if the context propagation is missed. The MarlinExperts LoRA path uses wrapper functions for activation and moe_sum, which may introduce subtle differences in tensor lifetimes or ordering. The change also modifies backend selection logic, which could cause unexpected fallbacks for LoRA-enabled models.

**Key insights**  
The refactor makes LoRA a first-class citizen in the MoE kernel, improving code clarity and maintainability. By moving LoRA compute into expert apply() methods, future MoE changes are less likely to break LoRA silently. The new MoELoRAContext and LoRAExpertsMixin provide a clean extension point for adding LoRA support to new expert implementations.

## 2. [[Bugfix] Fix the DSML token leakage in DSV4/3.2](https://github.com/vllm-project/vllm/pull/40806)

  - `tests/tool_parsers/test_deepseekv32_tool_parser.py`
  - `vllm/tool_parsers/deepseekv32_tool_parser.py`

### Summary

**What changed and why**  
Fixed DSML token leakage in DeepSeek V4/3.2 tool parser by replacing the boolean `is_tool_call_started` flag with a content index tracker and a new `_extract_content` method that uses `partial_tag_overlap` to avoid emitting partial start markers as content. Added comprehensive tests for chunked streaming scenarios.

**Technical impact**  
Eliminates a bug where DSML start marker fragments (e.g., '<｜') could leak into the content stream during streaming, causing malformed tool calls. The fix ensures that only complete non-tool-call text is emitted and that partial markers are held back until fully recognized.

**Potential risks**  
Low risk. The change is localized to the streaming extraction logic and is covered by new tests. The `partial_tag_overlap` utility is already used elsewhere, so no new dependencies. Edge cases with very small chunk sizes are explicitly tested.

**Key insights**  
The root cause was that the old code used a boolean flag to detect the start token, but when the token was split across chunks, the flag could be set prematurely, causing the partial token to be emitted as content. The fix uses a content index and overlap detection to safely hold back partial markers.

### Summary

**What changed and why**  
Changed the condition for truncating padded routed output from always truncating when hidden_dim_was_padded to only truncating when there is a shared expert (shared_output is not None) or a routed output transform (latent MoE up-proj). This fixes a bug where GPT-OSS (which has no shared expert) had its output incorrectly truncated, causing errors. The change preserves the intended behavior for models with shared experts (like Nemotron-Nano-v3) while restoring correct behavior for models without.

**Technical impact**  
Alters the post-processing of fused MoE output: truncation of padded dimensions now only occurs when shared experts or latent up-projection are present. For models without these, the output retains its original padded shape, which is necessary for correctness. This is a minimal change (5 lines added, 1 removed) that affects the output tensor shape in specific configurations.

**Potential risks**  
Low risk. The change is guarded by existing conditions and only affects the truncation step. It could potentially cause shape mismatches if other parts of the code assume truncation always happens, but the PR author tested with GPT-OSS and the test passes. There may be an open bug for Nemotron-Nano-v3 FP4, but this PR does not address that and is not expected to introduce new issues.

**Key insights**  
The bug was introduced by a previous PR (#40794) that unconditionally truncated padded output, breaking models without shared experts. This fix restores the conditional truncation logic, ensuring that only models that need truncation (those with shared experts or latent up-proj) get it. The change is a targeted revert of a specific behavior change, preserving the fix for shared experts while avoiding regression for other models.

## 4. [[kv_offload+HMA][11/N]: Support store with multiple KV groups](https://github.com/vllm-project/vllm/pull/39403)


### Base Information

- **PR Number:** #39403
- **Author:** [orozery](https://github.com/orozery)
- **Merged By:** [orozery](https://github.com/orozery)
- **Merged time:** 2026-04-25 10:00:47
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/39403/files) (1):**
  - `vllm/distributed/kv_transfer/kv_connector/v1/offloading/scheduler.py`

### Summary

**What changed and why**  
Extended the offloading connector to support store operations when KVCacheConfig contains multiple KV groups. Previously, the code assumed a single group; now it iterates over all groups and handles per-group block indices and sizes. This is part of a series to enable heterogeneous memory architectures (HMA).

**Technical impact**  
The `_get_reqs_to_store` method now correctly processes multiple KV groups, computing offloadable tokens per group, filtering skipped blocks (due to sliding window/SSM), and constructing `GPULoadStoreSpec` with per-group sizes and block indices. The `advance_stored_idx` helper was added to update the next stored block index for each group. This enables multi-group offloading, but sliding window and SSM are not yet supported.

**Potential risks**  
The assertion that skipped blocks cannot appear after non-skipped blocks may fail if the assumption is violated. The change modifies the logic for selecting GPU block IDs to offload, which could introduce off-by-one errors or incorrect block selection. The `touch` call now touches all offload keys for all groups, which may increase memory usage or I/O. The `advance_stored_idx` is called even when no new offload keys are found, which might skip blocks incorrectly.

**Key insights**  
The PR introduces per-group handling in the offloading scheduler, a necessary step for HMA support. The filtering of skipped blocks (block_id == 0) is a workaround for unsupported sliding window/SSM. The use of `block_size_factor` to select the last GPU block per offload block is a clever way to detect skipped blocks. The change is backward-compatible for single-group configurations.

- **[Changed Files](https://github.com/vllm-project/vllm/pull/40767/files) (1):**
  - `tests/kernels/moe/test_moe_layer.py`

### Summary

**What changed and why**  
Added a torch.distributed.barrier() call in the finally clause of the parallel worker function in test_moe_layer.py. This fixes a deadlock caused by non-deterministic garbage collection of DeepEP buffers cached in a WeakValueDictionary, where some workers' buffers were collected (no-op) while others waited indefinitely.

**Technical impact**  
Ensures all workers synchronize after each test by clearing caches before proceeding, preventing deadlocks in multi-worker tests with TP=2,4 on MI300. The change is minimal (1 line) but critical for test reliability.

**Potential risks**  
Low risk; the barrier adds a synchronization point that could slightly increase test execution time, but it is necessary for correctness. No behavioral change to production code.

**Key insights**  
The deadlock originated from a subtle interaction between weak references and distributed communication: garbage collection of cached buffers caused mismatched states across workers. A barrier after each test ensures all workers are in sync, a common pattern for avoiding such issues in distributed testing.
