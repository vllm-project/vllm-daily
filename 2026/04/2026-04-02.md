### Summary

**What changed and why**  
The PR introduces a batched KV cache swap operation (`swap_blocks_batch`) that replaces per-layer per-block `swap_blocks` calls with a single driver invocation. This reduces submission overhead by aggregating all copy operations into one batch, leveraging `cuMemcpyBatchAsync` on CUDA 12.8+ (with a fallback to looped `cudaMemcpyAsync` for older CUDA/ROCm). The change aims to improve performance in KV cache offloading scenarios, especially for models with many small block copies.

**Technical impact**  
This change significantly reduces driver submission overhead by batching memory copies, leading to measurable speedups in KV cache transfer times (3.6–7.4× in handler benchmarks) and improved end-to-end throughput (up to 18% for Qwen2.5-7B). The architecture now pre-computes base pointers and block sizes, constructing flat pointer arrays for all tensors and block pairs, which are passed to the new batched kernel. No extra GPU memory is required, and behavior remains unchanged.

**Potential risks**  
The fallback path for older CUDA/ROCm uses a simple loop, which may not fully amortize submission overhead in those environments. The implementation relies on `int64_t` and pointer size equality assumptions (`static_assert`), which could break on non-64-bit platforms. Error handling in `cuMemcpyBatchAsync` reports failure indices, but the fallback loop lacks granular error reporting. Large batch sizes could expose driver or memory bandwidth limitations.

**Key insights**  
The batched approach is most beneficial for models with many small block copies (e.g., Qwen2.5-7B with 4 KV heads), where submission overhead dominates. Developers should ensure CUDA 12.8+ is used for optimal performance and verify platform compatibility. Consider extending batching to other memory-bound operations in the codebase. Monitor performance on older CUDA/ROCm systems, as the fallback may not achieve the same gains.

### Summary

The changes eliminate separate quantization kernels per layer by integrating quantization directly into the attention operation, reducing memory round-trips. This is phase 1 of a two-phase implementation, with phase 2 expected to bring direct kernel support for quantized output.

**Technical impact**  
The fusion is implemented via a new pattern matcher (`MLAAttnQuantFusionPass`) that matches MLA attention followed by quantization ops and replaces them with a fused version. Since current MLA kernels do not natively support quantized output, the implementation uses a temporary buffer swap in `forward_impl`: computation occurs in full precision, then results are quantized into the pre-allocated quantized output buffer. This approach maintains compatibility across all MLA backends but does not yet eliminate memory round-trips, limiting immediate performance gains.

**Potential risks**  
The temporary buffer swap introduces extra memory overhead and may not yield significant speed improvements until kernels support direct quantized output. There is a risk of regression in compiled mode, as shown by the benchmark where fused compiled runs had higher latency (44.92s vs 18.56s) and lower throughput. Additionally, the fusion relies on pattern matching that may not cover all quantization variants or edge cases, such as mixed precision scenarios or unsupported hardware configurations.

**Key insights**
