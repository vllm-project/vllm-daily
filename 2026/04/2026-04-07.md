  - `tests/tool_parsers/test_gemma4_tool_parser.py`
  - `vllm/tool_parsers/gemma4_tool_parser.py`

### Summary

**What changed and why**  
The fix addresses a bug in Gemma4's streaming tool parser where buffered text deltas were incorrectly stitched back into the accumulated `current_text`. This caused HTML content in tool arguments (like `<html>` or `<meta>` tags) to be duplicated into malformed output (e.g., `<<htmlhtml`). The root issue was reconstructing `current_text` from buffered deltas, which corrupted the parser's internal state after tool calls.

**Technical impact**  
The change ensures `current_text` remains consistent with the upstream streaming state, using buffered `delta_text` solely for emission. This preserves the integrity of HTML and plain text content in tool arguments, preventing invalid intermediate states that could lead to duplicated or malformed output. The parser now correctly handles text streaming both inside and after tool calls.

**Potential risks**  
If the buffering logic (`_buffer_delta_text`) introduces delays or alters delta sequencing, it could affect real-time streaming performance. Edge cases involving nested or overlapping tool calls with complex HTML might still require validation. Additionally, the fix assumes upstream `current_text` is always accurate, which could be problematic if other components modify it unexpectedly.

**Key insights**  
Always maintain a clear separation between buffered emission data and accumulated parser state to avoid corruption. The added regression tests are crucial for catching similar issues with HTML or special characters in tool arguments. Developers should verify that the buffering mechanism does not introduce latency or drop characters in high-throughput streaming scenarios.

### Summary

**What changed and why**  
The fix addresses a crash when using quantized KV cache types (like `fp8_e4m3`) with the `extract_hidden_states` speculative decoding method. The issue occurred because `ExtractHiddenStatesModel` passed the global quantized `cache_config` directly to `CacheOnlyAttentionLayer`, which only supports `auto`, `bfloat16`, or `float16` dtypes. The solution creates a shallow copy of the cache configuration, resetting `cache_dtype` to `"auto"` for the hidden-states cache layer while preserving the original quantization for the target model's KV cache.

**Technical impact**  
This change decouples the dtype requirement for the hidden-states cache from the main KV cache quantization. The hidden-states cache layer now operates with the model's native dtype (via `"auto"`), ensuring compatibility without affecting the performance or memory benefits of quantized KV caches in the primary attention layers. The architecture maintains separation between speculative decoding intermediates and actual KV projections.

**Potential risks**  
If `cache_config` is `None`, the fix is bypassed, which is correct but should be verified in that code path. The shallow copy via `dataclasses.replace` assumes `cache_config` is a dataclass; changes to its structure could affect this. There's a minor risk if other parts of `cache_config` besides `cache_dtype` are incompatible with `CacheOnlyAttentionLayer`, but this seems unlikely given the context.

**Key insights**  
The fix elegantly isolates dtype concerns by modifying only the configuration passed to the hidden-states cache. Developers should note that `extract_hidden_states` intermediates don't benefit from KV cache quantization, which is appropriate. Ensure `is_quantized_kv_cache` correctly identifies all quantized dtypes. Consider adding a unit test for this specific scenario to prevent regression.
