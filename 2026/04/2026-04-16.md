  - `vllm/v1/attention/backends/mla/indexer.py`

### Summary

**What changed and why**  
The PR optimizes MLA attention metadata preparation for speculative decoding with MTP > 1 by splitting the variable-length decode path into uniform and non-uniform cases. A new Triton kernel (`_prepare_uniform_decode_kernel`) replaces PyTorch operations (`repeat_interleave`) when all decode lengths are equal, which is common during target-model validation of draft tokens. This reduces per-step overhead in `_prepare_decode_tensors`.

**Technical impact**  
The change improves performance for speculative decoding workloads by fusing metadata computations into a single GPU kernel for uniform decode lengths. The existing variable-length path remains unchanged, preserving backward compatibility. The optimization directly reduces CPU-GPU synchronization and kernel launch overhead, especially beneficial when `max_decode_len > 1` and decode lengths are uniform.

**Potential risks**  
The Triton kernel introduces a new dependency and may have platform-specific compatibility issues (e.g., non-CUDA backends). Edge cases where `min_decode_len == max_decode_len` but tensors are padded could incorrectly trigger the uniform path. Additionally, the kernel assumes contiguous memory layouts; changes to `block_table.stride(0)` or buffer shapes could cause silent errors.

**Key insights**  
The optimization is well-targeted for speculative decoding, showing measurable latency reductions. Developers should verify the uniform-length condition accounts for padding and ensure the Triton kernel is only invoked on supported platforms. Consider adding a fallback to the PyTorch path for non-CUDA devices or when Triton is unavailable.
