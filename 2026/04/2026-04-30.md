### Summary

**What changed and why**  
Forwarded request-level reasoning_parser_kwargs through the engine request path and made the structured output manager instantiate reasoning parsers per request instead of sharing a manager-level parser instance. This fixes a bug where reasoning parsers like DeepSeekV3ReasoningParser that depend on per-request chat-template kwargs were not receiving those kwargs, causing structured output to fail when combined with reasoning.

The new _get_reasoner method caches the reasoner on the request, so repeated calls are safe. The streaming input path raises NotImplementedError if reasoning_parser_kwargs is provided, which is acceptable since streaming with reasoning is not yet supported.

**Key insights**  
The fix correctly moves reasoning parser instantiation from manager-level to request-level, enabling per-request configuration. The approach is cleaner than alternative fixes that hack the reasoning parser implementation. The test coverage is good, including a test that verifies request-level kwargs override the default reasoner behavior.

### Summary

**What changed and why**  
This PR adds end-to-end NVFP4 KV cache support for the FlashInfer TRTLLM backend on Blackwell GPUs. It wires up the read/attention side by forcing the trtllm-gen backend for prefill and decode wrappers when NVFP4 is used, setting o_data_type to FP8 (since the NVFP4 kernel only outputs FP8), and handling dequantization back to model dtype via a pre-allocated FP8 buffer. It also passes block scales (kv_cache_sf) to the wrapper calls, renames get_fp8_dtype_for_flashinfer to get_dtype_for_flashinfer to handle nvfp4->uint8 mapping, updates ModelOpt to recognize NVFP4 KV cache quantization, adds validation to prevent NVFP4 with MLA, and extends tests and documentation accordingly.

**Technical impact**  
Enables a new KV cache quantization format (NVFP4) that reduces memory bandwidth by packing FP4 data and FP8 block scales, improving inference efficiency on Blackwell hardware. The change modifies the attention backend to conditionally use the trtllm-gen kernel and FP8 output buffers, which may affect performance and memory usage for all models using FlashInfer with NVFP4. The pre-allocated FP8 output buffer adds a small memory overhead proportional to max_num_batched_tokens * num_heads * head_size.

**Potential risks**  
The NVFP4 path is only supported on SM100 (Blackwell) GPUs; running on older hardware will fail. The forced trtllm-gen backend may introduce compatibility issues if other backends are expected. The dequantization step from FP8 to model dtype adds a small computational overhead. The pre-allocated FP8 buffer assumes a fixed max_num_tokens, which could be insufficient if the scheduler config changes. The changes to ModelOpt renaming may affect checkpoint loading for FP8 KV cache models if any external code references the old class name.

**Key insights**  
The PR carefully handles the mismatch between the NVFP4 kernel's FP8-only output and the model's native dtype by using a temporary FP8 buffer and dequantizing. It also correctly splits the packed NVFP4 cache into data and scale views for the kernel. The test coverage includes both prefill and decode paths with NVFP4, and the integration test runs the full MetadataBuilder->FlashInferImpl pipeline. The documentation update is minimal but accurate.
