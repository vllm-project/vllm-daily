- **[Changed Files](https://github.com/vllm-project/vllm/pull/41189/files) (1):**
  - `csrc/topk.cu`

### Summary

**What changed and why**  
Fixed a cooperative deadlock in persistent_topk when TopK=1024 and MTP=1 by querying occupancy for the actual vec_size instantiation, reserving headroom CTAs per SM, and adding a fallback to FilteredTopK when the cooperative launch would oversubscribe. The deadlock occurred because the kernel is persistent and without enough SMs to launch the grid, no kernel finishes.

**Technical impact**  
Changes the kernel launch logic to use correct occupancy per vec_size, reserves one CTA per SM as headroom for co-resident kernels, and adds a TORCH_CHECK and fallback path to prevent deadlock. This ensures the cooperative spin-wait barrier does not hang when CTAs cannot be scheduled co-resident.

**Potential risks**  
The fallback to FilteredTopK requires >=128KB shared memory per block; if that condition fails, the kernel will error out instead of deadlocking. The headroom reservation may reduce parallelism slightly when occupancy is low, but prevents hangs. The change is limited to the persistent_topk launch path and should not affect other kernels.

**Key insights**  
The deadlock was caused by overestimating occupancy (using vec_size=4 for all instantiations) and not reserving headroom for co-resident kernels. The fix uses the correct kernel template for occupancy query and reserves CTAs per SM to ensure the cooperative barrier can make progress. The fallback to FilteredTopK provides a safe alternative when oversubscription is unavoidable.

### Summary

**What changed and why**  
Fixed a bug where the per-request admission cap (max_admission_blocks_per_request) was incorrectly applied during per-step block allocation in addition to the request admission gate. This caused a mismatch between the number of blocks predicted by get_num_blocks_to_allocate and the number actually allocated by allocate_new_blocks, leading to a ValueError when the block pool ran out of free blocks. The fix introduces an apply_admission_cap parameter to control when the cap is applied, ensuring it is only used during admission checks and not during per-step allocation.

**Technical impact**  
Resolves a critical runtime error that prevented models with sliding window or chunked-prefill attention (e.g., DeepSeek V4) from running under high concurrency. **Potential risks**  
Low risk. The change is minimal and targeted, adding a parameter with a default of False to preserve existing behavior. The test covers the sliding window case; other specs (e.g., Mamba) pass the parameter through unchanged.

**Key insights**  
The bug was introduced by PR #40946 which added the admission cap but inadvertently applied it to per-step allocation as well. The root cause is that the cap was applied unconditionally in get_num_blocks_to_allocate, but allocate_new_blocks does not apply the cap, leading to inconsistency. The fix decouples the two paths via a parameter, aligning with the principle that admission caps should only gate request admission, not per-step block requests.

### Summary

**What changed and why**  
Refactored server shutdown logic in test utilities to fix a bug where multi-process DP tests waited unnecessarily for GPU memory release. The old sequential shutdown caused each server to wait for all siblings to release memory, leading to timeouts. The change introduces a parallel termination phase via `shutdown_many` class method, which kills all server process trees concurrently and then waits for GPU memory release once using the earliest baseline. **Technical impact**  
Reduces test runtime for distributed DP tests by approximately 10 minutes per test group by eliminating the 120-second timeout per server during shutdown. **Potential risks**  
Threading for parallel termination may introduce race conditions if process tree termination is not thread-safe, but the code uses daemon threads and joins before proceeding. The selection of earliest baseline assumes that `_pre_server_gpu_memory` is set correctly; if any server has `None`, it is skipped. The change modifies the public API of `RemoteVLLMServer` by adding a class method, but existing single-server usage remains unaffected.

**Key insights**  
The root cause was that each server's `_wait_for_gpu_memory_release` checks total GPU memory across all devices, so sequential shutdown caused the first server to wait for siblings still holding memory. The fix separates process termination from memory wait, enabling parallel teardown. This pattern is useful for any test fixture managing multiple GPU-bound processes.

  - `vllm/v1/engine/core.py`

### Summary

**What changed and why**  
Implements a two-phase pause protocol for DP engines to prevent deadlocks when pausing and resuming. Previously, returning from pause did not guarantee engines stay paused because stale START_DP_WAVE messages could re-activate an engine, causing collective operation mismatches. The new protocol splits pause into two phases: phase 1 sets a local pending_pause flag while continuing to step and honor wave messages; phase 2 uses a combined all-reduce to reach consensus among all ranks, after which START_DP_WAVE is ignored. Resume clears both flags and adds a barrier to ensure all ranks have resumed before stepping.

**Technical impact**  
Adds a combined all-reduce (sync_dp_state) that fuses the existing has_unfinished check with a pause consensus check into a single 2-element SUM all-reduce, avoiding additional communication overhead. Modifies the pause_scheduler method to return immediately after setting pending_pause and kick-starting idle engines, with the actual stop deferred until consensus. Resume now includes a barrier to synchronize all DP ranks before resuming. The change also adds a barrier utility method and updates the test to cover the new protocol.

**Potential risks**  
The two-phase protocol introduces a bounded delay (up to 32 dummy steps) before pause completes, which could affect latency-sensitive scenarios. The combined all-reduce logic uses a modulo check for pause consensus, which assumes no overflow; this is safe for typical DP sizes but should be verified for very large groups. The resume barrier adds a synchronization point that could introduce latency if not all ranks resume promptly. The change modifies core engine behavior, so thorough testing across different DP configurations is recommended.

**Key insights**  
The key insight is that ignoring START_DP_WAVE unconditionally while paused can cause deadlocks because other ranks may still need the paused rank to participate in the all-reduce. By using a two-phase approach with collective consensus, the protocol ensures that all ranks agree to stop together, eliminating the race condition. The fused all-reduce efficiently synchronizes both work status and pause intent without extra communication rounds.
