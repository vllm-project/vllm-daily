### Summary

**What changed and why**  
The refactor consolidates CPU KV-cache offloading by unifying LRU and ARC managers into a single `CPUOffloadingManager` with a pluggable `CachePolicy` strategy pattern. It removes the unnecessary `Backend` abstraction and restructures the code into a `cpu/` package for better organization. This eliminates ~40 lines of duplicated skeleton code and simplifies the architecture.

**Technical impact**  
The changes reduce code duplication and improve maintainability by centralizing common logic (ref-counting, event emission, block pool management) in `CPUOffloadingManager`. The new `CachePolicy` interface cleanly separates eviction algorithms, making it easier to add new policies. The atomic eviction behavior in both policies ensures consistency, and the fix to `LRUCachePolicy.touch()` prevents unreliable checks for blocks with `ref_cnt == 0`.

**Potential risks**

The atomic eviction change may subtly alter behavior in edge cases where partial eviction was previously allowed. Additionally, the registry-based policy selection (`_CACHE_POLICIES`) requires careful handling of string literals to avoid runtime errors.

**Key insights**  
This refactor significantly improves code clarity and reduces duplication, making the offloading subsystem more maintainable. Developers should note the atomic eviction guarantee and the fixed `touch()` method. When adding new cache policies, they must register them in `_CACHE_POLICIES` and ensure they implement the `CachePolicy` interface correctly. The restructured package layout (`cpu/manager.py`, `cpu/policies/`) sets a clear precedent for future modular extensions.
