**What changed and why**  
The change reorders operations in `Glm4vVisionTransformer.forward()` to move `cu_seqlens.to(self.device)` after computing `max_seqlen` and `seqlens`. However, any future modifications that depend on GPU-based computations for these variables could reintroduce synchronization issues or errors.

**Key insights**  
This is a high-impact, low-risk optimization that demonstrates the importance of minimizing unnecessary GPU-CPU synchronizations in performance-critical paths. Developers should audit similar patterns where tensor `.item()` calls or CPU operations on GPU tensors may cause hidden synchronization bottlenecks.

## 2. [[Bugfix] Handle libsndfile sf_error(NULL) race condition in audio fallback](https://github.com/vllm-project/vllm/pull/37782)

- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/37782/files) (1):**
  - `vllm/multimodal/media/audio.py`

### Summary

**What changed and why**  
The change adds error code `0` to the `_BAD_SF_CODES` set in the audio loading module. This addresses a race condition in libsndfile where concurrent failures in `sf_open_virtual()` on unsupported formats (e.g., MP4 bytes) can cause `sf_error(NULL)` to return `0` instead of the actual error code, bypassing the PyAV fallback and crashing requests.

**Technical impact**  
This ensures that when the libsndfile race condition occurs, the error is treated as a client-side issue (invalid audio file) rather than a server error, allowing the system to gracefully fall back to PyAV for audio decoding. It maintains backward compatibility and aligns with the existing error-handling logic for unsupported or malformed audio.

**Potential risks**  
Adding `0` could inadvertently catch other `sf_error(NULL)` scenarios where the error code is legitimately `0` (e.g., no error), though this is unlikely in failure paths. The fix is minimal and localized, but the underlying race condition in libsndfile remains unaddressed, meaning similar issues may surface in other contexts if libsndfile is used directly elsewhere.

**Key insights**  
This is a targeted workaround for a known upstream bug in libsndfile. Developers should be aware that error code `0` now triggers a fallback, and consider monitoring for any unintended side effects. For long-term stability, tracking the upstream issue (python-soundfile#479) and updating libsndfile when a fix is available is recommended.

## 5. [[Bugfix] Fix pooling non-determinism from pinned prompt_lens aliasing](https://github.com/vllm-project/vllm/pull/37775)

  - `tests/v1/worker/test_gpu_input_batch.py`
  - `vllm/v1/worker/gpu_input_batch.py`

### Summary

**What changed and why**  
The fix addresses non-deterministic pooling behavior caused by memory aliasing between `prompt_lens` and a pinned memory buffer. Previously, `torch.from_numpy(self.num_prompt_tokens[:self.num_reqs])` created a view into pinned memory that could be modified asynchronously by GPU operations. The change replaces this with `.clone()` to create an independent copy, ensuring deterministic pooling scores.

**Technical impact**  
This change eliminates a race condition where the `prompt_lens` tensor could be corrupted by concurrent modifications to the shared pinned buffer during batch operations. It ensures that pooling metadata remains consistent throughout the pooling pipeline, making model outputs deterministic across runs while maintaining performance through proper memory isolation.

**Potential risks**  
The `.clone()` operation introduces a small memory overhead and minor performance cost for copying data, though this is negligible compared to pooling computations. There's a risk that similar aliasing issues could exist elsewhere in the codebase where pinned numpy arrays are converted to tensors without proper copying. The test only validates the specific `prompt_lens` case, potentially missing other affected code paths.

**Key insights**  
Always use explicit copies when converting pinned numpy arrays to tensors that will be used independently. The test effectively demonstrates the aliasing problem by mutating the underlying buffer and verifying isolation. Developers should audit other `torch.from_numpy()` calls on pinned memory throughout the codebase to prevent similar non-determinism issues in other components.

  - `vllm/v1/engine/utils.py`
  - `vllm/v1/serial_utils.py`
  - `vllm/v1/utils.py`

### Summary

**What changed and why**  
This PR introduces a zero-copy IPC (inter-process communication) pathway for multimodal tensors between API servers and engine cores using `torch.multiprocessing.Queue`. The primary goal is to avoid serialization and GPU↔CPU copies when transferring video‑decode results or other multimodal data, enabling efficient multi‑GPU scaling for video‑heavy workloads. The mechanism is only active when the multimodal processing cache is disabled.

**Technical impact**  
The changes add a new IPC transport layer (`TensorIpcData`/`TensorIpcHandle`) and integrate it with the existing `MsgpackEncoder`/`MsgpackDecoder`. Tensor queues are created per engine, routed via a `set_target_engine` context, and plumbed through the engine lifecycle—from handshake metadata to core input decoders. Configuration is exposed via `MultiModalConfig.mm_tensor_ipc`, CLI flags, and an environment variable. The IPC path falls back to standard serialization when disabled or unavailable.

**Potential risks**  
- The IPC mechanism currently only supports DP=1, TP=1, and PP=1 configurations; using it with parallelization will raise an error.  
- Shared‑memory handles must be carefully managed to avoid leaks or stale tensors, especially during request abortion.  
- The fallback to standard serialization could introduce subtle performance regressions if the IPC path fails silently.  
- Multi‑producer scenarios (multiple API servers) rely on correct queue‑draining logic, which may become a bottleneck under high concurrency.

**Key insights**  
- The implementation is a foundational step for zero‑copy multimodal data transfer, but its current limitation to single‑engine setups restricts immediate use in scaled deployments.  
- Developers should verify that `VLLM_WORKER_MULTIPROC_METHOD=spawn` is set when using `torch_shm` IPC to avoid compatibility issues.  
- Comprehensive tests have been added for CUDA/CPU IPC, buffer management, and cleanup; these should be extended to cover real‑world multimodal workloads once the cache‑disable constraint is lifted.  
- The performance results show minimal difference in the tested scenario, suggesting that further benchmarking with larger tensors and higher concurrency is needed to validate the optimization’s value.
