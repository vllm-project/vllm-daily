### Summary

**What changed and why**  
The PR updates the Conv3d layer to consistently use `forward_mulmat` for PyTorch versions 2.9.0 and newer. This change replaces a version-specific check for exact matches ("2.9.0" or "2.9.1") with a broader version range check using `is_torch_equal_or_newer("2.9.0")`. **Technical impact**  
This modification ensures that the performance optimization (using `_forward_mulmat` instead of the default convolution path) is applied to all PyTorch versions from 2.9.0 onward, not just 2.9.0 and 2.9.1. **Potential risks**

Additionally, the behavior relies on the `enable_linear` flag, which must be properly configured to avoid unintended side effects.

**Key insights**  
The update future-proofs the code against upcoming PyTorch releases while maintaining performance for known affected versions. Developers should verify that `is_torch_equal_or_newer` functions as expected and monitor PyTorch release notes for any changes to Conv3D backend support to reassess this logic later.

### Summary

**What changed and why**  
This PR optimizes pooling operations by eliminating redundant device copies of token IDs when they're only needed on CPU. Previously, token IDs were copied from CPU to GPU and back to CPU unnecessarily; now they remain on CPU when GPU processing isn't required, reducing data movement overhead.

**Technical impact**  
The changes introduce a separate CPU-only tensor (`prompt_token_ids_cpu`) in `PoolingMetadata` and modify pooling layers (BERT, GritLM, special poolers) to use this CPU tensor directly. The changes assume all pooling operations that require token IDs can work with CPU tensors, which may not hold for future GPU-accelerated pooling implementations. There's also a risk of silent errors if `requires_token_ids` isn't properly configured.

**Key insights**  
This optimization is highly effective for CPU-only pooling workloads but creates a design assumption that token IDs for pooling are CPU-only. Future developers should be aware that adding GPU-accelerated pooling would require revisiting this architecture. The unit tests provide good coverage, but integration testing across all pooling models is recommended to ensure no regressions.

The fix explicitly separates the creation of image/audio processors and passes them as arguments to the main processor.

**Technical impact**  
This ensures that multimodal keyword arguments (image/audio data) are properly processed and passed to the model. **Potential risks**  
If the image_processor or feature_extractor creation fails (e.g., due to missing encoder attributes), the model initialization will crash. There's also a risk of inconsistent state if the tokenizer's encoder changes after processor creation, though this is unlikely in normal use.

**Key insights**  
Always ensure processor constructors explicitly list all required components for Transformers' dynamic dispatch. The fix demonstrates good separation of concerns by isolating processor creation. Developers should verify that similar multimodal models follow this pattern to avoid dispatch issues.
