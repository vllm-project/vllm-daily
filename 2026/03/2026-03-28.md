## 1. [[ROCm][CI] Fix cross-attention dispatch for encoder-decoder models](https://github.com/vllm-project/vllm/pull/38450)


### Base Information

- **PR Number:** #38450
- **Author:** [AndreasKaratzas](https://github.com/AndreasKaratzas)

- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/38450/files) (6):**

  - `tests/entrypoints/openai/speech_to_text/test_transcription_validation_whisper.py`
  - `tools/pre_commit/generate_attention_backend_docs.py`
  - `vllm/platforms/rocm.py`
  - `vllm/v1/attention/backends/rocm_aiter_fa.py`
  - `vllm/v1/attention/backends/rocm_attn.py`

### Summary

**What changed and why**  
Two ROCm attention backends (`ROCM_ATTN` and `ROCM_AITER_FA`) were modified to exclude `ENCODER_DECODER` (cross-attention) from their supported attention types. This prevents these backends from being selected for cross-attention layers in encoder-decoder models, fixing incorrect beam search results caused by flawed prefill kernel implementations for cross-attention.

**Technical impact**  
Cross-attention layers in encoder-decoder models (e.g., Whisper, BART) will now dispatch to alternative backends like `ROCM_AITER_UNIFIED_ATTN` or `TRITON_ATTN` on ROCm platforms. This ensures correct attention computation by avoiding backends that mishandle encoder KV caching or sequence boundaries during prefill. The dispatch system’s logging was enhanced to clearly indicate when fallback selection occurs.

**Potential risks**  
Removing cross-attention support reduces the usable backend options on ROCm, potentially affecting performance if fallback backends are less optimized. The fix assumes alternative backends are always available and correctly implemented for cross-attention; if not, model execution may fail. The underlying bugs in the excluded backends remain unaddressed and could affect other attention types or future use cases.

**Key insights**  
The root cause was that both backends’ prefill paths incorrectly assumed self-attention semantics, leading to double-counting encoder KV or wrong masking. Developers should verify cross-attention support when adding new backends and ensure prefill kernels distinguish between self- and cross-attention. Enhanced logging helps debug dispatch decisions, but a long-term fix would involve correcting the kernel implementations rather than disabling support.

**Key insights**  
This is a defensive fix that correctly addresses a thread-safety issue without introducing complex synchronization. Developers should note that the cache architecture itself remains non-thread-safe; future enhancements could explore making the cache thread-safe or redesigning the worker-cache interaction. ## 12. [Fix Device Index for ROCm Ray Workers in MoE Benchmark](https://github.com/vllm-project/vllm/pull/38108)

- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/38108/files) (1):**
  - `benchmarks/kernels/benchmark_moe.py`

### Summary

**What changed and why**  
The fix addresses a device index mismatch when running the MoE autotuner with Ray on ROCm (AMD GPU) systems. Ray isolates each worker to a single GPU using `ROCR_VISIBLE_DEVICES`, making the global `self.device_id` invalid within the worker's local context. The change replaces `self.device_id` with the local index `0` in the device guard.

**Technical impact**  
This ensures the `torch.accelerator.device_index` context manager receives a valid local device ordinal (always 0) for each Ray worker, preventing a CUDA/HIP error. The tuning workload can now distribute correctly across multiple GPUs on ROCm systems without crashing, maintaining the intended performance benchmarking functionality.

**Potential risks**  
If the code is ever run in a non-Ray environment where multiple devices are visible to a single process, hardcoding index `0` could incorrectly restrict execution to only the first GPU. Additionally, any future changes to Ray's device isolation mechanism might require revisiting this assumption.

**Key insights**  
The fix highlights the importance of distinguishing between global and local device IDs in distributed GPU contexts. Developers should verify that device indexing logic aligns with the runtime environment's visibility settings, particularly when using workload managers like Ray with ROCm. Consider adding an environment check to make the device index selection more robust across different deployment scenarios.
