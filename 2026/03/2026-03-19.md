### Summary

**What changed and why**

**Technical impact**
This changes the initialization flow from synchronous port selection to asynchronous binding and reporting. **Potential risks**
The 30-second timeout for receiving ZMQ addresses could cause startup failures if the coordinator process hangs during initialization. The pipe communication adds a new failure mode where EOF errors could occur if the child process crashes before sending addresses. The changes assume proper cleanup of pipe resources in all code paths.

**Key insights**
The solution elegantly solves the race condition by letting the binding process determine the actual ports. Developers should ensure the timeout value is appropriate for production environments and monitor for any startup failures. The pattern could be reused for other multi-process initialization scenarios requiring resource coordination.
