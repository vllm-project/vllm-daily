  - `vllm/model_executor/models/qwen3_next.py`

### Summary

**What changed and why**  
This PR introduces a new fused kernel `fused_sigmoid_gating_delta_rule_update` that combines the previously separate `fused_gdn_gating` and `fused_recurrent_gated_delta_rule` operations into a single kernel. The change aims to improve performance by reducing kernel launch overhead and optimizing computation, particularly for Qwen3-Next models with and without speculative decoding.

**Technical impact**  
The kernel fusion reduces execution time from approximately 5µs (two separate kernels) to 3µs (single fused kernel), representing a 40% reduction in kernel runtime. This optimization directly improves inference throughput, as evidenced by benchmark results showing 1-2% improvements in output token throughput across various model configurations. The implementation maintains functional equivalence while adding support for both speculative and non-speculative decoding scenarios.

**Potential risks**  
The kernel introduces complex conditional logic for handling different execution modes (speculative decoding, continuous batching, variable-length sequences), which could increase maintenance complexity. The accuracy testing shows minor variations in model outputs, though these appear within acceptable tolerance. There's also a risk of numerical precision differences due to the replacement of `b_q / sqrt` with `b_q * rsqrt` and the fused computation of sigmoid gating.

**Key insights**  
The performance gains are most significant in speculative decoding scenarios (2% throughput improvement) compared to standard decoding (1% improvement). Developers should verify that the fused kernel maintains numerical equivalence across all supported hardware platforms and data types. The implementation demonstrates effective kernel fusion strategy that reduces launch overhead while preserving functional correctness, serving as a good pattern for future optimization efforts.

### Summary

**What changed and why**  
The PR changes the `multi_thread_safetensors_weights_iterator` function to use a generator expression instead of a list comprehension for creating futures. This prevents storing all loaded weight files in memory simultaneously, reducing peak memory usage during model loading, particularly for large models.

**Technical impact**  
This change transforms the eager creation of all futures into a lazy generator, allowing the thread pool to process files incrementally. Additionally, the weight iteration now uses `state_dict.pop()` to yield items while removing them from the dictionary, enabling earlier garbage collection of each file's data.

**Potential risks**

There's a minor risk if the generator has side effects, but here it's just iterating over a static file list. The `pop()` operation modifies the state dict in-place, which is safe since it's local and not reused.

**Key insights**  
This is a memory optimization fix that's crucial for loading very large models, as demonstrated by the ~30GB reduction. Developers should be aware that using generator expressions with thread pools can prevent OOM errors when processing large datasets. The pattern of yielding and popping dictionary items is an effective technique for incremental processing and memory management.
