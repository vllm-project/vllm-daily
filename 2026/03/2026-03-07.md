  - `vllm/distributed/kv_transfer/kv_connector/v1/lmcache_mp_connector.py`

### Summary

**What changed and why**  
Added logic in `update_state_after_alloc` to release LMCache chunk locks for blocks that vLLM computes locally instead of retrieving from the cache. This prevents a memory leak in multiprocess mode where unreleased locks block LRU eviction, causing unbounded memory growth.

**Technical impact**  
The fix ensures proper lock lifecycle management by conditionally calling `free_lookup_locks`. When vLLM computes blocks itself (fully or partially), corresponding LMCache locks are freed early, allowing the cache’s eviction policy to function correctly and maintain stable memory usage.

**Potential risks**  
- The boundary handling between vLLM blocks and LMCache blocks relies on `free_lookup_locks` internal floor division; misalignment could theoretically cause premature lock release or retention if the logic is flawed.  
- The change assumes `tracker.num_lmcache_hit_blocks` and `tracker.num_vllm_hit_blocks` are always accurate; incorrect tracking could lead to freeing locks still needed or retaining locks unnecessarily.

**Key insights**  
- This fix addresses a critical resource leak that could degrade system reliability over time. Developers should verify the boundary alignment logic in `free_lookup_locks` and ensure tracker fields are reliably populated. Consider adding metrics to monitor lock release patterns in production.

The cleanup of profiling KV caches and graph pools must be thorough to avoid memory leaks or interference with subsequent operations.

**Key insights**  
Developers should enable `VLLM_MEMORY_PROFILER_ESTIMATE_CUDAGRAPHS=1` when using CUDA graphs to avoid OOMs, and may need to increase `--gpu-memory-utilization` to maintain the same effective KV cache size.
