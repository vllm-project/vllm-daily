### Summary

**What changed and why**  
The changes optimize scheduler overhead for async remote KV loads in P/D disaggregation by introducing a separate `skipped_waiting` queue. Instead of repeatedly moving blocked requests (e.g., waiting for remote KVs, FSM compilation, or streaming input) within the main waiting queue, they are now placed in `skipped_waiting` and promoted only when ready (e.g., on `finished_recving`). This reduces queue churn and CPU overhead when remote KV latency is high.

**Technical impact**  
The scheduler now maintains two waiting queues: `waiting` for ready requests and `skipped_waiting` for blocked ones. Scheduling logic prioritizes `skipped_waiting` in FCFS policy to preserve order, while priority scheduling compares heads of both queues. Blocked requests are only re-evaluated when their status changes (e.g., KV transfer completes), avoiding unnecessary scans. This improves throughput by ~5% in high-latency scenarios.

**Potential risks**  
If the promotion logic fails (e.g., `finished_recving` not triggering status updates), requests could stall indefinitely in `skipped_waiting`. The dual-queue approach adds complexity to scheduling order guarantees, especially under priority policy where queue head comparisons may introduce subtle ordering bugs. Edge cases like mixed blocked statuses (FSM + remote KV) require careful handling to maintain fairness.

**Key insights**  
Developers should ensure that all blocked status transitions (e.g., `WAITING_FOR_REMOTE_KVS` → `WAITING`) consistently call promotion logic via `_try_promote_blocked_waiting_request`. Testing must verify order preservation across queue migrations, especially for priority scheduling. Monitor for request starvation in `skipped_waiting` under high load.

**Key insights**
