### Summary

This PR enables NixlConnector to work with the Hybrid Memory Allocator (HMA), allowing KV cache transfers to leverage hybrid attention models (FA+SWA).   - `vllm/v1/worker/gpu_model_runner.py`
  - `vllm/v1/worker/kv_connector_model_runner_mixin.py`

### Summary

**What changed and why**  
This PR fixes a KV transfer issue in speculative decoding with NixlConnector when `kv_buffer_device=cpu`. Previously, `wait_for_save()` was called before the drafter's forward, causing drafter KV cache to never be copied from GPU to CPU, degrading acceptance length. The solution defers both `wait_for_save` and `clear_connector_metadata` until after the draft model forward via a unified `defer_finalize` parameter and adds a `finalize_kv_connector()` method.

**Technical impact**  
The changes ensure KV cache operations are correctly ordered for speculative decoding with disaggregated KV connectors. By deferring finalization, both target and drafter model KV caches are properly saved and transferred, maintaining acceptance length parity with standalone speculative decoding. The refactor consolidates deferred operations into a single control flow, simplifying the KV connector lifecycle.

**Potential risks**  
If `finalize_kv_connector()` is not called after deferred operations (e.g., due to early exits or exceptions), KV connector metadata may leak, causing undefined behavior in subsequent iterations. The fix assumes speculative decoding is the only scenario requiring deferred finalization; other use cases with `defer_finalize=True` could inadvertently skip necessary cleanup.

**Key insights**  
Developers should ensure `finalize_kv_connector()` is always invoked when `defer_finalize=True` is used, ideally within a `try-finally` block. The new CI test provides coverage for both `cuda` and `cpu` KV buffer paths, but manual verification of error handling in non-speculative decoding contexts is recommended. The unified deferral parameter improves code clarity but requires careful attention to its lifecycle management.
