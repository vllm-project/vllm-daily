**Key insights**  
The fix is critical for correct layerwise reload behavior in expert-parallel models. Developers should ensure SKIP_TENSORS remains up-to-date with all non-loadable buffers. ## 7. [[Bugfix] Zero-init MLA attention output buffers to prevent NaN from CUDA graph padding](https://github.com/vllm-project/vllm/pull/37442)


### Base Information

- **PR Number:** #37442
- **Author:** [elvircrn](https://github.com/elvircrn)

  - `vllm/v1/attention/backends/mla/cutlass_mla.py`
  - `vllm/v1/attention/backends/mla/flashinfer_mla.py`

### Summary

**What changed and why**  
The changes introduce pre-allocated, zero-initialized output buffers for MLA (Multi-head Latent Attention) decode operations to prevent NaN propagation from padding slots. When running CUDA graphs with padded batches, kernels skip writing outputs for slots with `seq_lens=0`, leaving stale data that can contain NaN from previous operations. This NaN can then contaminate real tokens during downstream per-tensor FP8 quantization.

**Technical impact**  
Both CUTLASS and FlashInfer MLA backends now maintain reusable zero-initialized output buffers as instance attributes (`_decode_out`). The CUTLASS implementation always uses the pre-allocated buffer via slicing, while FlashInfer uses it only for single-token decode (`q_len == 1`) due to a shape compatibility bug, otherwise zeroing padding slots post-kernel execution. This eliminates NaN contamination with zero runtime overhead during CUDA graph replay.

**Potential risks**  
The FlashInfer workaround for multi-token sequences (`q_len > 1`) introduces a post-kernel zeroing operation that could have minor performance implications. The conditional logic for buffer reuse (checking shape and dtype) adds complexity that must remain synchronized across both backends. There's also a dependency on the upstream FlashInfer bug fix for proper `out=` parameter support.

**Key insights**  
The fix elegantly addresses a subtle CUDA graph edge case by leveraging persistent zero-initialized buffers, avoiding per-call memset costs. Developers should note the backend-specific implementations: CUTLASS uses the buffer universally, while FlashInfer has limitations. This pattern of pre-allocating clean buffers could be considered for other attention backends with similar padding scenarios.

**What changed and why**

**Potential risks**  
If the coordination TCPStore fails or becomes unreachable, group initialization will hang. Socket lifecycle management is now more complex—sockets must be properly passed and closed to avoid resource leaks. The caching of TCPStore clients (`get_cached_tcp_store_client`) assumes a single store per `(host, port)`; if ports are reused incorrectly, stale connections could cause issues. Edge cases around partial failures during reconfiguration need careful handling.

**Key insights**  
The fix is critical for reliability in elastic scaling scenarios. Developers should ensure the coordination store is highly available and that socket ownership is correctly transferred (e.g., via `detach()`). The removal of static port lists reduces configuration complexity but requires thorough testing of reconfiguration paths. Monitor for socket leaks and ensure the cached store client is invalidated appropriately when the store changes.
