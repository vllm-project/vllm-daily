### Summary

**What changed and why**  
This PR fixes a bug where NemotronH models with MTP speculative decoding and chunked prefill produce garbage outputs. The issue occurs when short prefill chunks (below `reorder_batch_threshold`) are misclassified as decode operations, causing incorrect state management. The fix ensures these short prefills are handled correctly by adjusting `num_accepted_tokens` to reflect their actual token count.

**Technical impact**  
The changes introduce a new mechanism to track prefill chunks processed via the decode path, specifically for Mamba-based models. This ensures state slots are correctly indexed for subsequent iterations. The fix is localized to the Mamba attention backend and GPU model runner, with no impact on other attention backends like GDN, which already handles this case differently.

**Potential risks**  
The fix adds complexity to state management and introduces new tensors (`prefill_as_decode_num_tokens`). If not properly synchronized, it could lead to state corruption or incorrect token acceptance counts. Edge cases involving mixed batch types (prefill/decode) with varying chunk sizes may still need thorough testing.

**Key insights**  
Developers should note that this bug is specific to Mamba models with MTP speculative decoding and chunked prefill. The fix is necessary because short prefills are incorrectly routed through the decode path. Ensure comprehensive testing across different model sizes and chunk configurations to validate the solution.

  - `tests/kernels/attention/test_trtllm_kvfp8_dequant.py`
  - `vllm/v1/attention/backends/flashinfer.py`

### Summary

**What changed and why**  
The Triton kernel `trtllm_prefill_attn_kvfp8_dequant` was modified to support non-contiguous KV cache tensors, which occur in cross-layer unified allocation schemes (e.g., KV offloading). Previously, the kernel computed flat pointer offsets assuming contiguous memory layout, causing incorrect reads with non-contiguous strides. Now, the kernel uses actual tensor strides for page, K/V, and head dimensions, enabling correct operation with both contiguous and non-contiguous layouts.

**Technical impact**  
This change enhances the kernel's compatibility with advanced memory layouts, particularly cross-layer KV caches where per-layer views are non-contiguous. The kernel now dynamically computes source offsets using provided strides (`src_stride_page`, `src_stride_kv`, `src_stride_head`) while maintaining a contiguous output layout. This ensures correct dequantization of FP8 KV caches in unified allocation scenarios without altering the external API.

**Potential risks**  
The kernel now assumes that the `(block_size, head_size)` dimensions remain contiguous (strides `[head_size, 1]`), which is enforced via an assertion. If future layouts break this assumption, the kernel may fail. Additionally, the introduction of a per-head loop (`NUM_KV_HEADS`) could impact performance for large head counts, though this is necessary for stride-aware indexing.

**Key insights**  
Developers should ensure that any KV cache tensors passed to this kernel maintain contiguous `(block_size, head_size)` dimensions. The comprehensive test suite (71 tests) validates both contiguous and non-contiguous layouts, including edge cases like zero-padding pages and cross-layer patterns. This fix aligns with broader efforts to support unified KV cache allocation, as referenced in related PR #34158.
