## 2. [[Bugfix] Fix `RuntimeError: Already borrowed` that degrades VLM serving throughput under concurrent load.](https://github.com/vllm-project/vllm/pull/36557)


### Base Information

- **PR Number:** #36557
- **Author:** [hallerite](https://github.com/hallerite)
- **Merged By:** [vllm-bot](https://github.com/vllm-bot)
- **Merged time:** 2026-03-09 22:30:51
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/36557/files) (1):**
  - `vllm/renderers/base.py`

### Summary

**What changed and why**  
The fix addresses a `RuntimeError: Already borrowed` that occurs when the same Hugging Face tokenizer instance is shared between `AsyncMicrobatchTokenizer` (running on an executor thread) and the multimodal processor (running on the main thread). The error stems from concurrent access to a Rust `RefCell` in the tokenizer backend. The solution uses `copy.deepcopy(tokenizer)` to create a separate tokenizer instance for the multimodal processor, eliminating the race condition.

**Technical impact**  
This change prevents the retry loop in `call_hf_processor` from being triggered, which previously caused severe latency spikes under concurrent load. By decoupling the tokenizer instances, VLM serving throughput improves dramatically (from ~14–20 req/s to ~250 req/s) with no per-request overhead. The deep copy is performed once at initialization, adding minimal memory (~1–5 MB) and startup time.

**Potential risks**  
Deep copying the tokenizer may fail if the tokenizer object contains non-picklable attributes or custom state. Additionally, any future changes that reintroduce sharing of tokenizer state between the copies could lead to subtle inconsistencies. The fix assumes the tokenizer’s internal Rust backend is safely isolated after copying, which relies on the behavior of the underlying Hugging Face library.

**Key insights**  
The root cause was a classic concurrency issue with shared mutable state across threads. Developers should be cautious when passing heavyweight, stateful objects like tokenizers between concurrent components. Consider implementing a copy-on-share pattern for similar resources in the future. The fix is efficient and targeted, but ensure tokenizer copies remain independent in all usage scenarios.
