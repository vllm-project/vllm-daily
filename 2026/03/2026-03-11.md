**What changed and why**

Previously, when `stream_interval > 1`, multiple XML elements could arrive in a single delta, causing the state machine to skip parameters and silently drop arguments. The new approach buffers tokens until a complete `<invoke>...</invoke>` block is available, then parses it in one shot using the existing `_parse_single_invoke()` method.

**Technical impact**  
This change significantly simplifies the streaming state management—reducing it to just two variables (`is_tool_call_started`, `current_tool_index`). It also improves maintainability by reusing the non-streaming parsing logic and eliminating complex incremental tracking of XML elements.

**Potential risks**  
Buffering until a complete invoke block is available could theoretically increase memory usage for very large tool calls, though this is mitigated by typical tool call sizes. The removal of the old state machine may affect edge cases not covered by the new unit tests, such as malformed or nested XML structures. Additionally, the reset logic now triggers on `tool_call_starting` rather than only on start token presence, which could be more aggressive than necessary.

**Key insights**  
Developers should note that the parser now relies on complete XML blocks, making it robust to streaming variations. The extensive new test suite provides strong coverage for incremental and batched scenarios. When modifying this parser, ensure any new features maintain the buffer-until-complete invariant and avoid reintroducing chunk-dependent logic.
