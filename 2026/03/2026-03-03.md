## 1. [[Core] Remove busy loop from idle buffer readers](https://github.com/vllm-project/vllm/pull/28053)


### Base Information

- **PR Number:** #28053
- **Author:** [joerunde](https://github.com/joerunde)
- **Merged By:** [njhill](https://github.com/njhill)
- **Merged time:** 2026-03-03 23:44:21
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/28053/files) (5):**
  - `tests/basic_correctness/test_basic_correctness.py`
  - `tests/distributed/test_shm_broadcast.py`
  - `vllm/distributed/device_communicators/shm_broadcast.py`
  - `vllm/envs.py`
  - `vllm/v1/executor/multiproc_executor.py`

### Summary

**What changed and why**  
This PR replaces the busy-loop polling mechanism in shared memory buffer readers with a ZMQ-based notification system. The primary motivation is to eliminate CPU consumption during idle periods, which is critical for IBM SystemZ mainframes where CPU cores are expensive and shared across multiple tenants. The solution uses ZMQ sockets for write notifications and an in-process socket for clean shutdowns, allowing readers to sleep indefinitely when no work is available.

**Technical impact**  
The architecture shifts from active polling to event-driven waiting, fundamentally changing how readers detect available data. The `SpinTimer` class is replaced with `SpinCondition`, which manages both busy-looping for high-frequency reads and idle waiting via ZMQ poller. The `MessageQueue` API now exposes a `.shutdown()` method instead of accepting cancellation events, requiring changes to the multiprocess executor's shutdown logic and death monitoring.

**Potential risks**  
The ZMQ dependency introduces additional complexity and potential failure modes in socket communication. There's a risk of notification loss if the ZMQ high-water mark is exceeded during high load, though it's configured to minimize this. The shutdown mechanism now relies on proper socket cleanup, which could lead to resource leaks if not handled correctly. Performance regression is possible if the notification overhead outweighs the previous busy-loop efficiency.

**Key insights**  
This is a significant architectural improvement for CPU efficiency in idle states, particularly valuable for shared/resource-constrained environments. Developers should verify that the ZMQ notification system doesn't introduce latency spikes under high-throughput scenarios. The removal of the `VLLM_SLEEP_WHEN_IDLE` environment variable simplifies configuration but eliminates the previous opt-in mechanism for power saving.

### Summary

**What changed and why**  
The fix addresses a crash in speculative decoding when `allowed_token_ids` is set in `SamplingParams` without penalties. The bug occurred because `sampling_metadata.output_token_ids` could be empty (to avoid overhead), causing a batch size of zero and a `RuntimeError` in `repeat_interleave`. The solution replaces `len(sampling_metadata.output_token_ids)` with `len(metadata.num_draft_tokens)` to reliably obtain the batch size.

**Technical impact**  
This change ensures the batch size is derived from `metadata.num_draft_tokens`, which always contains one entry per request in speculative decoding. It maintains correctness when `output_token_ids` is populated (no behavior change) and fixes the crash when it’s empty, without introducing performance overhead or altering other code paths.

**Potential risks**  
- The fix relies on `num_draft_tokens` as a proxy for batch size, which is semantically tied to speculative decoding rather than general batch size. This could be confusing if the code is later reused outside speculative contexts.  
- If `metadata.num_draft_tokens` is ever empty or malformed (e.g., due to future changes), similar crashes may resurface.  
- The root cause—`output_token_ids` being empty when needed—remains unaddressed, though it’s intentional for performance reasons.

**Key insights**  
- The chosen fix is minimal and avoids performance penalties (like GPU-CPU synchronization) that alternative solutions would introduce.  
- Developers should note that `output_token_ids` may be empty for optimization purposes; any new code relying on it for batch size must consider this.  
- Consider adding a comment in the code to clarify why `num_draft_tokens` is used here, ensuring future maintainers understand the speculative decoding context.
