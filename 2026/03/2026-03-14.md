### Summary

**What changed and why**  
This PR removes the `torchcodec` dependency and replaces it with `pyav` (already listed as `av`) for audio extraction from video files. The change addresses compatibility issues on aarch64 systems where `torchcodec` lacks pre-built wheels, while `av` provides proper aarch64 support. The dependency change reduces architecture-specific installation issues without altering the external API or core transcription functionality.

**Potential risks**  
The removal of `torchcodec` eliminates a robust fallback decoder that handled various audio formats via FFmpeg. While `pyav` also uses FFmpeg, differences in decoding behavior or error handling could affect edge cases with malformed or uncommon media files. The consolidated error handling now raises generic `ValueError` for all decode failures, which may obscure specific error details previously available from `torchaudio` exceptions.

**Key insights**  
This change improves cross-platform compatibility but reduces decoding redundancy. Developers should verify that `av` (pyav) is correctly installed with FFmpeg support on all target platforms. Consider adding more granular error logging in the fallback path to distinguish between format unsupported, corruption, and decoder initialization issues. The test plan confirms basic functionality but should be expanded to include diverse audio/video formats.

  - `vllm/v1/structured_output/utils.py`

### Summary

**What changed and why**  
The fix addresses a dtype mismatch error when using xgrammar for guided decoding on macOS CPU inference. The xgrammar CPU kernel strictly requires `float32` logits, but macOS CPU inference can produce `float16` or `bfloat16` logits. The solution adds conditional dtype conversion in `apply_grammar_bitmask()` to temporarily convert logits to float32 for the kernel operation, then copies the results back to the original dtype.

**Technical impact**  
This change ensures compatibility between the xgrammar library's CPU kernel and vLLM's inference backend on macOS, allowing structured output generation (like JSON schema) to work correctly on CPU. The modification is localized to the utility function and only activates for CPU tensors with non-float32 dtypes, preserving existing behavior for GPU and float32 CPU tensors.

**Potential risks**  
The `copy_()` operation introduces a slight performance overhead due to the extra dtype conversion and memory copy. There's a theoretical risk if the xgrammar kernel modifies tensor metadata or shape, though this is unlikely. The fix assumes the issue is specific to macOS CPU, but the same problem could affect other CPU platforms using non-float32 dtypes.

**Key insights**  
This is a pragmatic workaround for a third-party library constraint. Developers should be aware that CPU inference with structured outputs now incurs a small performance penalty. Consider monitoring if the xgrammar library updates its kernel to support more dtypes, which would allow removing this conversion. The fix correctly preserves the original tensor's device and dtype after modification.

Developers should note that InstantTensor provides the most benefit on high-bandwidth storage systems (400 Gbps+) and multi-GPU configurations.
