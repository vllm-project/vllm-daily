  - `vllm/tool_parsers/hermes_tool_parser.py`
  - `vllm/tool_parsers/longcat_tool_parser.py`

### Summary

**What changed and why**  
The fix addresses a bug where the Hermes tool parser failed when `stream_interval > 1` caused tokens to arrive in multi-token chunks. Previously, the parser relied on buffering individual tokens to reconstruct `<tool_call>` tag boundaries and used `partial_json_parser` for incomplete JSON. The new approach parses the full `current_text` on each call and diffs against already-sent content, making it independent of chunk size.

**Technical impact**  
The parser now operates on accumulated text rather than token-by-token buffering, simplifying the logic and removing dependencies on `partial_json_parser`. State management shifts from tracking token buffers to maintaining indices of sent content and arguments (`_sent_content_idx`, `streamed_args_for_tool`). This improves robustness across varying stream intervals and aligns with how other tool parsers (e.g., Longcat) handle streaming.

**Potential risks**  
If `_sent_content_idx` or `streamed_args_for_tool` state is not properly reset between requests, it could cause cross-request contamination. The regex-based JSON extraction may be brittle with malformed or nested JSON structures. Additionally, the removal of token-level buffering could affect edge cases where tokenization ambiguities arise (e.g., partial Unicode characters).

**Key insights**  
The refactor significantly reduces complexity by eliminating token-buffering logic and external JSON parsing dependencies. Developers should ensure the parser’s internal state is cleared per request and verify regex patterns handle all valid JSON variations. The comprehensive test suite with varying `stream_interval` values provides strong validation, but integration testing under real streaming conditions is recommended.
