  - `vllm/v1/attention/backends/rocm_aiter_fa.py`

### Summary

**What changed and why**  
The change removes `self.sliding_window[0] != -1` from the condition that routes attention computation through the `unified_attention` path. Previously, this condition caused non-speculative workloads with sliding window attention to incorrectly use `unified_attention` during CUDA graph capture, leading to GPU memory faults. The fix ensures `unified_attention` is only used for actual speculative decoding (`decode_max_query_len > 1`).

**Technical impact**  
Sliding window attention now correctly routes through the existing `paged_attention_v1` path, which natively supports sliding windows and is CUDA-graph-capture-safe. This resolves a critical crash during CUDA graph capture for decode operations, allowing graph capture to complete successfully and improving server stability on multi-GPU setups.

**Potential risks**  
Removing the sliding window condition could theoretically affect performance if `unified_attention` were optimized for sliding window scenarios, but the existing `paged_attention_v1` path already supports sliding windows. Ensure thorough testing of sliding window attention across various model configurations to confirm no regression in functionality or performance.

**Key insights**  
The root cause was conflating two distinct features (sliding window and speculative decoding) in a single condition, inadvertently introducing CUDA graph capture unsafety. Developers should carefully isolate feature-specific code paths, especially when combining dynamic operations (like memory allocations) with CUDA graph capture. Always validate that graph-captured paths avoid runtime memory allocations and dynamic kernel selection.
