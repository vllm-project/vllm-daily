### Summary

**What changed and why**  
This PR disables dual stream execution for Qwen3 and Qwen3.5 models by removing the custom `gdn_in_proj` operation and reverting to sequential execution of input projections. The change addresses a regression in cold compile times (~4x slowdown) caused by the custom op's requirement to pass layer names as strings, which interferes with PyTorch compilation optimizations.

**Technical impact**  
The removal of `gdn_in_proj` and associated parallelization logic (`maybe_execute_in_parallel`, auxiliary streams, and events) simplifies the codebase but sacrifices potential throughput gains from concurrent kernel execution. **Potential risks**  
Reverting to sequential execution could reduce inference speed, especially for large batch sizes or long sequences where dual-stream execution previously hid latency. The temporary nature of this change (pending PyTorch 2.11 upgrade and issue #38123) risks accumulating technical debt if not re-enabled promptly. Additionally, the removal of custom ops may affect edge cases where the fake implementation for `torch.compile` was necessary.

**Key insights**  
This is a tactical trade-off favoring compilation speed over runtime performance, highlighting the sensitivity of PyTorch compilation to string parameters in custom ops. Developers should monitor inference benchmarks to quantify the performance impact and prioritize the TODO item to re-enable dual-stream after dependencies are resolved. The cleanup of unused imports and auxiliary stream infrastructure is a positive side effect.
