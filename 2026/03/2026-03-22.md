  - `vllm/v1/worker/gpu/mm/encoder_cudagraph_defs.py`
  - `vllm/v1/worker/gpu_model_runner.py`

### Summary

**What changed and why**  
This PR introduces a full CUDA graph implementation for Vision Transformer (ViT) encoders to reduce kernel launch overhead. It implements budget-based CUDA graph capture with greedy bin-packing, data-parallel support, and a model-agnostic protocol. The system captures graphs at configurable token budgets and pads sequence metadata to enable reuse across varying numbers of images during replay.

**Technical impact**  
The changes add a sophisticated CUDA graph management system that significantly reduces encoder latency (11-44% improvements shown in tests). The architecture separates model-specific logic (via the `SupportsEncoderCudaGraph` protocol) from generic graph management (`EncoderCudaGraphManager`). This enables transparent integration into the existing multimodal processing pipeline while maintaining backward compatibility through eager fallbacks.

**Potential risks**  
The greedy bin-packing algorithm assumes sorted inputs by token count; incorrect sorting could reduce packing efficiency. Data-parallel mode requires careful load balancing across TP ranks. The system depends on proper buffer padding and zeroing to mask unused positions in attention computations. Edge cases where inputs exceed all budgets could trigger frequent eager fallbacks, negating performance benefits.

**Key insights**  
The implementation demonstrates excellent performance gains, particularly with FlashInfer attention (44% mean improvement). The protocol-based design allows easy extension to other multimodal models beyond Qwen3-VL. Developers should ensure token budgets are properly configured for their workload distribution, and monitor hit rates to validate graph utilization. The system automatically falls back to eager execution when graphs don't fit, maintaining correctness.

### Summary

**What changed and why**  
This PR reverts a previous optimization (#34854) by switching Gumbel noise generation from FP32 to FP64 for improved numerical stability. The changes introduce a new Triton kernel (`tl_rand64`) to generate 64-bit uniform random numbers and ensure logits/probabilities are processed in FP64 during sampling and rejection sampling.

**Technical impact**  
The switch to FP64 eliminates potential numerical underflow/overflow issues in Gumbel noise generation, particularly when computing `-log(-log(u))` for extremely small `u`. However, it introduces a performance penalty: random sampling slows by up to 1.8x for large batch sizes (128+), while greedy sampling remains unaffected. The `local_max` tensor now uses `torch.float64`, increasing memory usage per token.

**Potential risks**  
The performance degradation for large-batch random sampling could impact throughput-sensitive workloads. The increased memory footprint from FP64 tensors may reduce memory efficiency, especially for large vocabularies. The `tl_rand64` kernel assumes correct 64-bit random number generation via bitwise operations, which could introduce platform-dependent behavior if not thoroughly validated.

**Key insights**  
This trade-off prioritizes numerical correctness over performance, which is appropriate for sampling stability. Developers should monitor the performance impact in production and consider batch-size-aware optimizations. The `tl_rand64` kernel should be tested across different hardware to ensure consistent random number generation. If future optimizations are needed, consider hybrid precision approaches (e.g., FP64 only for extreme values).
